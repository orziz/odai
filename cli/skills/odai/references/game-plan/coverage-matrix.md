# Game Plan Coverage Matrix

## 何时读取

由 `game-plan` 触发：判断游戏类型 / 策划工种归属，或与 `game-design` / `feature-plan` 裁边界。

## 边界判定

1. 根问题在规则、循环、成长、资源、平衡、内容、版本或商业模型 → 留 `game-plan`。
2. 根问题在游戏界面视觉、交互体验、角色场景、宣传品牌、特效演出或整体视觉方向 → 转 `game-design`。
3. 根问题在通用规格、接口约束、执行前提或一般性 bug 排查 → 转 `feature-plan`。
4. 同时含策划本体和视觉设计 → 先由 `game-plan` 稳定规则与口径，再交 `game-design`。

## 覆盖范围

`game-plan` 覆盖所有常见游戏结构与主要策划工种，不按题材名词直接做方案，而是先拆循环、成长、资源、内容、商业和运营结构。用户只说类型名时，还要补核心乐趣、目标玩家、平台、版本节奏和商业边界。

主要工种：

- 系统 / 玩法 / 功能：系统结构、局内外循环、触发、状态、功能耦合。
- 数值 / 经济：公式、成长曲线、难度曲线、资源产消、兑换链路、通胀 / 紧缩。
- 商业化：付费点、价值包装、货币体系、公平性边界、投放节奏。
- 关卡 / 活动 / 版本 / LiveOps：目标、结构、事件、奖励、节奏、资源回收、内容投放。
- 叙事 / 世界观 / 内容：设定、角色关系、剧情功能、内容产能、版本承接。
- 新手引导 / 体验策划：教学节奏、引导链路、卡点、失败反馈、早期留存。

## 输出与切换

按主矛盾选择最小交付：系统方案、玩法草案、数值框架、经济循环说明、商业化方案、关卡 / 活动 / 版本方案、叙事 / 世界观 / 内容草案。

切到 `game-design`：规则已稳，争议转到界面视觉、角色场景、品牌包装、特效演出、信息密度、可读性或操作负担。

切到 `feature-plan`：需要把策划结论包装成通用需求规格、执行前提或跨团队交接文档，或问题已变成非游戏专属功能 / 流程 / bug 诊断。
