# 实现专项边界

由 `implement-code` 按客观改动类型触发，只执行命中的小节。

## UI 实现

涉及 UI 时：

1. 先读现有页面、组件库、样式系统、design token、设计文档和相邻页面模式；设计基线缺失、冲突或无法判断复用对象时，升到设计阶段，不自行发明规则。
2. 同时检查组件复用与一致性，以及正常、空、加载、异常、禁用、成功反馈、响应式和回退状态，不只修静态画面。
3. 已确认浏览器或目标运行环境不可用、且无等价验证时，按 `implement-code` 的 `ready` 规则收口；构建 / lint 只报告其覆盖的静态属性，不得冒充视觉与交互验收。

## 大改动 / 多步落地

按垂直切片推进，不按层平铺：每个切片都是端到端可用的小改动，切片间让 `implement-code` 选出的相关验证保持绿，不留断裂中间态；一次提交只改一个逻辑关注点，删除与替换尽量各自可回滚。改动攒大前先验证；同一轮可以包含多个独立保持绿的切片。

## 对外接口 / API

在系统边界（路由、表单、外部响应入口）校验输入，内部信任；同类错误形状一致。优先加字段 / 端点而非改删既有契约以保后向兼容（任何可观测行为都可能已被依赖）；列表端点从一开始分页；命名随生态约定（REST 复数名词，布尔 is / has / can）。接口契约先于实现。

## 外部输入与模型输出

一切外部输入默认不可信并在边界校验；LLM / 模型输出同样如此，进入 SQL、shell、DOM、文件路径、反序列化或工具调用前必须校验。权限与安全边界写进代码，不靠 prompt；secret 不进模型上下文与日志。通用注入、鉴权与敏感数据由 `review-sslb` 兵部主审。
