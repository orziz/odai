# Planning Playbook

## 何时读取

由 `feature-plan` 及各规划类模块按各自触发行引用：附件 / 低信息输入、多条 / 混合输入需要需求条目账本与反向映射、分层步进提问、多方案比较、结构化对象属性完备性或规划文档草案。

## 最小顺序

1. 写当前理解；拿不准即列未确认点。
2. 疑似缺标点、乱标点、错别字、多字、少字、语序断裂、指代错位或语义误写时，先做纠偏理解：原话片段、重断句 / 分条后的理解、依据、是否需要用户确认。需求条目账本出现纠偏时，必须单列原话片段与纠偏理解；只把修正后的句子放进条目，视为静默改写。纠偏会改变全局契约治理点或存在多种合理解释时，列为待确认项。
3. 多条、混合或低结构输入先拆成需求条目账本：保留用户原话语义，标记每条属于目标、手段、约束、根因猜测、验收、风险、纠偏理解还是扩展候选；每个独立动词、对象、约束、验收点、非目标、目标文件或显式引用都单列原子项，不得把并列要求揉成一条或把推断补成用户已说。
4. 视需要用九层透镜排查缺口：真正目标、成功标准、输入 / 触发、输出 / 结果、范围 / 非目标、表层诉求、用户手段、约束声称、已有证据；已清楚的层不必机械展开。
5. 输入少或乱时，先整理已知、未知、冲突点和必须确认的问题，不只回“请补充”。
6. 需求条目账本是外显交付物，不是内部思考。账本内存在纠偏待确认、扩展候选、待确认 / 待验证项、对象缺证或新建对象边界时，本轮收口停在账本、草案和确认问题；不得同轮继续实现、补测试或写文档落地。
7. 形成草案、方案或执行前提前，在需求条目账本内做反向映射；原始条目只用四个覆盖状态：已覆盖、改写待确认、暂缓待确认、待验证补证。每个新增建议说明来自原始条目、项目证据、相邻风险还是主动扩展。
8. 需要文档时先写最小草案；位置按当前对话或项目 Markdown 选择。

## 低信息与理解扩展

- 有附件 / 参考材料时，先分析目标、显性规则、隐性前提、遗漏、与项目现状冲突，以及哪些可采用、哪些须确认。
- 几乎没需求时，先查项目说明、相似功能、已有模式和约束；无足够参照时先问缺失前提。
- 表层诉求不等于真正目标；用户手段不等于最优路线；约束声称不等于硬约束。
- 若用户已指定方案，反推核心问题，并判断是否有更小、更稳、更省长期成本的路线。
- 主动补看相邻场景、边界条件、失败路径、上下游依赖、环境差异、流程误用、维护与协作成本；用户信息不全或未想周全时，应主动端出遗漏和更稳路线。
- 主动扩展分级处理：关键缺口会影响全局契约治理点（目标、边界、授权、验收、风险、成本、停止条件或不可接受结果）时必须确认；相邻价值可作为推荐候选；锦上添花的增强只列为可选，不默认进本轮范围。
- 扩展候选必须和用户原始需求分栏，说明触发依据、收益、代价和不采纳的后果；未经确认不得写成既定需求、验收标准或实施范围。
- 结构化对象或参数集要推敲生命周期、状态、关联、检索分类、追溯合规；候选字段只作待确认提议。

## 提问、草案与比较

- 分层提问：目标层、场景层、边界层、落地层；同层一次问完，只问当前最影响治理判断的层。可由材料补齐的层先补证，不问用户。
- 探索期草案：当前理解、需求条目账本、待确认问题、未确认路径、本轮建议下一步。
- 完整方案按需补：目标与非目标、需求条目账本（含覆盖状态）、约束与风险、方案选项、扩展候选、执行拆解、用户文档、AI 文档。
- 有冲突或多解释路径时，必须写出冲突和待确认解释。
- 多方案比较给真正不同的 2-3 个方案，至少含最小可行与更彻底对照；纳入不做 / 观察、复用现有能力、绕过问题、退回上游。
- 用同一维度比较：解决程度、改动面、风险、维护成本、一致性、可逆性、依赖与前提。
- 推荐项说明为何匹配用户权衡、代价是什么、何时应改选；未经确认不得当既定路径。并行辅助按 `references/dao/agent-governance.md`。
