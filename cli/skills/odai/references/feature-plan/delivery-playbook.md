# Delivery Playbook

## 何时读取

由 `feature-plan` 触发：bug、用户说明 / AI 任务草案、可交给下一轮 AI 的执行文档、跨会话交付边界。

## 最小判路

1. bug / 报错 / 异常先拆：已确认现象、用户猜测、已有证据、缺失证据、下一步排查。
2. 输出类型四选一：用户说明、AI 任务草案、两者都要、临时草案。
3. 会改变方向、边界、验收或风险的缺口先问或补证；能自行验证的事实先验证。
4. 本模块只整理边界和文档，不进入非文档实施。

## Bug 场景

- 不默认用户描述就是根因；先区分现象、猜测原因、已验证根因。
- 先问 / 补：复现稳定性、环境、步骤、输入条件、日志 / 截图 / 录屏 / 请求记录 / 对比样本。
- 排查区分实现缺陷、需求未对齐、数据异常、权限限制、环境差异、时序问题、预期与设计不一致。
- 输出至少包含：已确认现象、疑似原因、已有证据、缺失证据、下一轮最有价值排查动作。

四阶段根因纪律：复现与隔离（先建对准症状的红信号）→ 根因定位（沿数据 / 控制流证伪首个假设）→ 最小修复（只能治标要标“缓解非根治”）→ 防同类复发（原场景确认消失，补回归 / 校验 / 边界处理并查同形实例）。

## 输出策略

- 用户说明：帮助用户理清需求、决策、补材料、确认边界。
- AI 任务草案：边界较清楚，用户需要交给其他 AI / workflow 执行。
- 两者都要：既需用户拍板，也需 AI 直接执行。
- 临时草案：仍处关键理解阶段，只输出当前理解、临时方案、待确认问题。

判断标准：是否明确要执行、边界是否稳定、受众是人还是 AI、交给 AI 是否会跑偏、题目定义是否仍停在表层诉求或未验证根因。

## 文档要求

用户文档说清真实目标、权衡偏好、不可接受结果、边界、风险、方案取舍、下一步；少术语，不写成纯技术清单。

AI 文档明确任务边界、真实目标、输入材料、执行顺序、每步产出、必须先确认的点、不得擅自扩展的点；适合继续交给下一轮 AI 时可附：

```text
请基于以上方案继续执行，范围限定在：<确认范围>。
要求：先复述边界与待确认点；信息不足先问最少必要问题，已足够则直接开始；按“步骤 -> 产出 -> 风险”推进；未经确认不扩展需求。
```
