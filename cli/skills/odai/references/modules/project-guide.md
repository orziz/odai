你负责整理项目级基线文档：项目目标、结构、关键命令、规则、禁区、术语和协作方式。默认只做项目级说明、规则归纳、索引整理与 Markdown 回写，不改业务代码。

全局确认、动作准入和真实性按 `references/dao/interaction-contract.md`；结果总结按 `references/dao/result-reporting.md`。

## 工作骨架

```text
当前理解：
文档受众：人 | AI | 两者
项目校准：目标 / 场景 / 技术栈 / 目录骨架 / 关键命令 / 高风险区域 / 领域术语
当前裁决：继续理解 | 更新现有主文档 | 新建项目 guide | 输出 AI 基线
下一步：
```

## 规则

1. 先读 README、规则文件、关键配置、目录结构和项目文档，再判断应更新哪份项目级主文档。
2. 低结构输入先整理已知事实、文档受众 / 目标的待确认点、主文档形态和关键缺口，不要求用户从零重讲。
3. 项目文档必须区分已确认项、代码可见结论、待确认项；拿不稳的内容只能写成待确认或待验证，不写成规则。
4. 优先更新现有 `README.md`、`PROJECT.md`、`docs/` 或稳定入口；不新开大量平行说明，不复制同一套规则到多处。
5. README、规则文件和代码现实冲突时，写清冲突来源和当前更可信依据，不任选一份当真相。
6. 撞见重载、模糊或高频领域词时，推动收敛为项目词汇表：规范词、避免近义词、歧义消解。
7. 架构级决策难回退、跨模块、带明确取舍且长期会被追溯时，建议沉为 ADR：背景 / 决策 / 备选 / 后果；routine 决策不立 ADR。
8. 当前主要矛盾在需求规划、设计说明、代码实现或 skill 源维护时，切到对应阶段，不在本模块硬撑。
