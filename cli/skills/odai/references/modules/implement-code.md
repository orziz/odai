将已稳定的目标落成代码、测试和必要文档。只在新证据会改写用户目标、范围、授权、验收或不可接受结果时回交 `道` / 上游；普通实现细节由当前模型自主决策。

## 准入快检

实现前确保能回答：

- 要改变什么，什么必须保持不变。
- 目标对象和显式引用在哪里。
- 用什么证据判定完成，环境无法验收时如何说明缺口。

答案可来自用户原话、上游文档、代码契约、失败测试或现有项目事实，不要求额外仪式化确认。

## 覆盖与范围

- 多条需求、多文件、批量引用、重命名或“一起改”时，建立覆盖清单：原始条目、目标文件 / 符号、显式引用、保持不变项、动作或不处理理由。
- 有会改写范围的纠偏、扩展候选、对象缺证或待确认项时，回交上游按 `references/feature-plan/planning-playbook.md` 处理；输入清楚时可用内部覆盖清单直接实施。
- 广范围词先完成只读盘点。范围、风险和验证已被证据锁定且不需要用户取舍时，可直接推进；否则回交 `道`。
- 找不到用户所指对象或相似实现时，不自行假定为新建对象。

## Bug 与红信号

用户给出的根因或修法仍是待证假设。优先建立能对准症状的失败测试、复现命令、日志断言或等价红信号；有稳定接缝时，先见红再修复。

无法运行对症红信号时，只有同时满足以下条件才可继续实施：

- 静态证据能沿代码、数据或控制流直接连接用户现象与缺口；相邻契约违例、一般可能性或无关绿路径不算。
- 改动局部、可逆，不需要新的产品 / 设计裁决。
- 能用静态检查、类型、已有测试或差异复核降低风险。
- 收口标 `ready`，明确未验证原始症状，不宣称 bug 已修复。

任一条不满足，停在补证或执行前提。详细诊断工艺见 `references/dao/diagnose-kit.md`。

偶发、环境相关或跨层症状无法复现，且没有对症日志、trace、转储或等价强证据时，列候选假设与最小补证条件，不因发现附近缺陷就进入实施。

## 实现原则

1. 先读相关代码、测试、文档与相似实现；局部修改、局部抽象、局部验证。
2. 复用顺序：仓库现有实现 → 语言 / 平台 / 框架原生能力 → 已安装依赖 → 新代码 / 新依赖。
3. 新增、修改或无法从仓库确认的外部 API / 库 / CLI 行为，先确认实际版本，再查对应官方文档或源码。
4. 不做无关重构、全仓 cleanup、根工具链改写或没有当前消费方的 speculative abstraction。
5. 默认检查正确性、边界、错误流、回退路径和资源生命周期。实现中发现的相邻价值只作建议，不静默扩范围。

## 模式与专项

- 默认直接实现。用户要求测试先行、核心规则 / 算法 / 状态机、稳定可复现 bug 或高回归风险时，可进入 TDD 并读 `references/implement-code/tdd-kit.md`。
- 涉及 UI、多步大改、对外 API、外部输入或模型输出时，只读 `references/implement-code/special-cases.md` 命中的小节。

## 验证与收口

- 将每个验收点和主要回归风险映射到有判别力、环境具备、成本合理的证据；测试、构建、lint、format 不是固定清单。
- 多文件或重命名任务用定向搜索 / diff / 类型 / 测试回扫旧符号、目标文件和引用点；测试通过不替代覆盖回扫。
- 验收环境不可用时，按 `references/dao/interaction-contract.md` 以 `ready` 收口。
- 同一方向连续两轮未达成时，读 `references/dao/inquiry-discipline.md` 换正交方向；感知验收未稳时停止直接调实现，回交对应设计 / 规格模块。
- 跨多文件或核心逻辑完成后，交 `review-sslb` 做与风险成比例的复核。

被下放为 agent 时，在冻结范围内可自主做局部实现与验证决策；新证据动摇上游治理点时停手回交。
