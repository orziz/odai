`道` 是 `odai` 的复杂任务总控：定方向与边界，收主路与生产者链，再落先手与验证。读取本文件表示轻量或直达不足以安全完成；不要把本可直达的任务补成重流程。

## 判层

- **道**：真正目标、名分、边界、优先级、授权或验收仍会改写任务。
- **术**：目标已稳，但主路、备路、依赖、生产者链或切换条件未稳。
- **法**：方向与主路已稳，只差先手、验证、止损和收口。

出现“一实施就不断改目标 / 验收”时，不是法层失败，而是上游未稳；回道 / 术重裁。

## 最小裁决

```text
当前理解：
道：为何做、守什么、不做什么
术：主路、备路、胜点、切换条件
法：先手、证据、止损、回写
当前裁决：定道 | 收术 | 落法 | 复核 | blocked
生产者 / 生产者链：
下一步：
```

对外输出按任务裁剪，不机械填满模板。

## 总控规则

1. 先读与当前待解决决策直接相关的项目事实：README、相似实现、已有文档、测试、命名与权限边界。可自证的不问用户。
2. 只有会改写治理点的缺口才阻断。命中时，本轮以一次成组确认收口，不替用户选择后继续下游；项目里存在明显缺陷只能缩小选项，不能证明用户选择了哪个目标、范围或验收。其他实现细节可标注前提并推进。
   泛化提质类先列目标、边界、验收与不可接受结果的已锁定 / 待确认状态；只问待确认项。项目事实不能单独替用户选择主观体验目标，方向选项也不能替代缺口。
3. 收术时只比较真实不同的主路、代价与切换条件。可一次定出 `feature-plan -> design-spec -> implement-code -> review-sslb` 等生产者链，段与段之间不回弹等待。
4. 上游已稳时，不为补齐阶段而生成长规格 / 设计文档；直接交适合的下游生产者。
5. 进入代码、配置、脚本、测试或构建动作前，读 `references/modules/implement-code.md`。
6. 承接上轮实施验收时，先按 `references/dao/interaction-contract.md` 恢复验收口径。
7. 新证据动摇上游时回写；局部实现失败交 `implement-code`，不把工具或实现问题误报成用户决策阻断。

## 生产者路由

按“当前还缺哪个决定”路由，不按文件类型、关键词数量或个人偏好猜主要矛盾。

| 当前仍缺的决定 | 生产者 |
|---|---|
| 通用产品规格、方案取舍、非代码诊断 | `feature-plan` |
| 非游戏页面、交互、状态、视觉与体验 | `design-spec` |
| 游戏系统、数值、经济、商业、关卡与内容 | `game-plan` |
| 游戏 UI/UX/UE、角色场景、品牌、特效演出 | `game-design` |
| 已稳定的实现、项目说明、审查、成果整理 | `implement-code` / `project-guide` / `review-sslb` / `ribao` |

改变用户可见行为、状态、流程或感知基线时，只有缺少产品 / 设计决策才转上游；若只是实现回到已有契约、失败测试或明确参数，可直接实施。

交叉域按以下顺序决胜：

1. 用户已点名模块且对象明确时从用户；安全与授权硬门仍有效。
2. 已有契约、失败测试、截图基准或明确参数足以确定行为时，直接 `implement-code`；不为文件同时含产品或视觉内容重开上游。
3. 预期行为、权限、流程、成功 / 失败语义仍需决定时，先走 `feature-plan`；游戏系统、经济、商业、关卡与内容规则对应 `game-plan`。
4. 行为已稳，只缺非游戏交互与视觉表达时走 `design-spec`；游戏 UI、反馈、角色场景、品牌或演出表达走 `game-design`。
5. 同时缺多层决定时，不强选唯一生产者，按上游到下游组成链：规格 / 策划 → 设计 → `implement-code` → `review-sslb`。游戏商业流程与 UI 同时变化时采用 `game-plan -> game-design`；技术诊断暴露规格缺口时采用 `feature-plan -> implement-code`。
6. 两条路线证据相等时，选择能解决下一个不可逆决定的最小上游生产者；仍会改写目标、边界或验收时回 `道` 成组确认，不随机决胜。
