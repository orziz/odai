# Game Design Coverage Matrix

## 何时读取

由 `game-design` 触发；用于判断游戏视觉收口范围，以及与 `game-plan` / `design-spec` 的切换边界。

## 判断规则

1. 根问题在游戏界面视觉、交互体验、角色场景、宣传品牌、特效演出或整体视觉方向，留在 `game-design`。
2. 根问题在系统规则、循环、数值、经济、商业、关卡或叙事本体，转 `game-plan`。
3. 根问题在非游戏专属的通用页面、交互、信息架构或体验说明，转 `design-spec`。
4. 同时含规则和视觉时，先由 `game-plan` 稳定规则口径，再把表现层交给 `game-design`。

## 覆盖范围

`game-design` 是完整游戏视觉设计，不只是 UI 美术：

- 游戏 UI/UX/UE：HUD、菜单、背包、商城、编队、养成页、战斗内外界面、引导反馈。
- 角色 / 场景 / 道具：形象方向、题材气质、识别性、阵营差异、资产复用。
- 宣传 / 品牌 / 包装：KV、海报、商店图、活动包装、品牌气质、传播一致性。
- 特效 / 演出：技能特效、转场、镜头包装、读招反馈、表现节奏、性能边界。

不把游戏视觉简化成界面排版；不把本模块吞成游戏策划总模块；不把 `design-spec` 当游戏视觉主模块。

## 输出与切换

按主矛盾输出：界面体验 → UI/UX/UE 方案；角色 / 场景 / 道具 → 视觉方向、风格基线、识别规则；宣传 / 品牌 → 品牌包装视觉方案；特效 / 演出 → 表现方向、节奏、性能边界。

切到 `game-plan`：争议点已在系统规则、数值、经济、商业、关卡或叙事本体，或上游口径未稳导致视觉方案无法独立成立。

切到 `design-spec`：任务不是游戏专项，只是通用页面、交互或体验设计；或用户要跨产品复用的通用设计说明。
