# 查证止损纪律

分析、规划、排查、诊断这些需要反复求解、逐步收敛的活，共享一套抗绕圈、抗锚死的纪律。诊断（bug / 报错 / 异常 / 行为不符 / 性能回归）是本纪律的特化：先按本纪律，再读 `references/dao/diagnose-kit.md` 补红信号闭环那套。

## 何时读取

主阶段进入分析、规划、非平凡排查或诊断（需反复求解、逐步收敛）时读取。轻量只读单点问答、成果整理、边界已锁的直接执行不触发。

## 判别门：反复没收敛时先分清哪类，别一律当绕圈

试 / 改同一处反复进行时，先判并外显一行是哪类：

- 需求 / 方向变更：用户改了要什么、目标、验收或方向 → 外显「目标变更 → 重定：<新目标>」，止损计数清零、重定当前理解（必要时回上游重规划），照新方向做，**不当绕圈、不降格用户方向**。这是用户改「要什么」（权威）。
- 未收敛：目标没变、只是没解出来 → 进下面止损。

## 止损：绕圈即停，换方向不换变体

同一目标连续两轮未收敛 / 未达成（无论失败形态是否相同）→ 停，外显一行「止损：<目标> 两轮未收敛，停止试变体」。绕圈的本质是反复试同一方向的变体，越试越锚死。止损不是再想一个变体，而是列出至今试过的方向，本轮换一个与之正交的方向或抽象层，不许只是上一版的参数 / 措辞 / 局部微调。

## 降格规则：观察是事实，解释全是假设

唯一不降的是**观察到的事实 / 数据**（具体输入 → 具体结果、用户亲眼所见、材料里白纸黑字写的）——它是一切求解要对准的锚。**除此之外全降为待证伪假设**：对现象的解释、定位（是哪块的问题）、根因（为什么）、结论、方案——无论出自用户还是自己，都只是假设，别锚死第一个看似成立的。用户的口头结论、口头根因同样是假设，不因「用户这么说」或「我刚才觉得是它」就当结论跳过验证。问题与答案常在第一直觉之外。一句话常混事实与假设，拆开对待。

## 假设纪律：可证伪清单

先一次性列 3–5 条排好序的**可证伪**假设，再动手验任一条——只生成单条会锚死在第一个看似成立的想法上。每条写出预测（若 X 成立，则做 Y 会怎样）；写不出预测的是直觉，磨快或丢掉。排好的清单先给用户看一眼，他常凭领域知识一句话重排或排除（不阻塞，用户不在就按自己排序继续）。验证时一次只动一个变量；优先做廉价的缩空间动作（看新近改动、对照一个确实成立的相似案例、逐差异列举）。
