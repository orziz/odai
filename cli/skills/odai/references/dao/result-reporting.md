# 结果总结与技能改进候选

## 收口原则

按任务风险和用户要求决定展开程度，不按固定字段填表：

- 轻量读 / 查 / 微改：只报结果、证据来源和验证方式；没做、没验就明说。
- 实现、测试、审查、文档回写、模块切换或维护：说明完成了什么、验证了什么、还有什么未验证或有风险。
- 复杂任务、blocked、权限 / 外部系统 / 正式审查、跨模块主路冻结或用户要求“完整 / 正式”时：展开目标与边界、关键证据、改动 / 结论、验证、未完成项、风险与下一步。
- 测试、页面、构建或其他验收因环境无法运行时：显式标 `状态：ready`，列已核证据、未验证项、原因和最小解除条件；不得只说“未跑测试”，也不得把静态核对写成 verified。

只回报真实发生的读取、命令、验证、agent / 模型或外部技能；未读、未跑、未调用的不得列入。agent、模型、外部技能只在真实调用时说明；宿主未暴露能力时说明能力缺口，不伪造参与者。

主文件已落盘的总结，须保证主文件自包含；结尾只注明“本任务已沉淀至主文件，任意会话说「继续」均可接续”，不指挥用户管理会话。轻量收口不附此句。

主文件已落盘时，收口未完成可恢复项按 `references/dao/dao-shu-fa-playbook.md` 的「后续队列」写回当前任务主文件；不得把未确认建议、风险观察或技能改进候选自动写成可执行待办，也不得用入队代替当前范围内仍可继续完成的动作。若本轮新增、更新或完成队列项，结果总结必须给出可点击主文件路径，并附队列状态摘要。

后续队列无法落盘时，按 `references/dao/dao-shu-fa-playbook.md` 输出「未落盘后续队列」；该段必须可被上下文压缩完整保留，并提醒后续优先写入主文件或让用户确认丢弃。

## 技能改进候选

非轻量任务暴露出换个任务仍会复发的规则空白、冲突或 stale 规则时，可附一条最关键候选：目标规则 / 文件、建议改法、证据和风险。候选只提议，不自动改 skill，不触碰精神内核与既定纪律。

判据是可泛化而非发生次数；一次但明显可复发即可提，纯本任务偶发或普通功能建议不提。技能改进候选属于维护 backlog 线索，不写入用户业务任务主文件的 `## 后续队列`，也不得作为当前业务任务的可执行待办推进。

用户采纳后按 `references/dao/local-overlay.md` 默认写本地叠加层；是否提升进源是维护者另行决定。真实性硬法统一按 `references/dao/interaction-contract.md`；本文件不得另写宽松例外。
