# Diagnose Kit

## 何时读取

进入 bug / 报错 / 异常 / 行为不符 / 性能回归诊断主阶段时读取；通常由 `道` 判入，或由 `implement-code` 连续两轮未达成验收后转入。先遵守 `references/dao/inquiry-discipline.md`，本文件只补硬 bug 诊断纪律；证据账本用 `references/dao/review-kit.md`，回归落 `implement-code` / `tdd-kit`。

## 铁律

先建红信号闭环，再谈假设。没有一条已实际跑过、能对准用户症状变红的命令，就停在建闭环；不得冻结根因或实施修复。可读代码、日志、配置和相似实现来构造复现、定位观测点和形成可证伪假设，但这些都不能替代红信号闭环。新增回归测试与修复必须分步：先让测试 / 命令在未修复实现上变红并记录红因，再最小修复转绿；不得把测试和修复合成一个未见红的 patch 后直接宣布完成。

停在建闭环不是 blocked：可继续只读补证、读相关实现、要复现条件、设计最小红信号命令、提高偶发复现率。只有补证和提问后仍无可跑路径，才按契约 blocked。

## 闭环准入 4/4

外显「诊断闭环 4/4」前，必须能指名已跑命令和输出，并满足：

1. 能变红：打到 bug 路径，断言用户确切症状，不是“没报错”。
2. 确定：同输入同判定；偶发 bug 已抬到可调试复现率。
3. 快：秒级优先，分钟级闭环要继续收紧。
4. 可无人值守：需要人点的，用一次性 HITL 脚本把人纳入闭环。

缺任一条，继续建闭环，不进入假设。

## 建闭环菜单

失败测试 → curl / HTTP 脚本 → CLI + 夹具 / stdout diff → 无头浏览器断言 DOM / console / network → 回放 trace / payload → 最小子系统 harness → property / fuzz → `git bisect run` → 新旧版本或配置差分 → 人在环脚本。

菜单里的 curl / CLI / 浏览器 / harness / HITL / fuzz 等临时脚本、夹具和输出默认不落用户项目；完整准入、cwd 与 tmp 处理按 `references/dao/interaction-contract.md` 的「临时产物规则」执行。

有闭环后继续打磨：更快、更尖、更确定。偶发 bug 先循环、并行、加压、缩时窗、注 sleep；50% 可调，1% 不可调。

## 排查纪律

1. 用户观察到的现象是唯一不降格事实；坏在哪、为什么坏、怎么改全是待证伪假设。
2. 闭环两轮未变绿就是止损信号；假绿以用户实测为准重建闭环，不拿自己那条绿路径顶替。
3. 变红后最小化复现：逐条删输入、步骤、配置、数据，每删一条重跑，保留“再删就绿”的最小集。
4. 优先看新近改动与同仓可工作的相似实现；逐条列差异作为假设来源，但不替代红信号闭环。
5. 假设清单按 `inquiry-discipline` 执行；每条预测写成“若 X 是因，则改 / 观测 Y 会怎样”，否则丢掉或磨快。
6. 探针优先 debugger / REPL；日志只打能区分假设的边界，带唯一前缀并收尾 grep 清净。性能回归先量基线、profiler、查询计划，再二分。

## 收口

回到 `review-kit` 诊断结构与 `implement-code` 收尾门：原始场景不再复现，回归测试或“确无接缝”已说明，探针清净，最终成立假设写进 commit / PR 供后续复用。
