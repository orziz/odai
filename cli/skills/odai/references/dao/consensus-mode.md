# 合议增强模式

## 何时读取

仅在用户显式激活 `增强模式` / `合议` / `多模型复核` / `开合议`，或按 `references/dao/decision-challenge.md` 征询后用户同意时读取。默认路径不加载。本模式前提是宿主真实暴露多模型或可下放 agent；不得伪造。

## 席位与用法

- 默认 3 席；用户指定席位数和模型时从用户。
- 至少 2 种异构模型，最多 1 席与主流程同模；不得用同模型副本、换 agent 名或换提示词冒充异构。
- 点名模型不可用、宿主没有多模型能力，或无法确认真实异构席位时，先说明能力缺口并让用户选择退化路径：按当前可用最高档做单席复核、退回主流程内显式反证 / 一席挑战（不声称已调用 agent 或模型）、或等待可用模型；不得自动降级后直接输出结论。
- 重大决策 / 方案合议：题面锁定、结论未冻结时，各席独立找反例、漏判、备路和不可接受结果；主流程收束，合议不替代用户确认。
- 待冻结方案复查：主路 / 边界 / 验收已形成待冻结结论时，至少 1 席独立挑战，主流程采纳 / 部分采纳 / 驳回后才能冻结。
- 执行分工：方案冻结后，把可切分子任务下放给有当前证据支持的匹配模型；单席只承接冻结题面、边界与验收，不得擅改路线。

## 下发与收口

每席一包：冻结题面 / 子任务范围、本席位模型与选择理由、关注点 / 切入点、同版 `odai` 契约与 `references/dao/agent-governance.md` 边界、输出格式与收口要求。

结论冲突时不按多数票，按证据强弱和是否动摇道 / 术 / 边界裁决；足以动摇上游的，回写重定。收口写明实际席位数、模型、分工、采纳 / 驳回结论；未真实派起的模型不得列入。宿主只给同系模型、给不全点名模型或无 agent 能力时，均先问用户选退化路径；未选择前最多做只读能力说明，不做主流程反证结论。
