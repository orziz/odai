# Review Kit

## 何时读取

由 `道` 触发；覆盖证据账本、复核、定案、执行单与结果总结。正式三省六部复核以 `references/modules/review-sslb.md` 为准。

## 最小规则

1. 建证据账本：确定项（已知 + 已验证）、待验证项、待确认项、证据、风险。新信息落格，被推翻的旧判断回写。
2. 先判断用户说的是目标、手段、抱怨、根因猜测还是已验证事实；不要拿错层级。
3. 证据不足不硬判严重；先写成待确认、待验证或留中待问。
4. 实施或已有改动后，按 `implement-code` 的验收映射做有判别力、成本合理的验证；环境不能验收时列清未验证项，不以编译凑数。

## 复核要点

- 交叉比对用户输入、项目文档、代码、日志、测试结果和历史结论是否一致。
- 用户输入疑似缺标点、乱标点、错别字、多字、少字、语序断裂、指代错位或语义误写时，复核纠偏理解是否保留原话、说明断句 / 分条或修正依据，并把影响治理点的纠偏列为待确认项。
- 多条、混合或低结构输入要复核 `references/feature-plan/planning-playbook.md` 的需求条目账本：原始条目是否使用统一覆盖状态，新增建议是否有原始条目、项目证据、风险或主动扩展依据。
- 复核账本是否漏拆：每个用户原话中的独立动词、对象、约束、验收点、非目标、目标文件和显式引用都应有单独状态；实施后还要核对每个原始条目与目标文件 / 引用点是否有 diff、验证或不处理理由。
- 闭合四组证据：题目层级（目标 / 手段 / 根因猜测）、目标与非目标、改动 / 授权 / 验收边界、关键环境事实及 UI 基线。
- 任一缺口会改写路线或验收，就回契约补证或提问。
- 工作草案最少包含：当前理解、权衡偏好与不可接受结果、确定 / 待验证 / 待确认项、目标与非目标、风险、路径问题、下一步。
- bug / 异常另拆：已确认现象、疑似原因、已有证据、缺失证据、下一轮最有价值排查动作。

## 严重度

- 严重：会影响方向选择、导致明显错误、引入高风险副作用，或实施前必须先修。
- 建议：不立即阻塞，但影响质量、维护性、协作成本或后续扩展。
- 无问题：当前范围内未发现足以提出意见的问题。

证据不足以支撑“严重”时，降为建议或留中待问。连续 N 轮 clean、全仓 / 全目录复查、安全 audit 或长修复后反复验证，交 `review-sslb` + `references/review-sslb/audit-loop.md`。

## 定案、执行与结果

复核后给明确去向：继续理解、输出执行单、直接推进、实施后复核。

执行单最少包含：起点与目标、本次范围与非范围、实施步骤或改动清单、验证方式与验收标准、风险与回滚、需同步的文档或协作事项；自包含到新会话可继续。

直接实施最低条件：目标明确、边界清楚、输入具备、环境可执行、无待确认项、不会造成明显不可逆副作用；UI 任务还需设计基线、复用策略和主要状态要求已明确。

结果总结按 `implement-code` 核验目标偏离、风险、副作用、测试 / 文档、已跑与未跑验证、验收点与实际场景，再按 `references/dao/result-reporting.md` 回报。只有验收口径成立才标 `verified`；实现完成但环境限制验收时标 `ready`；无不越线后续动作才标 `blocked`。
