# Delivery Playbook

总纲：先辨现象、边界与证据，再定交付与后续；未明者只列待确认，不强行落责。

## 何时读取

- 需处理 bug、报错、异常现象或偶发问题
- 需判断输出用户文档、AI 文档、二者兼备，还是只停临时草案
- 需判断何时只做文档交付、何时可继续进入受控执行

## 最小判路顺序

1. 若当前是 bug、报错或异常，先拆“已确认现象 / 用户猜测 / 已有证据 / 缺失证据 / 次步排查”，不要把用户口头原因直接写成根因。
2. 先判输出类型，只能先选四选一：只输出用户说明、只输出 AI 任务草案、两者都要、先停临时草案。
3. 只要会改方向、边界、验收或风险，就先问用户或补证据；其余能自证的事实先自证。
4. 即使用户已授权执行，本 playbook 也只负责把执行边界整理成明确草案，不进入非文档实施。
5. 若用户不会描述问题，就先替他把现象、猜测、缺口与次步整理出来，再让他修正。
6. 若拿不准，保守默认“先继续理解或输出草案”，并把需用户拍板的关键歧义结构化列出。

## BUG 场景

若用户问的是 bug、报错、异常、线上问题、兼容问题、偶发问题或“哪里不对劲”，不要默认用户描述就是准确根因。

先区分四事：

1. 用户说的是现象、猜测原因，还是已验证根因
2. 问题是稳定复现还是偶发
3. 问题发生在哪个环境、哪个步骤、什么输入条件下
4. 当前是否已有日志、截图、录屏、报错、请求记录或对比样本

默认处理顺序：

1. 先还原现象，不直采“原因”。
2. 主动怀疑用户给出的原因、责任归属、层级判断与修复方向，查其是否和现有证据相冲。
3. 先判更像实现缺陷、需求未对齐、数据异常、权限限制、环境差异、时序问题，还是用户预期与系统设计不一致。
4. 分层排查：前端、后端、接口、状态、权限、缓存、环境差异、时序等；仍不明确者转成必问。
5. 一次收集关键证据，如复现步骤、触发条件、具体表现、日志、截图或录屏。
6. 主动扩展相邻根因、边界条件、环境差异与隐藏触发条件，不只围着用户给出的单一路径排查。
7. 若证据不足但可先排查，就给分层排查路径，并说明下一轮最值动作。

凡足改根因判断、修复范围、执行边界、验收标准或风险等级之不确定，都必须先问用户或补证据。

## 输出策略

可选结果只有四种：

1. 只输出用户说明：主要是帮助用户理清需求、做决策、补材料、确认边界。
2. 只输出 AI 任务草案：边界已较清楚，用户更需要一份可直接交给其他 AI 或 workflow 的任务单。
3. 同时输出两份：既要给用户拍板，也要给 AI 直接执行。
4. 两份都不完整输出：当前仍处关键理解阶段，此时只输出“今判 + 临时方案 + 必问”。

判断标准：用户是否明要推进执行、当前边界是否已稳、主要受众是人还是 AI、若现在就交给 AI 是否会跑偏、对真实需求与题目定义的理解是否已收稳。

若当前已能形成工作草案，就先写草案，再在后续轮次更新；承载位置按任务需要选当前对话或项目 Markdown。

## 先补证据还是先提问

- 任何不确定项都必须先问用户或补证；不可只盯会改方向/边界/验收/风险/数据语义/交互语义/交付对象之项。
- 能从项目、材料、代码、日志、现有文档或低风险验证里自行补齐的事实，优先先自己验证。
- 若存在多种未定方向，只能写成待定路/待验项/风险项，不得伪装成事实，更不得自挑一条默认继续。
- 若用户说法与项目事实/现有代码/现有文档冲突，必须先指出冲突，再定后续交付表述。
- 若疑用户说的是“想怎么做”非“真正要解决什么”，或“猜测根因”非“已验证根因”，先修题面，再断是否产出 AI 执行文档。

## 确认后交付与执行

fp 只负责理解、对齐与方案，不负责实施。

默认顺序：

1. 先判本轮动作是继续理解、只读分析、补齐用户说明，还是补齐 AI 任务草案。
2. Markdown 草案、项目文档落地、只读分析、用户说明整理与 AI 任务草案整理可直接进行。
3. 当前若尚未收清边界，默认先继续结构化提问，再补齐方案草案。
4. 用户即使明确要求进入实现层，本 playbook 也只负责把执行边界、输入、风险与验收标准整理成可交接草案。
5. 若关键前提仍未闭合，则先停在用户说明、AI 任务草案或待定项，而不是硬执行。

若只差一步确认就能定稿/交付/执行，优先用低成本确认把这一步问完；若不止一步，也一次问清当前层级全部未消除的不确定。

## 用户文档要求

- 默认以 Markdown 承载；若用户指定路径、项目已有文档约定，或本轮目标本身就是沉淀规划文档，可直接写入项目内 Markdown；否则先在当前对话给出。
- 先复述你理解的真实目标、真实想法、权衡偏好与不可接受结果，再展开方案与边界。
- 少术语堆砌，必要术语要说人话。
- 明确目标、边界、风险、方案取舍与次步。
- 告诉用户需要补什么、定什么、协调什么；不要写成纯技术实现清单。

## AI 文档要求

- 默认输出为 Markdown 文本；可在当前对话中给出，也可按任务需要写入项目内 Markdown 文档。
- 明确任务边界、用户真实目标、权衡偏好与不可接受结果。
- 明确输入材料、执行顺序、每一步产出。
- 明确哪些点必须先向用户确认，哪些待定点不能擅自扩展。

若适合继续交给下一轮 AI，可直接附一条可执行指令草稿。

## 推荐指令草稿

```text
请基于以上方案继续执行，范围限定在：<本次确认范围>。

要求：
1. 先复述你将执行的边界与待定点。
2. 信息不足时，先提最少必要问题；信息已足，则直接开始。
3. 按“步骤 -> 产出 -> 风险”推进。
4. 未经确认，不得擅自扩展需求。
```
