# Workflow Kit

## 何时读取

当你需要以下任一信息时，再读取本文件：

- 默认应该把工作流结果落到哪里
- 主文件与执行单该如何分工
- 当前阶段与状态该怎么判断
- 同一任务下次该从哪里续跑
- 何时适合切到 fp 严格模式或 strict-sslb 模式

## 默认工作流骨架

```text
接单判路 -> 恢复上下文 -> 起草 -> 过堂 -> 定案 -> 推进 -> 收口
     \            \          \       \        \        \        \
      \            \          \       \        \        \        -> blocked
       \            \          \       \        \        -> blocked
        \            \          \       \        -> blocked
         \            \          \       -> blocked
          \            \          -> blocked
           \            -> blocked
            -> blocked
```

说明：

- `接单判路`：判断当前是收敛、诊断、复核还是推进起点
- `恢复上下文`：若是续跑，先恢复主文件、执行单、当前裁决与下一步
- `起草`：把任务收成可审、可续跑的工作草案
- `过堂`：用 workflow 模式或 strict-sslb 模式复核草案、实现或两者结合
- `定案`：给出继续澄清、输出执行单、直接推进或实施后复核
- `推进`：进入实施、持续更新或写执行单
- `收口`：确认结果是否可交付、可续跑、可移交
- `blocked`：当前被关键信息、权限、环境或分歧卡住

## 对用户优先汇报的最小四项

若没有特殊理由，聊天侧优先只稳定汇报下面四项：

1. 当前裁决
2. 你现在最需要知道的
3. 下一步
4. 若阻塞，最小解除阻断条件

只有在续跑交接、复杂阻塞、正式审查或用户明确要细节时，再补：

1. 当前阶段
2. 当前状态
3. 主文件
4. 执行单
5. 借力记录
6. 推荐路径 / 执行边界 / 主要风险

这样用户不需要理解内部机制，也不会被整套流程台账淹没；完整记录优先回写主文件。

## 降负担边界

降低用户负担，不等于替用户乱猜。默认原则是：

- 默认假定用户未必懂你的流程、未必能准确描述问题，也未必知道该选哪种模式；你要先自己分析、给推荐，再只把真正要拍板的事交给用户
- 能从项目现状、现有文档、代码、日志和证据里自己判断的，不要反问用户
- 会改变路线、边界、验收、风险或执行方式的关键缺口，必须压缩成 1 到 3 个问题问清
- 可继续推进的前提是“证据足够”或“假设安全可回退”，不是“看起来差不多”

## 权限速记

harness 默认权限高于单段规划 skill。可直接进行：

- 读、搜、整理、验证
- 文档更新
- 在当前确认路线内的小到中等规模实现改动
- 与当前任务直接相关的测试、lint、build、format

以下情况再问用户：

- 破坏性或不可逆操作
- 大范围重构或路线改判
- 推送、发布、部署、外部系统调用
- 权限、密钥、费用、生产影响相关动作
- 你判断继续做很可能偏离用户真实目标
- 关键事实缺失，继续做会改变路线、边界、验收或风险判断

## 关键澄清闸门

若以下问题拿不准，且不能从现有上下文直接补齐，就必须先问：

- 真正目标是什么，什么不算本轮目标
- 做到什么算完成，什么结果不可接受
- 哪些范围允许动，哪些不能动
- 关键事实是什么：报错原文、复现条件、影响对象、环境版本、权限前提
- 本轮是否允许直接落地；若暂不允许，是只输出分析 / 主文件 / 执行单，还是先停在待确认

只有同时满足以下条件，才允许临时自行假设：

- 假设是局部、可回退的
- 猜错不会改写用户可见行为、对外接口、数据安全、费用或发布结果
- 猜错不会让整条路线换向
- 已明确写成“假设项”

不要把 workflow、strict-sslb、执行单这类内部阶段选择题直接抛给用户；真正该问的是授权和边界，而不是让用户替你选流程。

若只差一步确认就能继续：

- 优先做低成本确认，不要让用户重新描述一遍任务
- 先给当前判断、推荐默认项和继续路径，再让用户拍板
- 若适合固定选项，优先给 2 到 4 个候选与一句影响说明

## 模式切换速查

### 默认 workflow 模式

适用：

- 混合任务
- 边做边收口
- 用户只想少打断、快推进

特点：

- 输出更压缩
- 重点看当前阶段是否还能往前推
- 只保留真正影响推进的意见

### fp 严格模式

适用：

- 当前主要矛盾是需求收敛
- 需要完整用户文档与 AI 执行单
- 用户明确要求“按 fp 起草”

特点：

- 文档结构更完整
- 更强调目标、边界、约束、待确认项和方案取舍
- 若环境支持真实调用，且用户明确提到 `fp` / `feature-plan`，或语义上明显在要“规划文档 + 用户文档 + AI 执行单”那套能力，优先真实借力 `feature-plan`

### strict-sslb 模式

适用：

- 当前主要矛盾是正式 review
- 需要 diff 审查、目录排查、历史文件复核
- 用户明确要求“按 sslb 过堂”

特点：

- 完整保留中书省 → 尚书省 → 六部 → 门下省 → 锦衣卫顺序
- 保留疑似未使用文件筛查
- 保留两张评定表
- 若环境支持真实调用，且用户明确提到 `sslb` / `r-sslb` / `review-sslb`，或语义上明显在要“正式三省六部审查 / 完整过堂 / 严格 review”那套能力，优先真实借力 `review-sslb`

## 真实借力优先级

- 用户未必会说 skill 全名；简称、缩写、口语指代、方法特征描述，都要主动做语义匹配
- 用户口中的 `fp`、`feature plan`、`规划那个`、`出用户文档和 AI 执行单那个`，默认优先匹配 `feature-plan`
- 用户口中的 `sslb`、`r-sslb`、`三省六部那个`、`正式过堂那个`、`审查那套`，默认优先匹配 `review-sslb`
- 场景明显命中完整规划文档 / AI 执行单时，优先真实借力 `feature-plan`
- 场景明显命中正式 review / diff 审查 / 完整三省六部时，优先真实借力 `review-sslb`
- 只有环境不支持真实调用、调用会打断续跑，或问题规模明显不值得切换时，才退回 harness 内部模拟
- 若语义上可能命中多个候选，但结合任务阶段、产物类型和用户意图仍能判断，就直接选最合适者；只有仍有真实歧义时再追问 1 个问题

## 默认产物包

若项目没有既有约定，优先采用这套最小产物：

```text
plans/
  <日期>-<任务名>.md
  <日期>-<任务名>.execution.md   # 仅在实施步骤较长或需要协作时创建
```

### 主文件职责

主文件默认承载：

1. 当前阶段与状态
2. 背景与目标
3. 证据账本
4. 工作草案
5. 过堂结论
6. 执行结论
7. 进度与收口
8. 文档更新记录

主文件是工作流主锚点，默认优先续写这里。

### 执行单职责

只有在以下情况才拆出 `.execution.md`：

- 实施步骤较长，主文件会失焦
- 需要多人协作或多轮续跑
- 需要单独跟踪步骤、验证和回滚

执行单最少包含：

1. 当前状态与对应主文件
2. 目标
3. 范围与非范围
4. 前置条件与依赖
5. 实施步骤
6. 验证与验收
7. 风险、兼容与回滚
8. 执行记录与阻塞事项

## 自动落盘优先级

决定往哪里写时，默认按以下优先级：

1. 用户已指定的路径、目录、文件名
2. 对话中已经明确指向的现有主文件
3. 项目现成约定，如 `docs/`、`specs/`、`design/`、`plans/`、`notes/`
4. 默认新建 `plans/<日期>-<任务名>.md`

如果已经形成第一版草案，就不要继续拖着不写文件。

## 续跑优先级

同一任务再次接手时，默认按以下顺序恢复：

1. 用户本轮指定的文件
2. 上轮已经存在的主文件
3. 主文件中引用的执行单
4. 项目内与该任务最匹配、且状态仍可继续的既有文档
5. 若都没有，再新建默认主文件

恢复后先看：

1. 当前阶段
2. 当前状态
3. 当前裁决
4. 下一步
5. 是否存在过期假设或已失效阻断

只有当某一个主锚点明显成立，或多个候选实际上指向同一任务时，才直接续跑。

若同时存在 2 个以上合理候选，且继续做会改变路线、边界或交付对象，必须先用 1 个问题确认具体接哪一个，不要只凭“最匹配”硬续跑。

## 状态速查

- `draft`：方向初步形成，但还有关键待确认项
- `reviewed`：已完成过堂，拿到了裁决
- `ready`：可以按下一步或执行单继续推进
- `executing`：已经进入落地执行或持续更新
- `verified`：已完成验证与收口
- `blocked`：被关键信息、权限、环境或分歧卡住

## 四类常见起手式

### 1. 模糊需求

用户输入示例：

```text
想把支付页改一下，但我自己也没完全想清楚。
```

默认动作：

1. 进入收敛起点
2. 建主文件并先写第一版草案
3. 把问题拆成确定项、假设项、待确认项
4. 最后只问 1 到 3 个真正影响方向的问题，并附当前推荐路径

### 2. 已有方案待审

用户输入示例：

```text
这是我写的方案草稿，帮我看下有没有明显风险。
```

默认动作：

1. 进入复核起点
2. 先抽取最小交接载荷
3. 根据需要走 workflow 模式或 strict-sslb 模式
4. 若能继续，直接落成执行结论

### 3. 已确认路线直接推进

用户输入示例：

```text
方案就按这个来，直接帮我落地。
```

默认动作：

1. 进入推进起点
2. 判断是否满足执行条件
3. 若满足，直接推进并更新主文件
4. 若不满足，改成执行单或低成本确认

### 4. 只说“继续”

用户输入示例：

```text
接着上一个来。
```

默认动作：

1. 优先寻找已有主文件与执行单
2. 读取当前阶段、当前状态、当前裁决、下一步
3. 若主锚点唯一则继续推进；若存在多个合理候选，先用 1 个问题确认
4. 回写本轮变化与新的下一步，而不是重问背景
