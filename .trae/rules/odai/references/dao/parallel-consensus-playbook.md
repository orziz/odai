# 并行合议手册

## 何时读取

当你需要以下任一信息时，再读取本文件：

- 判断当前环境能否进入多 agent 多模型合议模式
- 需要自动拉起多个 agent、不同模型做方案合议
- 需要在方案冻结后再组织独立复查
- 需要决定如何收敛分歧、何时交用户复核、何时回退原流程

## 两种模式

- 合议增强模式：在完整能力成立时，把原流程里的问题理解、问题复核、方案收束与冻结后复查交给多 agent、多模型并行增强
- 原流程模式：沿用当前模块原有工作流；若模块自身已有并行辅助分析口径，仍按其原规则执行

新增增强模式不会改写原流程。它只是把多 agent 多模型能力叠加到原流程同一条链路上，不会把“先理解问题、再向用户确认、再继续规划与执行”的顺序改掉。只要完整能力不成立，就保持原流程，不得虚构多模型参与。

## 完整能力成立条件

以下条件必须同时满足，才算“当前环境支持”：

1. 运行环境被明确识别为 Copilot。
2. 当前环境明确支持并行辅助分析或并行 agent 调用。
3. 当前环境明确支持为并行任务分配模型，并保证本轮并行任务组至少包含两种不同模型；可包含 1 个与主流程同模型的子 agent。
4. 当前任务不是明显小问题，且并行收益高于额外成本。
5. 当前阶段属于合法并行阶段：要么是在用户确认前用于问题理解、需求澄清与结构化提问准备；要么是在用户确认后用于问题复核、方案收束或冻结方案后的独立复查。

若当前仍处于用户确认前，并行内容只允许用于理解问题、整理证据、准备提问与校验题面；不得在此阶段冻结方案、决定实施主路或进入实施。

任一条件不成立，都视为“不支持”，直接回到原流程模式。不得用角色差异、提示词差异或同模型副本替代真实多模型能力。

## 启动前先做触发判定

默认不开合议增强模式。总控必须先根据任务特征做一次触发判定。

优先开启合议增强模式的强触发信号：

1. 用户明确要求多 agent、多模型合议或冻结方案后的独立复查。
2. 同时存在两条以上合理主路，且取舍会改变范围、边界、验收、成本或风险接受度。
3. 任务跨两个以上专项模块，或一旦误判就会把后续实现、设计或审查整体带偏。
4. 当前裁决属于高代价决策，例如方向定案、规格冻结、架构主路、广范围审查结论。

默认不开合议增强模式的排除信号：

1. 单文件小改、小范围修辞、低风险微调。
2. 现有模式成熟、主路单一、几乎不存在真实替代方案。
3. 即使判断失误，回滚和纠偏成本仍然很低。

若强触发信号不足，就先走原流程；但总控必须保留动态升降档能力。原流程推进中，一旦出现新的主路冲突、跨模块牵引或高代价冻结点，应立即重做触发判定，必要时切入合议增强模式。反过来，若增强模式已快速收敛且剩余风险很低，也可以回到原流程，不为形式保留并行。

触发判定可以在首轮提问前做第一次，也可以在用户确认后结合新增信息再做一次；两次判定结果允许不同。

## 合议增强模式

### 总链路

- 总控先判是否开增强模式。
- 若开增强，先在确认前让多模型并行分析用户真实意图、需求、问题与待确认点。
- 必须输出结构化问题并等待用户确认；增强模式只提高理解质量，不替代提问确认。
- 用户确认后，再做一轮问题复核与再分析。
- 再进入方案规划、内部收束与冻结方案后的独立复查。
- 只有仍会改变路线、边界、验收或不可接受结果的剩余分歧，才交给用户复核。
- 分歧收束后，回到原模块后续执行。

### 阶段零：确认前的问题理解与题面校验

主流程先形成当前问题定义草案，但不得把它当成已确认题面、冻结方案或实施主路。

- 至少要有 1 个并行 agent 负责问题定义挑战或校对；该席位优先使用不同于主流程的模型。若存在与主流程同模型的子 agent，该席位也必须承担不同职责，而不是重复主流程判断。
- 问题定义校验至少要检查四件事：问题定义是否偏题、用户真实意图是否被偷换、范围与非范围是否漂移、现有证据是否足以支撑当前理解。
- 当前阶段对外只允许落到：当前理解、已知事实、未确认点与结构化问题组；不得跳过提问直接给定案。
- 只要问题定义校验指出了会改变主路、边界、验收或不可接受结果的理解偏差，就不得进入后续规划合议；必须先补证据或向用户确认。

### 阶段一：结构化提问与用户确认

- 不论是否进入增强模式，结构化提问与用户确认都是必经门槛。
- 增强模式在这一阶段的目标，是把问题问准、问全、问到会影响路线、边界、验收与不可接受结果的点，而不是绕过用户确认。
- 未得到用户确认前，不得冻结方案、定实施路线或进入实施。

### 阶段二：确认后的再分析与冻结问题定义

收到用户确认后，主流程必须结合确认结果再做一次问题复核；增强模式可在这一轮再次并行调用多模型做重新校准。

主流程先冻结以下内容，再分派并行任务：

- 当前要解决的问题定义
- 当前范围、非范围与权限边界
- 当前成功标准、不可接受结果与验收口径
- 当前可用证据与禁止擅自假设的部分
- 本轮要收敛的核心问题

若用户确认后仍出现新的关键不确定，并且会改写路线、边界、验收或不可接受结果，就回到提问确认，不在未重新确认的题面上继续细化。

### 阶段三：方案规划与首轮合议

- 默认至少 2 个并行 agent，且至少使用 2 个不同模型；复杂任务可增到 3 至 5 个，但不要仅为形式扩大并行规模。
- 并行任务组至少包含 2 种不同模型；可包含 1 个与主流程同模型的子 agent，但不得让整个任务组退化为单模型，也不要复制多个同模型同职责席位。
- 每个 agent 都要拿到同一版已确认题面与边界，形成相对完整判断；允许首轮结论不一致。
- 角色最好错开：主路方案、反例攻击、风险审计、实现代价、复核查漏等；不要让多个 agent 只是重复同一视角。

### 阶段四：主流程收束与定向复议

主流程必须先做三件事：

1. 去重：合并等价结论与重复风险
2. 裁冲突：区分真分歧、表述差异与证据不足
3. 查遗漏：确认是否有任何关键边界、验收或风险被漏掉

能内部收敛的，必须先内部收敛，不把多份原始结论直接交给用户。

主流程在这一阶段只允许做“基于已返回结论与现有证据的收束”，不允许脱离返回结果与现有证据自行扩写主路、边界或风险。具体限制：

1. 不得新增首轮无人提出、且证据账本也不支持的新主路、新边界或新风险。
2. 若要否决多数意见、压过少数异议或改变某项风险定级，必须给出明确依据：项目事实、用户确认、代码或文档证据、测试或日志结果，或定向复议的新结果。
3. 若拿不出依据，就视为尚未收束，进入定向复议或交用户复核，而不是由总控直接裁定。

主流程在宣布“已收束”前，内部至少要对齐下面四项：

- 哪些是共识项
- 哪些是真分歧
- 每个关键判断分别由哪些 agent、哪些模型支持
- 每项裁断背后对应的证据是什么

上述四项未对齐前，不要宣称已经形成共识。

若首轮仍有实质冲突，主流程应优先发起定向复议，而不是立刻把问题升级给用户。定向复议只围绕仍会改变以下内容的分歧：

- 主路或备路选择
- 方案边界
- 验收口径
- 关键风险定级
- 不可接受结果

### 阶段五：冻结方案后的独立复查

当主流程已经形成冻结方案、冻结规格或冻结裁决后，若当前环境仍满足完整能力条件，可再拉起一轮独立复查：

- 复查优先使用不同于首轮的 agent 组合与模型组合
- 复查输入优先给冻结后的方案、关键证据和当前裁决，不直接转交整段讨论历史
- 复查目标是验证遗漏、边界与验证链条是否成立，而不是重复首轮表述

### 阶段六：回到原流程执行

当问题定义、方案主路、边界与验收已经收束后，回到当前模块原流程继续规划落法、实施、验证或审查。执行中若出现足以动摇上游判断的新证据，仍要回写上游，而不是把增强模式做成一次性前置仪式。

## 两类用户确认不要混淆

- 理解确认：这是原流程与增强模式都必须经过的门槛，发生在规划和实施之前；增强模式只能帮助把这一步问得更准，不能跳过它。
- 分歧复核：这是增强模式内部收束失败后的升级动作，只在完成内部收束或定向复议后、仍存在会改变路线与边界的剩余分歧时才交给用户。

## 何时才交给用户复核

只有满足以下全部条件时，才把分歧升级给用户：

1. 主流程已经完成至少一轮内部收束或定向复议
2. 分歧仍然真实存在，不是措辞差异或信息重复
3. 分歧会实质影响路线、边界、验收、风险接受度或不可接受结果
4. 该分歧无法通过项目事实、现有材料或低风险验证自行消除

交给用户时，只提交“最小必要分歧清单”，每项都要写清：

- 分歧点是什么
- 各选项分别意味着什么
- 对范围、成本、风险或体验的影响
- 为什么当前裁断需要用户确认

## 一致性判定

满足以下任一情况，可视为“已达成一致”：

- 所有并行判断在主路、边界、验收和关键风险上已经对齐
- 剩余差异只影响措辞、展示顺序或低优先级建议，不影响当前裁决
- 主流程已用现有证据压平了冲突，且没有留下会改变路线的未知项

未达成上述条件前，不要宣称“已经一致”。

## 主流程约束

- 不得用个人偏好覆盖多 agent 的共同结论。
- 不得无视问题定义校验指出的关键偏差。
- 不得把未被证据支持的判断包装成“综合裁定”。
- 不得把实质单模型结果描述为多模型合议。

## 输出与留痕约束

- 对用户只输出一份统一表述，加最小必要的待用户复核项。
- 不转储多 agent 原始讨论记录，不把过程性争论当主交付。
- 若用户明确要求留档，最多只在 Markdown 中保留最小必要结论、剩余分歧和最终确认结果。
- 不得编造“不同模型”或“多 agent”已参与；能力不明时宁可回退原流程。

## 短规则

- 先完成问题定义校验，再决定是否进入合议。
- 能在内部收束，就不升级给用户。
- 优先用低风险求证消除伪分歧。
- 先收束，再复查。
- 复查尽量与首轮异构。
- 能回退就回退，不得虚构多模型参与。
