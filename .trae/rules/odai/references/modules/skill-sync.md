`skill-sync` 以仓库内脚本 `scripts/skill-sync.js` 为执行体。

## 目标

默认同步统一入口 `odai`；若用户显式传入 skill 名称，也只接受当前仓库里真实存在的源 skill。

这里只司同步分发，不司起草源文件。标准源在 `skills/odai/`；内部模块/support files 亦在此目录。`skill-sync` 校验后只做 Claude/GitHub/Trae 同步与 README 回写；Codex 暂无专用出包，故标准源正文必须保持宿主无关，可被其直接复用。

## 输入约束

1. 默认无参同步 `odai`；若用户显式传名称，也只接受当前仓库里真实存在的 skill 名称。
2. 任一名称无法明确解析为 skill 名称，即提示用户重输。
3. 只认 `工程根目录/skills/<skill-name>/`，不从 `.agents` 取源。
4. 不引依赖，不引其他 skill 名称；统一入口内部模块非独立同步对象。
5. 多名称先全量校验，再逐个处理，避免半同步。

## 执行方式

1. Claude 命令层只做薄封装。
2. 实际生成逻辑统一由 `工程根目录/scripts/skill-sync.js` 完成。
3. 脚本是唯一执行真相；不要再让模型按规则手工生成目标文件。

## 脚本职责

1. 校验 `工程根目录/skills/<skill-name>/SKILL.md`、必要 support files 与 README 关键分节/锚点；关键校验失败即停。
2. 源 frontmatter 与正文为唯一正文载荷；正文段落顺序、标题层级、代码块、措辞原样保留，不重组、不风格化重写。
3. 同目录 `odai/references/`、`odai/assets/`、`scripts/` 皆为当前 skill 附属资源，一并同步到各手动安装目标。
4. Claude wrapper 元数据只认标准源 frontmatter 显式字段；不从现有 `.claude/` 产物反推或保留。
5. 生成手动安装文件时，改写入口与资源相对路径为目标目录真实可用路径，并覆盖 Claude/GitHub/Trae 目标文件。
6. 默认只处理 `odai`；若 frontmatter 有 `replaces`，同步清旧安装产物与 README 条目。
7. 更新 `README.md` 只处理“当前已提供”列表与对应分组落位；不改其他段落，不重写全文。
8. 源正文涉提问工具/确认组件等宿主能力，须写成宿主无关规则；不得把单一宿主工具名写成全平台硬规则。

## 严格限制

1. 不修改 `.agents` 下任何内容。
2. 不扫描或操作无关目录。
3. 不发明新的文件格式，不扩展安装体系。
4. 不为了“参考现有格式”而搜索全仓文件。
5. 如果源 skill 不存在，不做任何写入。
6. 除 `工程根目录/skills/<skill-name>/` 与 `README.md` 外，不读其他 skill 目录；统一入口内部模块属当前 skill 目录，可直接处理。
7. 不为“更像目标平台”重写源正文；正文直拷源文件，惟固定头部、仓库/资源相对路径改写、frontmatter 显式 Claude 包装字段例外。
8. 不调正文章节顺序，不合并段落，不擅增删规则。
9. 不自推平台差异，不自设额外字段；源文件未写、规则未死者，一律不补。
10. 写目标文件时，规则中 `$ARGUMENTS` 均作普通文本占位符；禁止展开、替换或引用当前传参。
11. 宿主提问工具只写宿主无关口径：工具可用且获准时调用当前宿主原生提问工具；不可用、未暴露或上层不允许时文字成组问；不得写单一宿主工具名或将其当全平台定义。

## 失败输出

- 若任一源文件不存在：`错误：工程根目录/skills/<skill-name>/SKILL.md 不存在，无法生成手动安装版本。`

## 成功输出

成功时只简要说明：

1. 处理了哪些源文件；
2. 各自更新了哪些目标文件；
3. README 是已追加、已更新还是已存在；
4. 已按最小必要范围完成同步。
