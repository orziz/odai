此为道下开发分支，专司开发任务收口。

先明所求，再择 SDD、BDD、TDD 主驱。贵在判领跑、切换与回上游补清，不并开三法。

用户输入纵散，先理为已知/未定/相左/缺口，再把关键不确定交用户确认。补看相邻模块/失败路径/隐藏成本/替代路；所得只留待验/待定/风险。

此模块由 `dao` 分流而来，不是并列总控。`project-guide`、`feature-plan`、`design-spec`、`review-sslb`、`implement-code` 皆按需调用，非前置依赖。

本支只补主驱动、阶段流转与转段条件。待定项不清，不入实施；主路与验证既定，则直推收束。

## 最小工作骨架

本 skill 可独立运行；当需要支撑文件补充说明时再展开 workflow-kit 或 review-kit。

```text
所求：
所重：
当前主驱动：SDD | BDD | TDD
当前主阶段：明确问题 | 诊断 | 复核 | 推进 | 收束
证据账本：
- 确定项：
- 待验项：
- 待定项：
- 风险项：
今断：继续理解 | 起草主文件 | 先做复核 | 直接推进 | blocked
所用：无 | feature-plan | design-spec | review-sslb | implement-code | project-guide
次步：
驱动切换条件：
解阻：
```

执行要点：

1. 主驱动、主阶段、今断与调用对象皆先只选一，不多向混跑。
2. 用户口头方案/修法/归因，仅记“用户当前表述”；先拆目标/手段/事实/猜测/偏好，真实意图未坐实不定路、不实施。
3. 低结构输入先代整已知/未定/阻点/必问；不把整理问题丢回用户。
4. 用户若只要草案/只停分析/只到文档，但所指任务、产物或完成态仍指代不清，先问清再停；不得把“先别继续”误判成边界已清。
5. 判主驱动：界未稳（范围/非目标/验收/依赖有歧）走 SDD；行为未稳（流程/状态/反馈/异常不明）走 BDD；行为已稳且逻辑险高/回归贵/需重构护栏，或用户明言测先行，走 TDD。
6. 非每事皆经 SDD → BDD → TDD；仅命中切换条件方切，不为方法完整强加阶段。
7. 涉页面/UI/交互/视觉/一致性/状态体验/design system/组件/token 时，先判实现偏差、旧基线未复用、或基线自缺；用户若明要提审美/高级感/品牌感/Apple 感/Google 感/标杆站质感，或现状显粗陋/廉价/拼贴，则先转 `design-spec` 立视觉标尺与复用清单；未明不按纯实现处理。
8. 涉实现结构，先判历史遗留未删、重复未并、补丁叠补丁、边界已坏，抑确需抽象；未证不增封装/管理/适配/组件层。
9. 实施前待定项须清空，且边界稳、风险可控；入 TDD 前，目标行为还须稳定且可测。
10. 用户明点 SDD/BDD/TDD，优先尊之；前提未立，先补足，不跳过上游。
11. 阶段/主文件/继续推进读 `workflow-kit.md`；证据/提问/复核/定案读 `review-kit.md`。

## 工作流原则

1. 先校题：真目标/成标/表层诉求/用户手段/约束声称/证据/权衡/不可受结果。
2. 对事实须疑，对意图须善；勿因一处事实口误误判真实诉求。
3. 补看目标/场景/角色/流程/边界/依赖/风险/成本/替代路/长期影响；未知者补证或发问。
4. 项目、代码、文档、日志与低风险验证可直证者，先自补；不能补者，再问。
5. 推进前先判减法：删旧、并分支、收接口/状态/层级、拆临时旁路能否解题；可则不新增包装/管理器/适配器/通用组件。多轮修改后，必须扫死代码、旧双轨、废状态与假通用层。
6. 多路线者，先判用户权衡与最不可受结果，再定路。
7. 项目基线 → `project-guide`；规格边界 → `feature-plan`；行为体验与高审美基线 → `design-spec`；正式复核 → `review-sslb`；落地实现 → `implement-code`。主矛盾命中其主责即切，不得久停本模块代做；`*-strict` 仅表严格继承。
8. 未确认内容只留草案/证据账本/复核结论，不带入实施。用户要完整 workflow，则本轮范围内一次推至收束，不停半成品。

## 三驱动路由原则

三驱动对应三层：SDD = 道，BDD = 术，TDD = 法。判主驱动，就是判当前卡层。

1. `SDD` 司把问题、目标、范围、输入输出、依赖、非目标、成功标准与风险写成可执行规格；常借 `feature-plan`，项目级约束明显时先补 `project-guide`。
2. `BDD` 司把上游规格压成用户行为、关键场景、触发动作、系统反馈、状态变化与验收口径；常借 `design-spec`。
3. `TDD` 只在目标行为已稳，且可用自动化测试表达失败时启用；常借 `implement-code` 的测试先行模式。
4. `SDD` 至少答：做什么/做到哪/非目标/验收；`BDD` 至少答：谁/场景/动作/反馈/状态/例外；`TDD` 至少答：先坏哪测/最小实现如何过/回归如何验。
5. 低风险样式微调、文案调整、一次性脚本或资料整理等任务，不强制补齐三层；够用即可。

## 默认流转与主文件

1. 默认流转：接单判路 → 恢复上下文 → 起草 → 复核 → 定案 → 推进 → 收束；无不越线动作方入 `blocked`。
2. 可写且本轮已成草案/复核/执行单/收束时，默认落同一 Markdown 主文件；无约定用 `plans/<日期>-<任务名>.md`；步骤变长再拆 `.execution.md`。
3. 用户言“继续”等，优先恢复既有主文件、执行单、今断与次步，不索重述背景。
4. 调用专项后，若产出足支下一阶段，即转其输入续推，直至本轮处理完或遇真阻。

## 权限与越线边界

1. 待定项未清前，可做搜索、阅读、文档回写、只读验证与补证；待定项清空后，方可做当前范围内代码 / 配置修改，与构建、测试、lint、format、验证。
2. 仍有不确定时，先补证据、起草或提问；只要待定项未清，就不入实施。
3. 单个工具、命令或环境检查失败，不自动等于 `blocked`；先换等价命令、缩小验证范围或继续能做之非实施动作。
4. 破坏/不可逆、大重构、项目级依赖或根配调整、发布/部署、外部系统、费用动作、认证/数据安全相关改动，及明显越界实现，皆须先问用户。
5. 更高权限不等于更鲁莽；此 workflow 之务，在先清待定项，再把整轮任务持续推进到可交付结果。
