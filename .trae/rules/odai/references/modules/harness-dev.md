道下开发分支，司开发任务推进与收口。先明所求，再择 SDD/BDD/TDD 主驱；只判领跑、切换、回上游补清，不并开三法。

此模块由 `dao` 分流而来，不是并列总控；按需选择主调用对象，并可按阶段串接 `project-guide`、`feature-plan`、`design-spec`、`review-sslb`、`implement-code`。待定项不清不入实施；主路与验证既定则直推到明确动作与验证收束。

## 最小工作骨架

```text
所求 / 所重：
主驱动：SDD | BDD | TDD
主阶段：明确问题 | 诊断 | 复核 | 推进 | 收束
证据账本：确定项 / 待验项 / 待定项 / 风险项
今断：继续理解 | 起草主文件 | 先做复核 | 直接推进 | blocked
所用：无 | feature-plan | design-spec | review-sslb | implement-code | project-guide
次步 / 驱动切换条件 / 解阻：
```

## 执行要点

1. 主驱动、主阶段、今断、调用对象先只选一，不多向混跑。
2. 用户口头方案/修法/归因只记当前表述；先拆目标/手段/事实/猜测/偏好，真意未稳不定路、不实施。
3. 低结构输入先代整已知/未定/相左/缺口/必问；不把整理问题丢回用户。
4. 用户只要草案/分析/文档但任务、产物或完成态不清，先问清再停；不得把“先别继续”误判为边界已清。
5. 可由项目、代码、文档、日志、低风险验证直证者先自补；不能补再问。
6. 实施前待定项须清空，且边界稳、风险可控；入 TDD 前，目标行为还须稳定且可测。
7. 页面/UI/视觉/design system/组件/token 先判实现偏差、旧基线未复用或基线自缺；审美升级/高级感/品牌感/标杆质感或现状粗陋廉价时转 `design-spec` 立标尺。
8. 实现结构先判删旧、并分支、收接口/状态/层级、拆旁路能否解题；无证据不增封装/管理/适配/通用组件。
9. 用户明点 SDD/BDD/TDD，优先尊之；前提未立则补足，不跳上游。
10. 阶段/主文件/继续推进读 `workflow-kit.md`；证据/提问/复核/定案读 `review-kit.md`。

## 三驱动

- `SDD`（道）：问题、目标、范围、输入输出、依赖、非目标、成功标准、风险；常借 `feature-plan`，项目级约束明显先补 `project-guide`。
- `BDD`（术）：用户行为、关键场景、触发动作、系统反馈、状态变化、验收口径；常借 `design-spec`。
- `TDD`（法）：目标行为已稳且可用自动化测试表达失败时启用；常借 `implement-code`。

非每事皆经 SDD -> BDD -> TDD；低风险样式、文案、一次性脚本、资料整理够用即可。

## 默认流转

接单判路 -> 恢复上下文 -> 起草 -> 复核 -> 定案 -> 推进 -> 收束；此为状态顺序，不是必经循环。先定本轮完成态，已无不越线动作方入 `blocked`。

可写且本轮已成草案/复核/执行单/收束时，默认落同一 Markdown 主文件；无约定用 `plans/<日期>-<任务名>.md`，步骤变长再拆 `.execution.md`。用户言“继续”等，优先恢复既有主文件、执行单、今断与次步，不索重述背景。调用专项后，若产出足支下一阶段，以其产出作为下一动作输入；每次推进须有可见状态变化、验证结果或待定收束，不在同一判断上连续自转。

## 权限边界

待定项未清前，可搜索、阅读、文档回写、只读验证、补证；清空后方可做当前范围内代码/配置修改与构建、测试、lint、format、验证。单个工具/命令/环境检查失败，不自动 `blocked`；先换等价命令、缩小验证或继续非实施动作。破坏/不可逆、大重构、项目级依赖或根配、发布/部署、外部系统、费用、认证/数据安全相关动作，先问用户。
