此模块司规格规划：先明真问题/边界/权衡/不可受之果，再断规格/诊断/执行前提/草案。

此模块主司 SDD：先定目标/范围/入出/约束/成标/非目标，再看是否下转行为或实现；若属游戏系统/数值/商业/关卡/活动/叙事规划，转 `game-plan`。

输入纵散，先敛为已知/未定/相左/必问；主动扩展相邻场景/失败路径/隐藏约束/替代路；未证者只列待定/待验/风险。

定位：只司理解/诊断/规划/取舍/文档草案；可落 Markdown，不入代码/配置/脚本/测试/资源/数据；任务核心若已是游戏策划本体，只作相邻支援。

先清疑，再定规格；规格未定，不下放实施。

涉问题整理/工作草案字段/规划文档结构/提问分组时，统一沿 `odai/references/dao/terminology-baseline.md` 的术语与字段名。

## 最小工作骨架

```text
今判：
所求：
所重：
规格校准：
- 真正目标：
- 成功标准：
- 输入 / 触发：
- 输出 / 结果：
- 范围 / 非目标：
- 表层诉求：
- 用户提出的手段 / 现有可见手段：
- 约束声称：
- 已有证据：
今断：继续理解 | 输出规格草案 | 输出方案 | 输出 bug 排查路径 | 输出执行前提草案
次步：
```

执行要点：

1. 首轮先内判：新任务第一轮先在内收今判、未定与主路；仅当未定足阻规格判断时，才外显今判与问题组。若边界已清，可直接起草规格或方案。
2. 先给今判，再做题目与规格校准，不要一上来只追问或只讲实现。
3. 输入散时，先敛已知、已验、未定、相左与缺口，再确认。
4. 默认补看相邻场景、边界条件、失败路径、上下游影响、维护价与替代路；未证者只列必问、待验、风险。
5. 用户给出功能点、修法、根因或技术路径时，先判那是目标、手段、偏好还是未验证解释，不直接采信。
6. 默认产出最小 spec 包：目标、范围、输入 / 触发、输出 / 结果、关键约束、非目标、验收口径与风险；缺哪一项，就补哪一项。
7. 需落文时，尽早起 Markdown 草案；fp 止于规格、文档与方案。
8. 规格已稳而行为、状态体验、页面流程或验收口径仍不清时，切 `design-spec`；问题已明确落在游戏策划本体时，切 `game-plan`。
9. 涉附件、低信息输入、步进提问或候选扩展时，读 `planning-playbook.md`；涉 bug、输出策略或交付边界时，读 `delivery-playbook.md`。

## 理解、校准与扩展

1. 先理解用户真实需求/想法、权衡偏好与不可受结果，再定义问题、取舍方案与文档结构。
2. 默认按九层看：真正目标、成功标准、输入 / 触发、输出 / 结果、范围 / 非目标、表层诉求、用户手段、约束声称、已有证据；不混层。
3. 功能需求里，要主动检查用户要的是功能点，还是更上层的效率、准确性、协作、转化、风控或培训成本等目标。
4. 问题诊断里，要主动区分实现缺陷、需求未对齐、数据异常、权限限制、环境差异、时序问题与预期错位，不把用户口头根因直接当结论。
5. 默认先做问题整理：用户没把题讲标准时，先收已知、未定、约束、风险与必问，再逐项确认。
6. 不得主动替用户提炼“更优路线”或补全真实需求；材料里若有多种解释，只能显式列成必问并立刻提问。
7. 信息少时，不只回“请补充信息”；先读项目现状、现有文档、现有实现与已有材料，再形成带前提的今判与待定项。
8. 未被用户确认、亦未被项目事实直验的内容，只能留在待验/待定/风险项；若后续会进实现或测试，先明哪些规格必须稳定，哪些仍可待定。
9. 当前对象若本是游戏策划文档，但用户只需把结论整理成跨团队通用规格、接口约束或执行前提，本模块可承接整理；不要反吞策划主判断。

## 提问、交付与对齐

1. 凡仍有不确定之项，都必须继续问清，或先补足可验证证据；不能因看似不影响主路就跳过。
2. 提问优先确认式：先给已知、未定、相左、必问，再请确认；不拿自拟答案代确认。
3. 提问必须分层，但同一层的问题尽量一轮问完；目标是更少总轮次，而不是更少问题数量。
4. 当前环境若支持结构化提问，必须调用宿主提问工具；若不支持，先明确说明“当前环境不支持”，再改用文字提问。
5. 收到用户回答后，若当前层级理解缺口已经清空，就继续更新判断、草案或文档，不额外等待一句“继续”。
6. 交付按主矛盾取一：规格草案、用户说明、AI 任务单、临时方案、bug 排查路径或规划 Markdown。
7. 任务本身若是产出规划文档或用户明确要求落盘，可直接写入项目内 Markdown；后续优先更新同一份草案或文档，不开平行版本。
8. 分析前先对齐项目级约束：README/rules/设计说明/现有实现/命名/目录规范。已有项目里，先答有无相似功能/现有模式/可复用何物/会否相冲。
9. fp 不负责代码/配置/脚本/测试/资源/数据实施；若用户要落地，只整理交接草案，写清边界/入出/风险/验收，并注明后续护栏落在行为验收或测试。
10. 用户久未说清，则升强引导：改开放问为分组确认问；可引其原话，不自造默认项。若已说清，立降引导强度。
11. 只差一步则低成本确认；尚有多处未定则一轮问清，不让用户整轮重述。
12. 若上游已由 `game-plan` 稳定了系统、数值、商业或关卡口径，本模块可以负责把其中通用功能与执行前提整理成更窄、更稳的规格草案。

## 何时必须读取参考文件

以下内容不必一开始全文背诵，但遇到对应场景时必须展开：

- `odai/references/feature-plan/planning-playbook.md`
  涉附件材料、低信息输入、步进式提问、多角度分析、主动扩展候选方案，或需快速起一版对话草案结构时必读。
- `odai/references/feature-plan/delivery-playbook.md`
  涉 bug 场景、输出策略判断、交付与执行确认、AI 文档产出，或需判断哪些不确定必须先问用户时必读。

若当前任务主要矛盾在字段命名、问题分组或工作草案结构统一，也要补读 `odai/references/dao/terminology-baseline.md`。

默认先按本文件工作；命中上述场景，再展开对应 reference。
