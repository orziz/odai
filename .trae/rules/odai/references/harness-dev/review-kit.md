# Review Kit

## 何时读取

- 需维护证据账本、提问与阻断规则
- 需主动怀疑用户输入、现有文档、现有代码与历史结论
- 需判工作草案最低写法、严格模式、复核顺序、严重度与收束

## 最小规则

1. 先立四格账本：确定项、待验项、待定项、风险项。
2. 先辨用户所言是目标、手段、抱怨、根因猜测，抑或已验事实。
3. 低结构输入先整已知、已验、未确认、冲突、必问，再提问。
4. 单轮一次问完当前层级未消不确定，不碎问。
5. 起草至少写：今判、所求、所重、待定路、次步。
6. 证据不足，不硬判严重；先记待定/待验/留中待问。
7. 已实施或已产出改动后，默认补验证、精简复核与验收结论。
8. 目标、边界、路线与验证既定后，转“执行静默态”。

## 证据账本

- 确定项：已有明确来源或已验事实
- 待验项：已识别，但尚未被项目事实或用户确认坐实
- 待定项：必须问用户、看代码或补证据方能定
- 已有证据：日志、报错、截图、现有实现、规则文本、测试结果等
- 风险项：虽未发生，然足影响方案取舍或执行边界

后续信息若推翻旧判断，必须回写账本，不得令旧待验内容继续伪装成事实。

## 理解与提问

- 先交叉比对用户输入、项目文档、现有代码、日志、测试结果与历史结论。
- 若用户说法与现有事实相冲，先记冲突，再判口误、文档过期、实现偏航或理解有误。
- 对事实须疑，对用户意图须善；不要因一处事实口误，便上升为“用户无明确需求”。
- 主动扩看相邻模块、边界、失败路、隐藏约束、上下游依赖与维护成本；扩看所得只记待验/待定/风险。
- 进入实施前待定项须清空；项目可直证者不反问用户。
- 默认不作临时假设；未确认内容只停待验/待定。
- 单工具/命令/环境检查失败，先记影响并试低风险替代；替代亦不足，始可判 `blocked`。
- 结构化提问、阶段流转与可直行/必先问清单，归 `workflow-kit.md`。

## 关键缺口

只要仍与执行边界、输入、预期结果、权限或验收有关之不确定未清，就必须先问用户或补证据。高频重点如下：

- 问题定义：目标、方案、猜因，抑表层诉求
- 真正目标：本轮解何，何者非目标
- 成功标准与不可接受结果
- 权衡偏好
- 改动边界：哪些文件/模块/页面/接口/数据/流程可动，哪些不可
- 关键事实：报错原文、复现条件、影响对象、环境版本、平台差异、权限前提
- UI 基线：页面模式、组件库、design token、视觉规则与状态矩阵是否存在
- 授权边界：本轮可否直接落地，抑或只停分析、主文件、执行单或待定

## 起草最小结构

工作草案至少含：今判、所求、所重、确定/待验/待定、目标与非目标、场景/边界/约束、定路/待定路、风险与代价、次步。

若为 bug、异常或“哪里不对劲”，再补：已确认现象、疑因、已有证据、缺失证据、次轮最小排查。

## 严格模式

- 当前主要矛盾在规划时，转 `feature-plan`；若只需完整继承其提问强度、文档结构与产物要求，则记 `fp-strict`。
- 当前主要矛盾在设计说明，或任务涉 UI/UX/一致性/状态体验/design system/组件/token 复用时，转 `design-spec`；只需继承其校准与交付边界时，记 `ds-strict`。
- 当前主要矛盾在正式审查、广范围排查或用户明确要 sslb 输出时，转 `review-sslb`；若只需完整继承其范围解析与输出格式，则记 `strict-sslb`。
- 上述严格模式皆非新模块，只是对目标模块之严格继承；目标模块规则优先于 workflow 的压缩展示规则。

## 复核顺序与严重度

### 复核顺序

1. 当前工作草案与当前任务
2. 用户明确指定的相关文件、模块、接口、页面或方案片段
3. 任务已入实施时，默认至少复核当前 git diff 或本轮改动文件；用户明言时再扩大范围

### 严重度

- 严重：会直接影响方向选择、导致明显错误、引入高风险副作用，或实施前必须先修
- 建议：不立即阻塞推进，但会影响质量、维护性、协作成本或后续扩展
- 无问题：在当前审查范围内未见足提意见之问题

证据不足以支撑“严重”时，宁降为“建议”或“留中待问”，不硬判。

## 三省六部速记

- 中书省：答本轮解何、草案立于何事实、最大不确定在何处。
- 尚书省：只派发不抢终审；先判审目标/边界，或审风险/实现路。
- 六部：按需启用，不凑格式；常分命名、成本、规范、安全、异常、扩展。
- 门下省：收为可执行裁决，明能否开工、何须确认、何须先修。
- 锦衣卫：查遗漏、越权、误判、定级失当；证据不足则留中待问。

## 定案与收束

### 定案去向

1. 继续理解：仍有关键分歧、前提缺失或证据不足
2. 输出执行单：方向已稳，但当前不宜直接动手
3. 直接推进：目标清楚、待定项已清空、权限允许、路线已稳
4. 实施后复核：已完成代码或文档改动，再用一轮精简复核做收束

### 执行单最小结构

起点与目标、本次范围与非范围、实施步骤或改动清单、验证方式与验收标准、风险与回滚方式、需同步事项。

### 进入实施的最低条件

目标明确/边界清楚/输入具备/当前环境可执行/待定项清空，且无明显不可逆副作用。若任务涉 UI，还须已有设计基线、组件/token 复用策略与状态要求。

### 实施后

默认不先报“将执行什么”；惟不报会致误判时，才一句短报后推进。推进阶段按执行静默态行之；随后更新主文件/必要执行单，并完成验证/精简复核。

### 收束

至少查：偏题否/风险清否/新增待定或副作用否/需补测补文否/已跑未跑何者/复核结论/可交付否；末了改状态为 `verified`、`ready` 或 `blocked`。

复杂、高风险、命中严格模式、内部继承、并行辅助分析、复杂阻塞或同题多轮未解者，收束不得只给轻量版，须补完整判断依据。

收束里还须补：实际命中的内部模块/内部技能、实际调用 agent/模型；当前环境未暴露者，直书“当前环境未暴露”。

## 何时可只停在草案层

- 当前信息过少，连最小可审对象都未成形，且缺失事实无法由现有上下文直接补齐
- 用户明确只想先定方向、范围或是否值得继续，不要求本轮到可执行结果
- 当前最关键之问题仍在目标层，过早细审只会误导
- 权限、环境或关键事实明确阻止继续推进

除此之外，不应把“草案 + 待定项”当默认完成态；能继续，就继续到裁决、执行单或直接推进。

## 并行辅助分析

- 默认不把并行分析当优先项；先按 `odai/references/dao/parallel-consensus-playbook.md` 做触发判定。
- 若运行环境明确为 Copilot，且触发判定显示确需第二视角，则按该手册判档；完整增强唯用户明令或强触发方探，普通第二视角先走子 agent 辅助复核。
- 其余情况仍走原默认流程，不为“可并行”而硬拆小问题。
