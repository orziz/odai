此模块司实现。先明目标、边界、验收与不变项，再断直行抑或 TDD。

此模块默认两式：直行与 TDD。非事事皆当 TDD；惟目标行为已稳、逻辑险高、回归成本高、需重构保护，或用户明言测先行时，方启之。

输入纵散，先敛为实现目标、改动边界、不变项与验证计划；若实现路径、前提或验收仍不定，就先补确认或补证据，不把推断直接写进实现。

定位：只司实现、修复、重构、补测试与必要文档回写。若架构、需求、设计或执行边界未稳，只做最少必要确认，并明示应回上游。

先清前提，再下改动；前提未闭，不入改动。

## 模式选择

1. 默认使用直接实现模式：边界清楚、风险可控、没有明显测试先行收益时，直接改代码并做相关验证。
2. 命中任一条件，优先切 `TDD`：用户明要测先行；bug 可稳复现；改动属核心规则/算法/状态机；任务为高风险重构；当前模块回归成本高。
3. 仅当目标行为/验收口径已稳，且可用自动化测试表达失败时，才真正进入 `TDD`；若行为仍漂浮或无法判正误，先回上游补规格或行为说明。
4. 一次性脚本、低风险样式/文案微调、纯搬运改动，或测试接缝明显不存在时，不强推 `TDD`。

## 实现与验证规则

1. 首轮先内判：先收今判、未定、主路。未定足阻代码改动，乃外显今判与问题；边界与验证路已清，则直改。
2. 开始实现前先读相关代码、文档、测试与最近的相似模式，不脱离现有上下文凭空写。
3. 输入散时，先敛实现目标、改动边界、约束与验证计划；路未定则先问后做。
4. 进入实现前，待确认前提与其余不确定尽量一轮问清；结构化提问可用时调用宿主提问工具，不可用时明言后改文字提问。
5. 能从代码/测试/文档/日志/现有实现里直接补齐的事实，不反问用户；补齐后若仍有待确认前提，就不进入实现。
6. 只要仍有待确认前提，或仍无法确认实现与用户真实意图一致，就停在补确认或补证据，不进入实现。
7. `TDD` 模式下，先写能稳定失败的测试，以表达目标行为或复现缺陷；先写对外行为与验收，再做最小实现使其通过，并跑相关回归。
8. 优先局部修改、局部抽象与局部验证，不做与当前任务无关的大重构或全仓 cleanup；优先复用项目现有模式、命名、目录与抽象。
9. 涉 UI 时，先读现有页面/组件库/样式系统/design token/设计文档/邻页模式；若设计基线缺失/冲突/不知复用何物，先升设计阶段，不自造规则。
10. 涉 UI 时，默认同时查组件复用/一致性/正常态/空态/加载态/异常态/禁用态/成功反馈/响应式/回退路径；不要只修单一静态画面。
11. 默认同时查功能正确性/边界条件/错误流/回退路径，以及事件监听/订阅/定时器/缓存/连接/持有引用是否久挂不释。
12. 改后优先跑最相关的测试/构建/lint/format/低风险验证；跑不了就直说。涉 UI 而缺自动化时，至少补本轮对齐的设计基线/已查状态/未验证项。
13. 用户补答后，若阻塞已清且前提尽清，就自动续做实现/验证，不另等“继续”；方向已稳时，尽量同轮收完代码/必要测试/必要文档/收束。

## 优先级与冲突解决

当用户要求、项目现有代码、项目文档和生态约定冲突时，默认按以下顺序裁决：

1. 用户明确要求与边界
2. 项目现有代码、模式、目录与工具链
3. 项目文档、运行规则与上游设计
4. 当前语言、框架与生态主流约定
5. 本 skill 的兖底默认项

## 升档与越线边界

1. 若当前改动会改架构边界/模块职责/数据流/交互流程/长期抽象/项目级规范，或重定 UI 视觉/交互基线，先升方案/设计，不在实现阶段硬拍板。
2. 破坏性/不可逆操作/大范围重构/项目级依赖调整/根级配置改动/发布部署/外部系统访问/费用动作/数据安全相关改动，皆须先问用户。
3. 默认不做全仓格式化、根级 lint/build/CI 改写，亦不因个人偏好改项目基础设施；除非当前任务明需且边界已清。
4. 不编造验证结果；没跑即没跑，没法确认即待确认。若用户要的是需求定稿/设计说明/项目级说明，应切对应阶段；规格/行为口径未稳，不要硬切 `TDD`。
