# UI 审美与交互复核清单

## 何时读取

命中以下任一主题时，读本文件：

- 用户说页面丑、廉价、像模板、像 AI 味、缺高级感
- 用户说交互别扭、状态不全、反馈不清、点了没反应、回退不顺
- 审查对象是页面、组件库、设计稿、设计说明、UI PR、前端实现或 design system 调整
- 需要在 review-sslb 中判断问题是实现偏差、设计基线缺失，还是审美标尺没立住

## 审查顺序

1. 先看基线：现有 design system、组件、token、视觉规则、审美标尺、状态矩阵是否存在。
2. 再看主次：三秒内能否看出主内容、主操作、次级信息与危险操作。
3. 再看系统：栅格、间距、字阶、颜色角色、圆角、边框、阴影、动效是否像一套系统。
4. 再看状态：正常、悬停、聚焦、按下、选中、禁用、加载、空、错误、成功、回退是否齐。
5. 再看压力态：最长文案、窄屏、小窗、高密度数据、低性能与键盘路径是否仍可读可用。
6. 末看代价：感知性能、认知负担、响应式、可访问性与复用维护成本是否过界。

## 四类定因

1. 实现偏差：已有设计基线，但实现未复用、未对齐或状态漏接。
2. 基线缺失：没有足够设计说明、标尺或状态矩阵，实施方只能凭感觉补齐。
3. 审美偏航：有功能也有状态，但层级、比例、节奏、字阶、留白与强调控制失衡，整体显廉价或拼贴。
4. 系统破口：局部改动打破 design system / token / 组件契约，导致跨页不一致、维护成本升高。

## 六部看点

### 礼部

- 是否一眼能分出主次、强弱、先后。
- 排版、留白、对齐、密度、色彩与字重是否协调。
- 是否靠大圆角、毛玻璃、渐变、厚阴影、插画或 3D 效果硬撑“高级感”。
- 是否同屏出现多个主 CTA、多个强调色、多个抢眼模块。
- 工具页是否误用营销 hero、过度情绪化构图或大片无效留白。

### 刑部

- 关键路径是否可完成，是否会误触、迷路、卡死。
- 空、错、慢、禁用、成功、撤销、返回是否有清楚反馈。
- 焦点、键盘、表单校验、危险确认与异常兜底是否成立。
- 是否只画理想态，把失败态与回退态漏掉。

### 工部

- 是否优先复用现有组件、页面模式、design token 与设计规则。
- 是否平白新增一套平行视觉语言、重复组件或临时样式分支。
- 是否为单页好看而牺牲跨页一致性与维护可控性。

### 户部

- 动效、图层、阴影、模糊、图片与数据密度是否拖慢感知速度。
- 信息是否过载，是否需要用户反复扫读、切换视线或记忆过多状态。
- 窄屏、小窗、低性能环境下是否仍可读可用。

## 红线

以下命中其一，通常至少列为建议；若伤关键路径或核心体验，可升严重：

1. 主操作不突出，危险操作与普通操作分不清。
2. 去掉颜色和装饰后，层级立即坍塌。
3. 缺少加载、错误、空态、禁用、成功或回退之一。
4. 为追求“高级感”牺牲对比度、字号、命中面积或读写效率。
5. 局部好看但不复用系统，导致相邻页面像不同产品。
6. 动效或装饰影响响应速度、阅读节奏或操作确定感。
7. 文案、数字、按钮、表头或导航在常见宽度下溢出、裁切、重叠或推动布局跳动。

## 输出要求

1. 不用“审美不好”“感觉不高级”这种空话；须点明失衡在层级、比例、字阶、留白、对比、状态或系统何处。
2. 判断“丑”时，优先落到可改项：主次、字阶、留白、强调数、组件复用、状态覆盖、动效边界。
3. 若无设计基线，不把实现方独自背锅；应写明缺的是标尺、设计说明还是状态矩阵。
4. 若页面方向已明确偏展示型或品牌型，勿拿后台工具页标准硬压；反之亦然。
5. 审 UI PR 时，若未见截图、状态样例、响应式说明或可访问性说明，先列证据缺口；不只凭静态 diff 断“已成品”。

## 快速判词

- 丑但可修：多为礼部与工部问题，先收层级、字阶、留白、强调数与复用面。
- 能用但别扭：多为刑部问题，补状态、反馈、回退与焦点流。
- 局部精致但整体散：多为系统破口，补 design system / token / 组件契约。
- 花哨但不高级：多为装饰盖过结构，先减特效，再重立视觉骨架。
