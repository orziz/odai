# Planning Playbook

总纲：先清疑、定界、收束题面，再起方案；未明者只列待确认，不强下判断。

## 何时读取

- 需处理附件、截图、原型、流程图或其他材料
- 用户输入很少，需先形成一版带前提的方案
- 需分层提问、起草最小方案，或主动扩展相邻场景与隐藏约束

## 最小顺序

1. 先写“今判”“所求”“所重”；拿不准者直列待定。
2. 再拆六类：真正目标、成功标准、表层诉求、用户提出的手段、约束声称、已有证据。
3. 输入很少或很乱时，先替用户整理已知、未定、相左与必问，不只回“请补充信息”。
4. 主动联想只许落为必问、待验项或风险项，不得冒充事实。
5. 提问前先给已知、未定与相左；单轮问完当前层级全部高价值问题。
6. 若需规划文档或文本草案，先写最小工作草案；承载位置按任务需要选当前对话或项目 Markdown。

## 材料与低信息输入

若用户提供附件或非纯文本内容，先做材料分析，再做方案设计。至少看：目标/显性规则/隐性前提与遗漏/是否与现状相冲/可采者/待确认者。

若用户几乎没给需求，只给参考、只说“做一个 XX”，甚至连参考都没有，也不能只回“请补充信息”。默认先做：

1. 检查项目级说明和文档。
2. 找项目内可复用的相似功能、已有模块、已有模式与约束。
3. 先替用户整理已知、关键缺口与必问。
4. 主动扩展相邻场景、边界条件、失败路径、上下游依赖与隐藏成本。
5. 若仍无足够参照，先把缺失前提问清，再决定是否继续。

## 强引导模式

当用户不会提问、不会提需、不会拆题，或只给零散线索、表层方案与模糊抱怨时，默认入强引导。

执行要求：

1. 先替用户整理，不先要他自整：把原始输入改写成“已知 / 未定 / 相左 / 必问”。
2. 先给可确认清单，不先给开放题。
3. 用户若已给多个说法或选项，可按其原话归并后请其确认；不要额外生成模型自拟默认项。
4. 主动补看用户未主动提到、但足改路线判断的维度。
5. 用户若已表达清楚，立降引导强度，不再把他当成需教学之人。

若用户明确授权你代拟方案，必须标明：哪些只是待定草案，哪些仍需用户拍板；未确认前不得拿去推进。

## 先理解用户，再拆需求

写方案前，默认先把输入拆成六类：

1. 真正目标：到底要解决什么业务或使用问题
2. 成功标准：什么结果算完成，什么结果不能接受
3. 表层诉求：用户嘴上要求的功能、页面、接口、修法或流程变化
4. 用户提出的手段：用户顺手给出的方案、技术路径或实现偏好
5. 约束声称：用户说“必须这样”“不能那样”的部分
6. 已有证据：代码、页面、日志、数据、截图、文档或现象样本

执行要求：

- 表层诉求不自动等于真正目标；先判用户是不是在拿“想怎么做”代替“为什么要做”。
- 约束声称不自动等于硬约束；要检查它是业务红线、合规边界、技术事实，还是习惯、误解或旧说法。
- 用户已指定方案时，也要反推它试图解决的核心问题，并判断是否有更小、更稳、更省长期成本的路线。
- 若你判断题目定义本身可能错了，要明确指出题面歧义，并把不同解释逐项向用户确认，不要在未确认的题面上继续细化。

## 用户真实想法与主动扩展

默认至少补齐：真目标/最重权衡/最不可受之果/为何此时提出/硬边界/可商量项。

主动扩展至少看：相邻场景/角色变、边界/败路、上下游依赖/环境差、手段与真目标之别、硬约束真假、需求未对齐/流程误用、维护协作成本。

执行要求：

- 先以最大善意理解用户意图，再质疑事实、方案或归因。
- 不要因一处事实表述不准，就倒推其没有明确需求或明确想法。
- 今判若不稳，明列待定点并让用户校正。
- 主动扩展所得只许写成必问、待验项或风险项。
- 能从项目/材料/代码/日志/低风险验证中先补齐事实或补强证据者，先补；仍不确定时再追问用户。

## 步进式询问

提问必须分层，但同一层的问题要一次性问完。默认分四层：

1. 目标层：要解决什么、想达成什么、为什么现在做、最不能接受什么
2. 场景层：谁来用、何时用、在哪个流程里用、和现有功能怎么衔接
3. 边界层：本次包含什么、不包含什么、有哪些约束、优先级和权衡偏好
4. 落地层：交付形式、验收方式、时间预期、兼容要求

执行要求：

1. 先判当前卡在哪一层。
2. 只问当前最影响判断的那一层，必要时顺带少量下一层高关联问题。
3. 单轮优先一次问完当前层级全部高价值问题。
4. 已有信息足够时，不为凑流程硬问。
5. 提问前先给出今判中的已知、未定与相左，再让用户确认。
6. 只差一步拍板就能定稿时，优先把那一步写成明确确认问题；若用户自己已给出多个选项，可引用其原话。
7. 若用户明显不会提问，优先把开放问题改成分组后的确认问题；除非用户自己已给出 A / B / C，不要自拟选项。

## 工作草案

探索期优先压为：今判、所求、所重、必问、待定路、待定、次步。

当前信息若已足够、且确需完整方案，再按需补：目标与非目标/约束与风险/方案选项/已确认或待确认草案/执行拆解/用户文档/AI 文档/更新记录。

若存在明显冲突、疑点或多种解释路径，再补一句“当前冲突何在 / 尚待确认之解释为何”，免得文档只剩表面复述。

## 并行辅助分析

仅当运行环境明确支持并行辅助分析，且并行处理能明显提升整理效率时，才允许并行参考。

若环境认作 Copilot，且 `odai/references/dao/parallel-consensus-playbook.md` 判定值得开且条件齐，可入完整增强；条件不齐则降到三席模拟或子 agent 复核；其余原流程。

讨论/决策阶段的并行只许同题复核，不拆阶段分工。题面/主路/边界/验收冻结后，执行阶段方可分工；新发现若足改上游判断，先回写；小题不滥开并行。
