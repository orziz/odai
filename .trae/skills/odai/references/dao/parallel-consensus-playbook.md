# 并行合议手册

## 何时读取

- 判能否开完整增强、三席模拟或辅助复核
- 判何时升档、何时降档、何时回原流程
- 需冻结后独立复查，或需给子 agent 组统一下发包

## 四种模式

- 完整增强模式：多 agent、多模型，同题并判；至少三种当前真实可用之不同模型。
- 当前模型三席模拟复核模式：可起三席而异构未能坐实；三席同模型、异关注点交叉复核。
- 子 agent 辅助复核模式：一至二席补证、反证或辅助复查。
- 原流程模式：不并行；冻结后之执行分工亦归此类。
- 后二者皆降档，不得称“多模型合议”或“完整增强”。

## 模式成立条件

### 完整增强

1. 当前环境明确为 Copilot，且可稳定拉起并行子 agent。
2. 总控已尽直查、AI 续查与宿主/API 续查，可证当前会话、当前环境里至少三种真实可用的不同模型，且未被权限、配额、账号或实际派位拒绝所挡。
3. 同版 `odai` 入口、当前命中模块与必要 support files 可可靠下发。
4. 当前任务属中/大复杂，并行收益胜于额外成本。
5. 当前阶段合法：确认前只许问题理解与提问准备；确认后始可再分析、方案收束或冻结后独立复查。

### 当前模型三席模拟

1. 当前环境明确为 Copilot，至少三席可稳定拉起，同版 `odai` 契约可可靠下发。
2. 已尽直查、AI 续查与宿主/API 续查，仍不能合法坐实三种真实可用的不同模型。
3. 三席可可靠沿用主流程当前模型；若模型标识未露，亦至少可可靠沿用同路默认模型。
4. 任务仍属中/大复杂，且三席交叉复核值得其成本。
5. 当前阶段仍属合法并行阶段。

此模式只算降档，不是多模型能力。

### 子 agent 辅助复核

1. 当前环境明确为 Copilot，且至少一席子 agent 可稳定拉起。
2. 同版 `odai` 契约可可靠下发。
3. 本轮确有补证、反证或辅助复查之收益。
4. 当前阶段合法。

若模型异构未被坐实，只得称“辅助复核”，不得称“多模型合议”。

## 模型能力自查口径

- 先查当前上下文、工具直出、会话日志/缓存、平台返回、会话资源与宿主同会话回填。
- 当前判断点若本地自查不足，必须回交通知总控；总控再行 AI 续查；仍不足，则通知宿主/API 续查。
- 同会话、同环境里只要尚有未试取数路径，不得以“当前环境未暴露”为终断口径。
- 真实可用，只指当前用户在当前会话、当前环境里可选、可分配或可直调者；候选名单、后台库存、旧日志、历史权限与理论可查而未回填者，皆不算。
- 用户自述、模型自述、旧缓存、需用户额外安装、补录或手工操作后方能得者，皆不作证。
- 若原生派位、宿主/API 代查或实际拉起已回“超出成本层级 / 权限不足 / 额度不足 / 当前账号不可用”或等价拒绝，则该模型即刻剔除。
- 在 Copilot 子 agent / 派位链路中，理论成本标签只作记录；惟实际成本拒绝，始可据之剔除。

## 真实增强自测与验收口径

当本轮本就在测完整增强，或总控欲凭新暴露之模型列表首次自动升到完整增强时，必须补做最小真实派位探测：

1. 先定主流程当前模型，并取至少两个已过滤后之不同候选模型。
2. 若运行面支持显式选模，则对同一只读小任务分别显式发起三席最小探测。
3. 三席皆成功，且无权限、配额、账号或实际成本拒绝，方可记“当前会话已验证可用”。
4. 若候选列表虽露，但运行面不支持显式选模，或不能稳定将模型 id 传入席位，则最多只算“候选已暴露”，不得称“真实增强已验”。
5. 收束须写：本轮实际探测之模型、被剔除者及其因由、最终是否达三模型显式成功。

## 启动前先做触发判定

先判复杂度与收益，不以“能并行”即必并行。

### 强触发信号

- 同时存在两条以上合理主路，且取舍会改写范围、边界、验收、成本或风险接受度。
- 任务跨两个以上专项模块，且模块之间有真实牵引。
- 今断属高代价冻结点，如方向定案、规格冻结、架构主路或广范围审查结论。
- 原流程已收束一轮，仍遗会改写路线、边界、验收或不可接受结果之剩余分歧。
- 若误判，回滚与纠偏之代价显著高于一次并行复核之成本。

### 排除信号

- 单文件小改、小范围修辞、已知模式下之低风险修补。
- 主路单一、替代方案稀少、误判纠偏甚廉。
- 当前只是已冻结方案下之执行分工、补证、实现或文档整理。

### 增强尝试链

- 若命中中/大复杂且未触排除，依次试：完整增强 -> 当前模型三席模拟 -> 子 agent 辅助复核 -> 原流程。
- 用户纵明言要求增强，若条件不立，亦不得硬凑；只得停在本轮最高合法档位。
- 原流程推进中若新生主路冲突、跨模块牵引或高代价冻结点，应立时重判。
- 若问题已压成单主路、低风险或执行分工，则应及时回原流程，不为形式留并行。

## 拉起子 agent 时之统一约束

- 主流程与诸席位皆承同一 `odai` skill 契约，不得主流程守规而子 agent 自由发挥。
- 每席至少拿到：同版 canonical 入口、当前命中模块、必要 support files、当前题面/边界/验收/不可擅自假设项。
- 一旦需拉起子 agent，先读 `odai/assets/dao/subagent-execution-template.md` 组统一下发包；不得以散乱 prompt 临时拼凑。
- 若还需席位选模，再读 `odai/references/dao/model-selection-baseline.md`。
- 每个下发包只对一席；不得将多席混于一包。
- 若不能确认同版 skill 契约已被可靠下发，则不得入任何增强模式。

## 主流程专属权限

- 是否开增强、何时降档或回原流程，由主流程断之。
- 席位数、模型、关注点、是否冻结题面、是否独立复查或执行分工，皆由主流程定之。
- 子 agent 只许在既定题面、边界与关注点内分析、挑战、补证或执行分工；不得自改题面、擅换主路、再拉新 agent，亦不得替主流程升级给用户。

## 统一下发包最小字段

1. canonical 入口路径
2. 当前命中模块路径
3. 必读 support files
4. 当前阶段与本席位关注点；若是执行分工，则写分工范围
5. 本席位 agent、模型与选模理由
6. 今判、所求、所重
7. 已知、已验、未定、相左
8. 当前题面、范围 / 非范围、成功标准、不可接受结果
9. 禁止擅自假设项、输出要求、失败回退口径
10. 当前已暴露之权限、配额、实际派位拒绝信息及其处理

## 阶段规则

### 确认前

- 只准整理问题定义草案、补证据、提反证、校题面与准备结构化问题。
- 对外只准落到：今判、已知、未定与问题组。
- 不得冻结方案、定实施主路或进入实施。

### 确认后

主流程先冻结以下最小交付，再行并行：

- 当前问题定义
- 范围、非范围与权限边界
- 成功标准、不可接受结果与验收口径
- 当前可用证据与禁止擅自假设项
- 本轮待收敛之核心问题

### 首轮合议

- 完整增强模式下，至少三席，且至少三种不同模型；其中至多一席与主流程同模型。
- 默认至少设：一席反证 / 逆向挑战、一席风险 / 边界 / 验收、一席补证 / 实施代价 / 冻结后复查。
- 关注点只为补挑战，不为拆题分工；每席都须围绕同一核心问题给出相对完整之判断。

### 收束与复议

主流程先行三事：去重、裁冲突、查遗漏。

- 能内部收束者，不交用户。
- 惟剩余分歧仍足改写路线、边界、验收或不可接受结果，始交用户复核。

### 冻结后独立复查

- 复查输入以冻结方案、关键证据与今断为主；不转交整段讨论史。
- 能换模型则换模型；不能换模型，至少换关注点。
- 若复查新证据足以动摇冻结判断，立刻回写上游。

### 执行分工

- 方案既冻结后，方可拆实现、验证、补证、文档等子任务。
- 此属原流程执行组织，不得称增强模式。
- 各席仍须显式承同一冻结题面、主路、边界与验收。

## 输出与留痕约束

- 收束必须写明：本轮模式、实际命中模块、实际调用 agent、实际调用模型；未暴露者直书“当前环境未暴露”。
- 未入完整增强时，须直陈其因：模型列表未露、仅一模型、异构不足、三席模拟不立、辅助复核亦不值，或实际拒绝所阻。
- 不转储无益争论，不堆叠原始分歧；只留最终判断、关键反证与是否需回写上游。
- 若子 agent 输出已越出本席边界，主流程须以失约论，回原流程重整。
