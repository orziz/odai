# 模型选择基线

## 何时读取

当你需要以下任一信息时，再读取本文件：

- 总控 agent 已在本轮通过直查确认可分配模型列表或可用模型标识，主流程需要给并行席位分配模型
- 需要判断当前暴露的模型是否足以成立完整多模型合议模式
- 需要给结果总结补出“为何没有进入完整多模型合议模式”的模型层原因
- 需要为统一下发包填写“本席位模型”与“本席位模型选择理由”
- 当前方案已经冻结，需要给执行分工席位分配模型

## 总原则

1. 先判模式是否合法，再谈模型最优分配；不满足完整能力条件时，不得靠“看起来不同”硬判成多模型合议。
2. 主流程优先求稳，子席位优先求异；能异构就异构，但异构要建立在环境已明确暴露模型标识之上。完整增强模式看的不是调起了多少 agent，而是执行前并行任务组是否真的做到了足够的模型异构。
3. 讨论/决策阶段先按同题合议中的席位关注点选模型，执行分工阶段先按已冻结方案下的子任务性质与风险面选模型；不要把“强模型”堆到所有席位上做同质化重复劳动，也不要把执行分工写回成多模型合议。
4. 只允许使用环境已暴露的信息、平台文档、仓库内已验证经验或本轮可直接观察到的表现给模型打标签；不得编造模型特长。
5. 若总控 agent 已直查仍未发现模型列表，或只暴露了模糊标签，则回退到“子 agent 辅助复核模式”或原流程，并在结果总结里明说原因。执行前若无法确认至少三种不同模型，也不得进入完整增强模式。

## 进入选择前必须拿到的输入

1. 当前总控 agent 在本轮通过直查确认到的可分配模型列表或可用模型标识。
2. 每个模型可直接读取到的标识：provider、model id、display name 或平台暴露的等价字段。
3. 当前已决定的模式：完整多模型合议模式、子 agent 辅助复核模式，还是方案冻结后的执行分工；模式决定应先由 `references/dao/parallel-consensus-playbook.md` 的触发判定与能力判断得出，此处只承接结果。
4. 当前要分配的席位清单与关注点；若已进入执行分工，则写明每个席位的分工范围。
5. 当前任务的主要约束：稳定性、长上下文、速度、成本、读档规模、是否需要强反例挑战。

若以上信息缺失到无法判断“是不是不同模型”或“该给谁分什么模型”，就不要继续假设，直接降档。

## 直查结果采信条件

若模型列表不是当前环境直接写在上下文里，而是总控 agent 通过本轮直查得到，或由宿主运行时在收到总控续查通知后于同一会话、同一环境内代查并直接回填到当前 agent 运行面，则主流程还必须额外确认以下事项：

1. 结果来自当前会话、当前环境、当前工具面里已经暴露的能力，或来自宿主在同一会话、同一环境内实际回填到当前 agent 运行面的结果，而不是用户口述、模型自述或历史残留信息。
2. 结果能明确指向当前可分配的模型或当前可读取的模型标识，而不是文档示例、未来能力说明或不绑定本轮的泛化知识。
3. 若当前原生 agent 调用能力已经暴露模型选择参数、模型标识或等价字段，应优先采信这些原生信息。
4. 平台文档或仓库文档只能帮助总控 agent 决定去哪里查，不能单独作为“当前环境已暴露模型列表”的证明。
5. 只要直查结果缺字段、去重依据不清，或仍无法证明模型是当前可分配的，就按“未查到足够模型信息”处理并降档。

说明：平台或扩展 API 可能支持查询当前模型与候选模型，但只有当对应查询路径或结果已经直接暴露给当前 agent 运行面时，主流程才能采信；平台支持，不等于本轮 skill 已拿到。若宿主运行时在收到总控续查通知后先调用平台原生 API，再把结果直接交给当前 agent，则这部分结果可按直查结果采信；若宿主没有实际回填，就不算已暴露。

## 模型归类方式

先按已暴露信息和已验证经验，给候选模型打侧重标签。一个模型可以同时命中多个标签，但标签必须有根据；这些标签可用于安排合议关注点，或在方案冻结后安排执行分工，但不代表可以在讨论/决策未收束前把任务拆成分工。

- 综合收束型：适合主流程、方案收束、长链路总结、冲突裁决。
- 题面挑战型：适合找偷换概念、补未确认点、打断默认假设、提反例。
- 风险审计型：适合风险分级、边界校验、遗漏检查、冻结后复查。
- 速扫补证型：适合大量读文件、补目录、补相邻证据、快速查漏。
- 长上下文型：适合需要一次吞入较多材料、长文档或多文件证据的席位。

若环境没有暴露足够信息来可靠打标签，只保留最保守标签：`可用但特长未验证`。

## 席位分配基线

### 主流程

- 优先选择综合收束型。
- 若多个模型都能胜任，优先选已验证更稳、长上下文更充足、总结偏差更少的那个。
- 主流程模型一旦确定，完整多模型合议模式下的执行前并行任务组至少要覆盖三种不同模型；其中最多 1 个并行席位可与主流程同模型。

### 确认前题面校验席位

- 优先选择不同于主流程的题面挑战型或风险审计型。
- 目标不是机械重复主流程总结，而是在同一题面上主动找偷换意图、范围漂移、证据缺口与错误前提。

### 确认后首轮合议席位

- 执行前的完整增强模式，至少配置 3 个并行席位并覆盖至少 3 种不同模型。
- 至少保留一个反证 / 反例挑战关注点席位、一个风险审计或边界 / 验收关注点席位，以及一个速扫补证、实施代价或冻结后复查关注点席位。
- 反证 / 反例挑战席位的任务，是优先对暂定主流判断做逆向挑战，尽力寻找能推翻它的反例、边界缺口或证据缺口；若首轮已经自然出现实质分歧，则该席位直接围绕真实分歧展开，不另造伪反方。
- 其中最多 1 个席位可与主流程同模型；不要让两个席位同时使用同一模型做同一关注点，也不要把一个席位只分配设计、另一个只分配实现这类拆题分工。

### 冻结后独立复查席位

- 能换模型就换模型；能换关注点也要换，但仍要围绕同一冻结方案做完整复查。
- 若只能继续使用同一模型，必须至少更换席位关注点，并在结果总结里说明未能做到模型异构。

### 方案冻结后的执行分工席位

- 只在题面、主路、边界与验收已经冻结后，才允许按实现、验证、补证、文档等子任务分工选模。
- 执行分工优先按子任务性质选模型，例如实现推进、验证审查、速扫补证或文档整理；但每个席位都必须显式承接同一冻结题面与边界。
- 若某个执行席位的工作会实质改写路线、边界、验收或不可接受结果，不继续当执行分工处理；先回到 `references/dao/parallel-consensus-playbook.md` 重做触发判定，再决定进入增强模式、辅助复核或原流程收束。

## 不同模型的判定口径

1. 只有当平台明确暴露出可单独分配的不同模型标识，或总控 agent 已在本轮通过直查直接读到这些标识时，才可判定为“不同模型”。
2. 同一模型的不同 system prompt、温度、推理档位、输出风格或角色设定，不算不同模型。
3. 同一基础模型的包装别名、路由别名或仅展示名不同，不算不同模型。
4. 若平台明确把不同版本、不同尺寸或不同发布日期的模型列成可分配的独立模型，可以视作不同模型；但完整多模型合议仍优先选择跨家族或跨能力侧重点的组合，而不是只换一个轻微变体。
5. 只要无法确认是不是不同模型，就按“不能确认”处理，不得为了开完整模式而乐观放行。

## 完整多模型合议模式的分配顺序

1. 先选主流程模型。
2. 再为执行前并行任务组分配至少 3 个席位，并确保至少覆盖 3 种不同模型；其中最多 1 个席位可与主流程同模型。
3. 先补题面挑战或反证 / 反例挑战关注点，并确保其中至少 1 个席位负责对暂定共识做逆向挑战；再补风险审计或边界 / 验收关注点，再补速扫补证、实施代价或冻结后复查等第 3 个关注点；三席都仍要围绕同一核心问题给出完整判断。
4. 若需要第 4 个席位，再回答下面三个问题：
   - 这个席位解决的是哪类剩余不确定？
   - 为什么现有席位还不够？
   - 为什么这个模型比现有模型更适合这个席位？

若这三个问题答不清，就不要继续扩席位；若连三种不同模型都凑不齐，就直接降档，不要硬称完整增强模式。

## 子 agent 辅助复核模式的分配顺序

1. 默认只拉 1 个子 agent；若用户明确强制要求增强模式但完整能力不成立、且部分能力成立，则优先拉 2 个子 agent，作为最高合法降档。
2. 若环境暴露了模型列表，且存在明显更适合做反例挑战或风险审计的不同模型，可以优先选那个模型。
3. 若模型列表未暴露，或只暴露单一模型，则仍可进入辅助复核模式，但结果总结里必须写清：当前环境未暴露模型列表，或当前仅暴露单一模型，因此未进入完整多模型合议模式。
4. 只有当第 1 个子 agent 返回后仍存在真实主路分歧或高风险未收束，才考虑追加第 2 个子 agent。

## 选择步骤

1. 先把当前环境直出或总控 agent 本轮通过直查得到的模型标识标准化成一组稳定字段。
2. 去掉明显只是同模型不同预设、不同标签或不同角色包装的重复项。
3. 判断当前可确认的不同模型数量；若当前目标是执行前完整增强模式，少于 3 种不同模型就直接降档。
4. 若当前是合议 / 复核阶段，按席位关注点给候选模型打标签；若当前是执行分工阶段，按子任务性质与风险面打标签；没有把握就写“特长未验证”。
5. 若当前是合议 / 复核阶段，先分配主流程，再分配挑战席位，再看是否需要补审计或复查关注点席位；若当前是执行分工阶段，先确认冻结方案，再分配实现、验证或补证席位。
6. 把每个席位的模型与选择理由写进统一下发包。
7. 若任一步骤拿不准，就降档，不为追求形式保留完整多模型合议模式。

## 结果总结必须回报的模型层信息

- 当前进入的是完整多模型合议模式、子 agent 辅助复核模式，还是方案冻结后的执行分工。
- 当前环境是否在总控 agent 本轮直查后暴露了可分配模型列表或可用模型标识。
- 本轮实际调用了哪些模型；若环境未暴露，则明确写“当前环境未暴露”。
- 若用户明确强制要求增强模式，还要明确写出：本轮是否已进入最高合法降档，以及实际拉起了几个辅助复核席位。
- 若没有进入完整多模型合议模式，模型层面的直接原因是什么：
  - 总控 agent 已直查仍未发现模型列表
  - 仅暴露单一模型
  - 无法确认至少三种不同模型
  - 模型虽不同，但没有足够依据做可靠同题合议分配

## 短规则

- 先判合法，再判优选。
- 讨论阶段先定主流程，再把执行前完整增强模式配到至少三种不同模型，再配挑战关注点席位；执行阶段先确认冻结方案，再配子任务席位。
- 能异构就异构，不能异构就诚实降档。
- 不靠别名、角色和推理档位伪造不同模型。
- 模型选择理由必须写入统一下发包。
