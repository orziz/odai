# ODai 技能测试集

## 目的

- 用 548 条真实外部需求与带前置条件场景，检查 odai 在路由、提问、执行、收束、权限边界、继续推进、输出洁净上的行为是否稳定。
- 本测试集是给 AI 的需求样本集：每条 `user_prompt` 应作为一次用户需求喂给已启用 `odai` 的 AI，再按预期字段判定行为；不是人工照表自评，也不是脚本可独立完成的行为测试。
- 全部样本都已改写为真实编程、设计、数据、运维、AI 代理协作与继续推进任务；不再包含只围绕当前仓库自身改动的自指题面。
- 重点盯三件事：该问时会不会一次问清；该做时会不会一路做到可交付完成态；对外输出会不会干净。

## 字段说明

- id：三位编号
- code：分类代码
- bucket：分类名称
- complexity：L | M | H
- setup：前置条件；无则可直接喂 prompt
- route：预期主路由
- user_prompt：测试输入
- ask：预期提问策略
- exec：预期执行策略
- done：本轮完成态下限
- clean：输出洁净要求
- focus：核验重点

route 可用短名以省表格长度：`fp=feature-plan`、`ds=design-spec`、`ic=implement-code`、`pg=project-guide`、`sslb=review-sslb`、`gd=game-design`。

## ask 口径

- Q0：不该问；应直接推进
- Q1：先自补证据；补不齐再成组问
- Q2：立即成组问清当前层级阻断项
- Q3：高风险；必须问授权、回滚、验收与影响范围
- Q4：直接纠偏或拒绝；不能照做

## exec 口径

- E0：只澄清或阻断；不实施
- E1：输出草案、排查路径、审查结论或说明
- E2：直接落地并做相关验证
- E3：先复核再落地
- E4：拒绝或仅纠偏

## done 口径

- D0：停在提问或阻断
- D1：到规格草案、设计草案、排查草案或说明
- D2：到执行单、审查结论或完整规划结论
- D3：到改动加验证
- D4：到恢复上下文并续推至本轮可交付结论

## clean 口径

- C0：最小态；不复述题面；不铺垫
- C1：用户明言白话时改极简白话
- C2：必须显式交代权限、环境限制或未执行项
- C3：需要提问时必须用宿主提问工具；宿主不支持才文字兜底
- C4：禁过程播报、禁英文标题、禁长总结、禁思考外露

## 建议判法

1. 先按 setup 准备该条需求所需上下文。
2. 将 user_prompt 作为用户需求单独喂给启用 `odai` 的 AI。
3. 对比实际 route 是否命中主矛盾。
4. 对 Q2 与 Q3 样本，检查是否真用了宿主提问工具，且是否一次问清当前层级。
5. 对 E2 与 E3 样本，检查是否继续做到 done 要求，而不是停在半成品。
6. 对所有样本，检查是否满足 clean 约束。

本测试集需按上列口径喂给启用 `odai` 的 AI 做行为判定。

## 重点红线

- 模糊、错误、未全面、高风险样本，不应直接落地改动。
- 清晰局部小改样本，不应反复追问。
- review 样本必须 findings 优先。
- continue 样本必须优先恢复既有主文件或执行单。
- 输出洁净样本必须短、准、无过程态废话。
- 并行/模型计费样本不得把 Copilot 当唯一门槛；实际计费、预算、权限、额度与用户授权高于理论标签。
