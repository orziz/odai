# odai

`odai` 是面向 AI agent 工作的单一治理入口。它不重复教模型通用推理、搜索、阅读、编码或总结，而是规定 agent 不该自行拍板的部分：真实意图、边界、授权、验收、交接、证据和停止条件。

入口里头有个叫 `道` 的总控：先校准目标和风险，必要时编排模块链；简单任务保持轻量推进，只有缺口会改变路径、范围、授权、验收、风险或停止线时才问你。

- **只想干活？** 一个入口：`odai`。
- **要维护这个仓库？** 另有 `skill-author` 工具，详见 [MAINTAINING.md](MAINTAINING.md)。

> `main` 分支装的是这套统一入口。想用更早的「一个能力一个技能」的旧结构，可以装 `old` 分支（见[如何安装](#如何安装)）。

## 30 秒上手

1. 把 `odai` 装进当前环境：

```bash
npx skills add https://github.com/orziz/odai --skill odai
```

（想装整个仓库、精简版或旧结构，见[如何安装](#如何安装)。）

2. 直接把任务交给 `odai`。第一句尽量带上三样：**目标**（想要什么）、**材料**（手上有什么）、**约束**（不能碰什么、必须满足什么）。说不清该走哪个模块也没关系，`道` 会先帮你判断。

3. 心里已经有数、想走哪条路也可以直接点名：

- 「用 `odai` 接这个需求，先判断该走哪个模块，拿不准就问我。」
- 「用 `odai` 走 `道`，先把边界和主路定下来。」
- 「用 `odai` 把这个一路做到结果总结——`道` 自己定路线，不半途交接。」
- 「用 `odai` 按 `review-sslb` 审这个 PR。」
- 「用 `odai` 用 `ribao` 整理今天的产出。」

## 这到底是个啥

可以把 `odai` 理解成强模型和 agent 之上的治理层：

- **对外只有一个门。** 用户任务从 `odai` 进入；清楚、轻量的任务保持轻量，模糊或跨域任务交给 `道`。
- **在自主性有风险的地方设硬线。** 它守真实意图、范围、授权、验收、真实性、接力和证据。
- **门后是一组分工明确的模块。** 规划、设计、审查、实现、游戏策划、游戏视觉、成果整理、项目说明按需加载。
- **Agent 只是受控证据 / 执行单元。** 子 agent 可以生成候选、证据、审查或冻结范围内的执行结果；主流程只采纳可复验的部分。

几条值得先知道的事：

- 干活只认 `odai` 这一个入口；维护仓库用的是另一个工具 `skill-author`。
- 分发统一走 [skills.sh](https://skills.sh) 标准（`npx skills add`），canonical source 就在 `skills/` 下，不再有各端镜像产物。
- 旧的「多技能并列」结构已经搬到 `old` 分支，仍需要的话单独装。

## 适合谁用

符合下面任意一条，这个仓库就会比较顺手：

- 想用一个入口约束 agent 工作，又不想把每个任务都变成僵硬流程
- 希望 AI 能自主查证和推进，但在范围、授权、验收、风险边界前先停下来校准
- 喜欢在模糊或高影响任务里先让 `道` 定方向、边界、主路和第一步
- 想把审查、诊断、TDD、UI 质量、游戏策划、agent 下放等规则按需加载，而不是每次全塞进上下文
- 经常要整理 README、项目规则、AI 接手说明，或日报 / commit / PR 描述
- 想用 skills.sh 一条命令把入口接进任意 agent，不手动复制文件

## 如何安装

### 省事装法

只装统一入口（大多数人选这个）：

```bash
npx skills add https://github.com/orziz/odai --skill odai
```

适合：想快速把入口接进环境、不想手动复制一堆文件、平时直接用 `odai` 触发内部模块。

其他几种装法，按需选：

```bash
# 连同仓库里的其他技能一起装
npx skills add https://github.com/orziz/odai

# 更省 token 的精简版（mini 分支）
npx skills add https://github.com/orziz/odai#mini

# 旧的「多技能并列」结构（old 分支）
npx skills add https://github.com/orziz/odai#old
```

什么时候选 `old` 分支：你还在用旧入口、需要 `harness-dev` / `harness-dao` 等的独立安装方式，或者正在做旧结构的迁移对照。

```text
                          ▼
       ┌──────────────────────────────────────┐
       │  odai · SKILL.md  入口判定             │
       │  点名直达 · 轻量证据门 · 指定实现直达  │
       └────────────┬─────────────────────────┘
          直达        │   模糊 · 跨域 · 开发类
       ┌────────────┘                 │
       │                              ▼
       │                ┌──────────────────────────┐
       │                │  道 (dao) — 总控 workflow  │
       │                │  道 → 术 → 法              │
       │                │  定方向 · 定轨道 · 监督     │
       │                └─────────────┬─────────────┘
       │   收术定生产者链 ·            │
       │   轨道往前接、不回弹 ─────────┤
       ▼                              ▼
   ┌────────────────────────────────────────────────┐
   │  专项模块（生产者）                              │
   │   feature-plan · design-spec · implement-code    │
   │   game-plan · game-design · review-sslb          │
   │   project-guide · ribao                          │
   └────────────────────────────────────────────────┘

   支撑文件（道按需加载）：
     interaction-contract（硬法） · dao-shu-fa-playbook
     review-kit · diagnose-kit · agent-governance · result-reporting
     consensus-mode · external-skills
```

`道术法` 是**活的阶段**——为何 → 怎么走 → 怎么做——每个任务都流经；
不是盖在模块上的身份标签。任务从 `odai` 一个门进来，简单的直达某个
模块，其余交 `道`：由它定轨道、一路带到结果总结。

## 怎么用更顺

`odai` 不会把所有模块机械串一遍。它先判断任务能否直达或保持轻量；不能时，`道` 再判断真正缺的是哪一层、该调哪个模块、该产出什么形态。

内部只有一条总线：

- **`道`**：默认总控，也是唯一的通用 workflow。更适合「先定方向、边界、主路和第一步」，也负责挑模块链、定产物形态，并推进到结果、已验证缺口或真实阻断。

单一清楚段落也可以越过 `道`，直接点名单个模块：

- `game-plan`：游戏系统、玩法、数值、经济、商业、关卡与内容规划
- `game-design`：完整游戏视觉设计——UI/UX/UE、角色场景、宣传品牌、特效演出
- `feature-plan`：写规格、做方案取舍、诊断 bug
- `design-spec`：页面、交互、状态、视觉、体验说明
- `implement-code`：边界清楚后的写代码、修 bug、补测试、重构落地
- `project-guide`：README、项目规则、AI 接手基线
- `review-sslb`：三省六部式代码审查（旧的多种审查风格已并到这一个入口）
- `ribao`：日报、commit message、PR message

几个好用的触发示例：

- 「用 `odai` 接这个需求：先判断走哪个模块和产物形态，拿不准就问我。」
- 「用 `odai` 先用 `道` 把边界、主路和关键风险定下来，再往下推。」
- 「用 `odai` 处理这个实现问题，由 `道` 一路推进到结果总结。」
- 「用 `odai` 用 `project-guide` 整理这个仓库的 AI 接手基线。」

## 它怎么跟你打交道

那些会主动向你补关键信息的模块，默认都遵守一条交互约定（`skills/odai/references/dao/interaction-contract.md`）：

- 先尽量自己阅读、查证或跑低风险验证。
- 只把会影响目标、边界、授权、验收、风险、停止线或不可接受结果的决定交给你校准。
- 区分未验证说法和已确认事实；没读、没跑、没调用、没验过的内容不冒充结果。
- 收到你的回答后，默认直接接着干当前这一步，不会再等你补一句「继续」。
