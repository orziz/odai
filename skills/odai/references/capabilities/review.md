# 审查

用户要求审查代码、diff、目录、仓库、UI 或设计系统时读取。默认只读、findings first；用户同时授权修复时，先冻结审查结论，再进入交付。

## 形成 finding

范围优先级：用户显式范围 → 与变更语义相关的文件与引用 → 当前 diff。先理解变更意图、项目规则和实际调用关系；证据不足不判缺陷。

每条 finding 给出紧凑定位、问题、触发条件或因果、实际影响、最小修复方向和针对该问题的复验方式。`先别改`只限制写入，不省略建议。严重度按当前目标的真实风险：

- **阻塞**：会造成正确性、安全、数据、关键体验或发布风险，阻断当前目标。
- **非阻塞**：不阻断当前目标，但有明确维护、性能、清晰度、一致性或后续风险。
- 没有 findings 就明确当前范围未发现问题，不凑数；证据不足的观察单列为未覆盖或待确认。

## 覆盖面

按变更选择有关检查面，不把清单变成仪式：

- 意图、命名、语义、所有权与实际行为是否一致。
- 数据、计算、I/O、资源、性能和生命周期是否正确。
- 格式、状态、兼容、可访问性与用户操作闭环是否完整。
- 输入、注入、权限、敏感信息和外部副作用是否守界。
- 异常、边界、依赖失败、恢复与回退是否成立。
- 职责、耦合、复用和扩展成本是否与当前改动相称。

正式准入、连续 clean、全仓 / 安全 audit 或长修复回路读 `references/techniques/review-modes.md`；UI 与交互按 `references/capabilities/design.md` 和相应领域资料复核。审查 agent 始终只读；采纳和修复协作按 `references/dao/leverage.md`。

结论先给 findings，再给总体裁决、未覆盖范围和剩余风险。建议数量不替代风险判断。
