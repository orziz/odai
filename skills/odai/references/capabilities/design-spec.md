# Design Spec

行为、交互、信息架构、页面、视觉、品牌或体验仍需设计决定时读取。用户要求直接落地且边界已清楚时，设计只服务实施，不另造文档。

## 形成可交接设计

1. 明确谁在什么场景完成什么任务、系统如何反馈、什么结果算成立。
2. 检查现有产品、组件、token、品牌、平台习惯和相邻模式，区分可复用基线、实现偏差和真正缺口。
3. 只覆盖当前任务相关的流程、信息层级、布局、状态、反馈、文案、一致性、可访问性、响应式、性能和维护成本。
4. 正常、空、加载、异常、权限 / 禁用、成功与退出 / 回退按实际触达选择；“要求覆盖”不证明已经实现，逐项标清现有、已确认或候选。
5. 交付仍缺的最小决定、风险和可执行验收；答复前按上游事实与已读专项完成门核对，每个必需字段、状态、视口、输入方式和无障碍行为都映射到可观察检查。

交付前过来源门：类型、枚举、默认选择、排序、详情、刷新、权限、状态行为和交互，只要没有项目事实支持，就在每个出现位置明确标为“提案 / 待确认”，不得放进“实现契约”或写成硬性要求；开头一条总免责声明不能替代逐项标注。相邻材料只有在 brief、实现或数据中存在目标关联证据时才能采用，同仓库、同领域或关键词相似不够。

用户点名的视口、输入方式、数值和可访问性要求必须逐字保留到可复验行为。上游给出多个具体视口时，用最小“视口 × 布局 / 字段 / 主操作 / 状态 / 验收”矩阵逐行覆盖；键盘焦点、控件语义、读屏标签和状态通知写清如何工作。

“更高级、更顺、更有质感”等目标先结合现有基线、参考和场景形成可逆探索；只有路线会造成实质不同产品结果，或连续调整仍不收敛时才请求选择。未知指标、客户、评价、奖项、截图、运行数据和业务枚举必须标为候选或占位，不伪装成事实。

本文件已经足够形成一般交接。只有仍缺会改变当前设计的专项工艺时，才选择一份下一主资料：通用页面与工作台用 `references/domains/ui-design.md`，实时交互系统的 HUD、反馈、表现与资产用 `references/domains/interactive-systems.md`。领域自动判断，不让用户选择领域包，也不要两份都读。
