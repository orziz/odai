# 设计

行为、交互、信息架构、页面、视觉、品牌或体验仍需设计决定时读取。设计是成事手段：用户要求直接落地且边界已清楚时，只补实施所缺决定；用户只要方案时，可交接设计本身就是结果。

## 形成可交接设计

1. 明确谁在什么场景完成什么任务、系统如何反馈、什么结果算成立。
2. 检查现有产品、组件、token、品牌、平台习惯和相邻模式，区分可复用基线、实现偏差与真正缺口。
3. 只覆盖当前任务相关的流程、信息层级、布局、状态、反馈、文案、一致性、可访问性、响应式、性能和维护成本。
4. 正常、空、加载、异常、权限 / 禁用、成功与退出 / 回退按实际触达选择；要求覆盖不证明已经实现，逐项标清现有、已确认或候选。
5. 交付仍缺的最小决定、风险和可执行验收；每个必需字段、状态、视口、输入方式与无障碍行为映射到可观察检查。

类型、枚举、默认选择、排序、详情、刷新、权限、状态行为和交互没有项目事实支持时，在每个出现位置标为“提案 / 待确认”，不得放进实现契约。相邻材料只有存在目标关联证据时才能采用，同仓库、同领域或关键词相似不够。

用户点名的视口、输入方式、数值和可访问性要求须保留为可复验行为。上游给出多个具体视口时，用最小“视口 × 布局 / 字段 / 主操作 / 状态 / 验收”矩阵覆盖；键盘焦点、控件语义、读屏标签和状态通知写清如何工作。

“更高级、更顺、更有质感”等目标先结合现有基线、参考和场景形成可逆探索；只有路线会造成实质不同产品结果，或连续调整仍不收敛时才请求选择。未知指标、客户、评价、奖项、截图、运行数据和业务枚举不得伪装成事实。

本文件足够形成一般交接。仍缺专项工艺时只选一份：通用页面与工作台读 `references/domains/ui-design.md`；实时交互系统的 HUD、反馈、表现与资产读 `references/domains/interactive-systems.md`。
