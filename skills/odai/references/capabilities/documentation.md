# 文档与汇报

用户要求把已有事实、进展、变更、决定或项目知识整理成面向特定受众的文档性产物时读取。若任务核心是形成新方案、设计或审查结论，仍由对应能力负责；具体名称、模板和载体服从任务，不按产物名称硬编码路线。负责从证据形成可审阅内容，或更新用户明确要求的文档；不因“写、整理、生成”等内容动作本身推定用户已授权外部提交、发送或系统操作。对象与动作已由用户指令、上下文或既有约定充分明确时，在边界内继续完成。

## 先定交付与授权

先分清用户要的是：直接返回文案、修改项目文档，还是填报 / 发布到指定系统。外部对象与写入动作无法从用户指令、上下文或既有约定充分确定时，默认交付可审阅正文；当前页面、可用工具、待办列表或既有表单的存在本身不构成写入授权。对象与动作已充分明确时，按 `references/dao/authority.md` 核对账号、可见范围、字段与确认边界后继续完成。

只写证据支持的字段。完成百分比、工时、负责人、日期区间、状态和承诺不得从任务数、提交数或语气推断；缺失会阻断交付时只问一个最小问题，否则明确留空、标为待补，或交付不依赖该字段的正文。不要为填满模板编造事实。

## 取证与裁剪

证据依次取自：用户明确描述与指定材料 → 目标文档、模板和项目规则 → 指定范围内的 git log、status、diff stat 与相关 diff → 相关文件和当前对话。只读足以支撑当前内容的最小范围，不把大 diff、全仓盘点或无关历史灌进上下文。

没有可核实内容时直说；可以给带明确占位符的骨架，但不能用通用套话冒充真实进展。语言、格式和语气服从用户要求、受众与项目惯例。

## 日报、更新与变更文案

- 日报 / 周报 / 工作总结优先写可核实的产出、目的或价值、结果；真实存在时再写问题、风险和下一步。默认简洁，每条尽量包含“动作 + 结果”，不写流水账或机械罗列文件名。
- 明日计划以已确认的未完成项、下一动作和用户安排为主；有证据的建议可单列为候选，不冒充安排或承诺。没有依据就省略或标为待确认，不为格式完整编造。
- 干系人更新区分事实、提案、风险、决定和下一动作，不把讨论中事项写成承诺。
- commit message 凝练当前变更；PR 文案按需补背景、主要改动、验证、风险和真实未决项。没有运行的验证不写成已通过。

## 项目指南

先读现有 README、规则文件、关键配置、目录和相关文档，再选择唯一主文档。优先更新已有 README、PROJECT 或 docs 入口；没有合适 owner 且用户确实要求落盘时才新建，不复制平行指南。

区分用户确认、代码可见、已验证和仍待确认的内容；文档与代码冲突时标明来源和影响，不任意选择一方。按受众裁剪项目目标、结构、关键命令、高风险区域、术语、协作方式和验证入口；高频歧义才进词汇表，高代价且需长期追溯的架构决定才建议 ADR。

## 验成

正文任务以可直接审阅或发送的真实文案为完成；文件任务以目标文档的实际 diff 和必要检查为完成；外部填报 / 发布以指定系统的真实响应和最终状态为证据。只生成正文时，不声称已填报、已发送或“功能完成”。
