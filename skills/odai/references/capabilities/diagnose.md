# Diagnose

Bug、报错、白屏、行为不符、偶发问题、性能回归或原因不明时读取。目标是缩小不确定性并找到与症状有直接关系的处理路径，不是证明第一个猜测。

## 最短诊断链

1. 明确环境、入口、输入 / 步骤、实际结果、期望结果和发生频率；可从项目与现有证据恢复的不要追问。
2. 建立最便宜且有判别力的症状信号：复现、失败测试、日志、trace、CLI / HTTP 断言、页面观察、性能基线、差分或人在环验证。
3. 用最近变更、成功对照、数据 / 控制流、配置差异、二分或 profiler 缩小少量可证伪假设。
4. 选择能区分假设的下一项观察；有直接证据后再修，最后重跑原场景和高价值相邻回归。

用户提供的根因和修法是重要线索，但仍需与真实症状入口、使用链或直接因果相连。相邻函数异常、自造输入和新测试可以探索候选，却不能单独证明已经修复用户所述问题。

无法复现时，根据风险决定：局部、可逆、静态因果清楚且能用其他证据降险的改动可以实施，但必须标明原场景未验证；涉及跨层或高风险时先补证。

同一方向连续尝试没有获得新信息时，换正交证据路径或回到问题定义、契约和环境层。探针只记录能区分假设的观测，使用后清理。

若下一步会改变超时、重试、并发、资源或其他运行参数，先确认观测证据支持同一方向；证据相反或副作用未知时不改。改动成立时仍给最小试点、观察窗口、成功 / 停止指标和恢复原值的回退方式；未获实施授权时只交付分析与操作建议。

需要写代码时按实现边界做最小完整改动；只有当前仍缺具体工艺或完成判定，才另选对应主资料，不顺着引用串读。
