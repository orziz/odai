# Implement Code

非简单的代码、配置、脚本或测试实施需要补工艺时读取。准确局部修改且路径与验证已知时直接做，不加载本文件。

## 实施与验证

1. 读与目标直接相关的代码、测试、配置、文档和相似实现，确认保持项与可判定结果。
2. 做最小而完整的改动，复用仓库实现、原生能力和已有依赖；不借题抽象、重构或清全仓。
3. 多文件、重命名或批量改动先盘点命中、引用、旁路与兼容；证据能锁定就行动，需要用户取舍才停。
4. 外部 API、库和 CLI 按项目实际版本核对；外部输入与模型输出在信任边界校验，secret 不进上下文和日志。
5. 每个验收点和主要回归风险映射到成本合理、有判别力的证据，验证通过再收口。

Bug 先建立与真实症状和因果链相连的证据；用户建议的修法被证据推翻时不得连带实施。稳定复现、核心规则、状态机或高回归风险适合时让失败信号先于实现；普通改动不强制 TDD、SDD 或 BDD，也不事后包装成 TDD。测试覆盖对外行为、契约、边界和错误路径，不靠放宽断言、吞错、skip、顺序或无依据 sleep 造绿。

## 命中边界

- UI：复用现有组件和 token，覆盖相关状态；构图、断点或关键交互变化时用最小真实视口与交互集合验证。
- 大改动：按可独立验证的垂直切片推进，不留下断裂中间态。
- 对外接口：边界校验、错误形状一致，优先后向兼容；列表接口考虑分页。
- 外部输入与模型输出：进入 SQL、shell、DOM、路径、反序列化或工具调用前校验；权限写进代码，不靠 prompt。

跨多文件、核心逻辑或高风险改动完成后按风险补审查；环境受限、旧状态承接或未验证时如实收口。只有出现具体未决工艺才加载下一份资料。
