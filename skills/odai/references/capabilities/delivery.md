# 诊断与交付

Bug、报错、行为不符、性能回归，或非简单的代码、配置、脚本与测试实施需要补工艺时读取。目标不是证明第一个猜测或产出一份补丁，而是用最窄充分证据推进到用户定义的可交付结果。准确局部修改且路径与验证已知时直接做，不加载本文件。

## 先辨因

1. 从项目与现有证据恢复环境、入口、输入 / 步骤、实际结果、期望结果和频率，只追问无法自行查证且会改变路径的缺口。
2. 建立最便宜且有判别力的症状信号：复现、失败测试、日志、trace、CLI / HTTP 断言、页面观察、性能基线、差分或人在环验证。
3. 用最近变更、成功对照、数据 / 控制流、配置差异、二分或 profiler 缩小少量可证伪假设；选择能区分假设的下一项观察。
4. 用户给出的根因与修法是重要假设，仍须与真实症状入口、使用链或直接因果相连。相邻函数异常、自造输入和新测试可探索候选，不能单独证明已修复用户问题。

用户只求原因时，交付有证据的判断与可执行下一步；用户要求修好且授权充分时，定位不是终点，继续最窄完整改动并复验原场景。

无法复现时，局部、可逆、静态因果清楚且能用正交证据降险的改动可以实施，但须标明原场景未验证；跨层或高风险先补证。同一方向连续尝试无新信息时，换正交证据或回到问题定义、契约和环境层。

## 再闭合结果

1. 读与目标直接相关的代码、测试、配置、文档和相似实现，确认保持项与可判定结果。
2. 做最小而完整的改动，复用仓库实现、原生能力和已有依赖；不借题抽象、重构或清全仓。
3. 多文件、重命名或批量改动先盘点命中、引用、旁路与兼容；证据能锁定就行动，需要用户取舍才停。
4. 外部 API、库和 CLI 按项目实际版本核对；外部输入与模型输出在信任边界校验，secret 不进上下文和日志。
5. 每个验收点与主要回归风险映射到成本合理、有判别力的证据，验证通过再收口。

稳定复现、核心规则、状态机或高回归风险适合时让失败信号先于实现；普通改动不强制 TDD、SDD 或 BDD，也不事后包装成 TDD。测试覆盖对外行为、契约、边界和错误路径，不靠放宽断言、吞错、skip、顺序或无依据 sleep 造绿。

改超时、重试、并发、资源或其他运行参数前，观测证据须支持方向、幅度和保护链；证据相反或副作用未知时不改。需要试点时明确环境 / 流量边界、单一变量、观察窗口、通过 / 停止指标与恢复原值的回退。

## 命中边界

- UI：复用现有组件和 token，覆盖相关状态；构图、断点或关键交互变化时用最小真实视口与交互集合验证。
- 大改动：按可独立验证的垂直切片推进，不留下断裂中间态。
- 对外接口：边界校验、错误形状一致，优先后向兼容；列表接口考虑分页。
- 外部输入与模型输出：进入 SQL、shell、DOM、路径、反序列化或工具调用前校验；权限写进代码，不靠 prompt。

跨多文件、核心逻辑或高风险改动完成后按风险补审查；环境受限、旧状态承接或未验证时如实收口。只有出现具体未决工艺才加载下一份资料，不顺引用串读。
