# Review SSLB

用户要求代码、diff、目录、仓库、UI 或设计系统审查时读取。默认只读、findings first；用户同时授权修复时，先完成并冻结审查结论，再进入实现。

## 审查方式

范围优先级：用户显式范围 → 与变更语义相关的文件与引用 → 当前 diff。先理解变更意图、项目规则和实际调用关系；证据不足不判缺陷。

每条 finding 给出紧凑定位、问题、触发条件或因果、影响和可执行建议。`先别改`限制写入，不省略建议；没有最小修复方向和针对该问题的复验方式，finding 尚未闭合。严重度按实际风险：

- 🔴 会造成正确性、安全、数据、关键体验或发布风险，阻塞当前目标。
- 🟡 不阻塞当前目标，但影响维护、性能、清晰度、一致性或后续风险。
- 🟢 当前范围未发现问题；没有 findings 时不要凑数。

## 三省六部

默认只把六部作为内部检查面，不外显仪式：

- 吏：命名、语义、可读性和所有权。
- 户：计算、数据、I/O、资源和性能。
- 礼：格式、残留、一致性；UI 看层级、排版与视觉秩序。
- 兵：输入、注入、权限、敏感信息和攻击面。
- 刑：异常、边界、依赖失败、类型与恢复；UI 看关键状态和操作闭环。
- 工：职责、耦合、复用、扩展成本与资源生命周期。

用户要求完整三省六部，或审查是正式合并 / 发布 / 安全准入时，读 `references/techniques/review-full.md`。连续 clean、全仓 / 安全 audit 或长修复回路读 `references/techniques/audit-loop.md`；UI 与交互按 `references/capabilities/design-spec.md` 和相应领域资料复核。

结论先给 findings，再给总体裁决和未覆盖范围。审查 agent 始终只读；结果采纳和修复协作按 `references/dao/coordination.md`。
