# Feature Plan

目标、范围或关键行为尚未稳定，需要规格、方案、系统、内容、数值或执行前决定时读取。用户已明确要求实施且证据足够时，不用规划代替行动。

## 形成决定

1. 从用户原话、项目材料和现状恢复使用者、场景、期望结果、约束与不可接受结果；已有答案不重问。
2. 分开目标、事实、症状、根因假设、建议手段和硬约束。用户给的原因或修法优先验证，不自动成为事实或唯一方案。
3. 只检查会改变当前决定的相邻场景、失败路径、上下游、兼容性和长期成本。
4. 真有多路时比较解决程度、改动面、风险、维护、可逆性和依赖，给推荐与改选条件。
5. 裁成当前交付所需的最小规格：目标、范围、关键行为、非目标、验收、风险和下一步。

## 模糊与开放项

- 疑似误写、证据不足或要求冲突时保留原话，区分当前理解与候选；可从项目查证的先查证。
- 未决项不擅自拍板，也不只罗列选项；来源中的 `open / proposed / pending / 待定` 在产物中仍明确开放。证据足够时给“待确认”的推荐、理由和改选条件，不把推荐默认值写成用户决定。只有用户权衡不可替代时才按授权边界提问。
- “顺便优化、都重构、还有没有”等开放表达先产出有边界的候选和优先级，未授权扩展不进入实施。
- 验收写成可观察动作与通过条件；没有真实数据时，把数字和效果预期标为假设并给验证与回调方式。
- 交付具体数值前，把主值、区间端点、取整、累计和极值逐个代回同一公式；候选、灰度值与上下限同样复算，文字结论不得超出计算结果。
