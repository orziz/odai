# 长审查收敛回路

任务明确要求连续 N 轮 clean、全仓 / 全目录复查、安全 audit 或长修复后反复验证到收敛时读取。普通单轮审查不读。

## 开始前冻结

- 明确审查范围、证据基线、相关验证和“clean”的判定。默认 clean 表示没有阻塞当前目标的 finding 且相关验证未失败；用户要求零 finding 时从用户。
- 阻塞 finding 需要当前范围可触发的路径和可说明的实际影响；证据不足则留为待确认或非阻塞观察。
- 维护建议与真实缺陷分开，不用建议数量阻止收敛，也不静默吞掉。

## 轮次

每轮基于当前 checkout 和同一冻结范围重新检查。范围内修改、证据基线变化、阻塞 finding 或相关验证失败会使连续 clean 计数归零；明确无关的范围外变化不重置。

每轮紧凑记录：`clean round x/N｜结果｜范围｜关键证据`。不能复用修复前结论、旧线程自报或未完成扫描。

## 协作与修复

使用 agent 时按 `references/dao/coordination.md`：扫描者只读并回交 finding 与复现证据，主审负责去重、定级和计数。扫描失败是验证未完成，不能靠增加席位冒充完整覆盖。

用户已授权修复时，先冻结经核证的完整清单，再按 `references/capabilities/implement-code.md` 实施。修复后重跑相关验证并把 clean 计数归零；只有主审实际复验后才关闭 finding。
