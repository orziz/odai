# Planning Playbook

附件 / 低信息输入、低结构或混合需求、需要原子化的多条输入、纠偏、扩展候选、多方案比较或结构化对象规划时读取。

## 需求账本（owner）与内部覆盖清单

多条输入先原子化：独立动词、对象、约束、验收点、非目标、目标文件和显式引用都要有覆盖状态。

- 输入清楚，没有范围冲突、开放式扩展、对象缺证、待确认项或会改写治理点的纠偏时，使用内部覆盖清单直接推进，不强制外显；不得只因条目多就停手。
- 存在扩展候选、对象缺证、新建边界、多种合理纠偏或待确认项时，必须外显清单，本轮停在清单、草案与确认问题。

外显清单至少区分：

- 原始需求 / 约束 / 验收；疑似误写片段必须逐字引用，不得先正规化。
- 纠偏理解：用“原话 → 纠偏理解”并列写出依据，例如 `“权限限” → 疑似“权限”边界`。
- 扩展候选：说明来源、收益、代价和不采纳后果。
- 覆盖状态：已覆盖 | 改写待确认 | 暂缓待确认 | 待验证补证。

不把推断补成用户已说，也不把扩展候选做进代码、测试或正式文档。

## 理解与方案

按问题需要检查：真正目标、成功标准、输入 / 触发、输出 / 结果、范围 / 非目标、用户手段、约束声称和已有证据。已清楚的层不展开。

- 有附件 / 参考时，区分可采用规则、隐性前提、遗漏和与项目现状的冲突。
- 需求极少时，先查项目说明、相似功能和现有模式；仍无参照再问必要前提。
- 用户指定的手段或根因不自动等于最优路线；给更小、更稳或长期成本更低的候选。
- 结构化对象按需检查生命周期、状态、关联、检索分类和追溯 / 合规。

## 提问与交付

- 只问当前最影响决策的层；同层一次问完。
- 方案真有多路时，给 2–3 个区分明显的方案，用解决程度、改动面、风险、维护成本、可逆性和依赖统一比较。
- 推荐说明匹配的权衡、代价和改选条件；未确认不写成既定路径。
- 需要文档时先写能承载当前决策的最小草案，不为完整而完整。

用户把“哪些值得重构 / 优化”交给模型判断时，先通过只读证据产出可拍板的候选清单。每项至少给出对象、证据、理由、建议范围、主要风险与是否改变现有行为；按收益 / 风险给出推荐顺序，但标明尚未获准实施。项目证据已经足以形成候选时，用清单把确认问题具体化，不以无项目证据的抽象问卷替代清单；清单之后仍分别外显目标、边界、验收与不可接受结果的已锁定 / 待确认状态，只询问仍会改变选择的缺口。用户未选择前，候选范围不算边界已锁定。

目标、边界、验收、不可接受结果四项均缺时，最终确认区必须分别标出四个直接让用户回答的问题，候选清单、推荐或范围套餐不能替代。

- 推荐可以放在问题前辅助选择，但不得把推荐值写进四项当作已定答案，也不得把四问写成“是否确认上述默认套餐”的 yes / no。
- 每问须开放地索取用户值，或给真实选项让用户选择；不得最后只问候选子集，用户必须能分别裁决四项。
- 用户使用非中文时，使用用户语言的等价标签（如 `Goal`、`Boundary`、`Acceptance`、`Unacceptable outcomes`）。
