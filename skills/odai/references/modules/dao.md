此为 odai 默认总控 workflow。

先明所求/所重/边界/所忧，再断模块/产物/止处。道定界，术定路，法定作验。

模块 id 与 frontmatter name 仍作 `dao`；对外统称 `道`。每次入题重判道/术/法/先手；诸内部模块皆借力对象，切入后仍由此总控裁决/推进/复核/收口。

此文件只补三层裁断、路由与 support file 触发；提问、确认、术语、静默与记忆等跨模块约束，统承 odai 入口。

## 执行要点

1. **承本入口总纲：谋定而后动。凡疑——足改路由/边界/验收/不可受结果者——皆结构化问之，禁瞎猜、禁默认补全。** 三层切勿混写。
2. 定道、收术前先对齐项目现实：README/现有文档/相似实现/命名目录/权限边界；能自证者先自证。
3. 定道先辨目标/手段/事实/猜测/权限/不可受之果。
4. 收术比主路/备路/代价/胜点/切换条件；能伐谋则不硬打。
5. 落法时凡称已知者，须能化为先手/验证/止损；先求最短闭环。新证若足摇上游，必回写道与术。
6. 切模块只算借力，非换 workflow；道未清、术未稳，只做搜索/阅读/补证/起草，不入实施。
7. blocked 仅限：越权未准、关键依赖不可得且无等价路、或续动直犯已定边界。blocked 须写阻因与最小解除条件。
8. 涉 UI：道层定体验目标与复用原则，术层定页面与交互路线，法层方落组件/token/状态/验收。
9. 非文档交付任务，默认不把讨论过程/候选草稿/推演痕迹写入项目文件；须留痕者，仅存最小必要 Markdown 结论。
10. 需起主文件/判层切换/增强尝试链/冻后复查时，分别读后列 support files。

## 路由与产物裁决

1. 先辨用户所言属目标/手段/抱怨/根因猜测/审查请求/实现请求，抑成果整理；勿按表面关键词硬路由。
2. 方向/边界/优先级/主路/取舍未定，停 `道`；开发推进、阶段切换、执行判定与收束，转 `harness-dev`。**主矛盾一经命中专项模块主责，即立切，不得久停 `道`/`harness-dev` 代做。**
3. 多模块同中，先按阻断/上游/主矛盾裁：破坏/不可逆/根配/全局先定界；体验行为先 `design-spec`；代码实现先 `implement-code`；规格/诊断先 `feature-plan`；项目说明先 `project-guide`；审查先 `review-sslb`；游戏本体先 `game-plan`/`game-design`；同步维护先 `skill-author`/`skill-sync`。切后若上游不稳，立回上游，不并跑。
4. 游戏系统/玩法/数值/经济/商业化/关卡/活动/叙事规划 → `game-plan`；完整游戏视觉方向 → `game-design`。
5. 通用规格/方案/需求理解/bug 诊断 → `feature-plan`；非游戏页面/交互/状态/视觉/体验/设计说明 → `design-spec`。
6. 代码实现/修复/补测/重构 → `implement-code`；项目级说明/README/规则/AI 接手基线 → `project-guide`。
7. 代码审查 → `review-sslb`；日报/commit/PR/成果归纳 → `ribao`；维护模块源文件 → `skill-author`；多端同步与 README 回写 → `skill-sync`。
8. 产物由你裁：未定足阻，乃出今判与问题组；余视边界/权限，出草案/规格/设计/执行单或直推收束。执行中对外只留今断/次步或阻因/解阻。
9. 凡对真实意图/模块选择/产物形态/边界/优先级/验收/不可受结果仍拿不准，皆一次性问清，不靠猜测续推。

## 三层路由原则

1. 道答：为何做、守何、不做何、界何在。
2. 术答：走何主路、胜点何在、为何不取他路、何时切换。
3. 法答：先何作、如何验、何时止、哪些结果回写上游。
4. 所调皆借力对象，非切流目标；调毕仍由本 workflow 接结果回三层续推。
5. 常借：道层 → `project-guide`/`game-plan`/`game-design`/`feature-plan`；术层 → `game-plan`/`game-design`/`design-spec`/`review-sslb`；法层 → `implement-code`/`ribao`。
6. 用户若直给动作或修法，先判其属道/术/法；上游未稳须直言卡层。能伐谋则先改认知/边界/协同/路线，不直入重实现。

## 默认流转与边界

1. 默认流转：理证 → 首问 → 确认 → 判道 → 收术 → 落法 → 复核 → 收束。确认前不冻方案、不定主路、不入实施；确认后并行仅供复析/复核/冻后复查。
2. 方案冻结回执行阶段后，按入口默认静默约分工推进；执行分工非增强模式。分工中若新证足改上游，立写回。
3. 道清、术稳、边界明，方入代码/配置/脚本/测试/构建。
4. 破坏/不可逆/大范围重构/项目级依赖/根配置/发布部署/外部访问/费用/认证与数据安全相关改动，皆先问用户。
5. 单工具败不即 `blocked`；先换等价路/缩范围/续做非实施动作。须沉淀则续写同一主文件；用户言“继续”时，优先恢复既有主文件/次步。
6. 复核不过即回写上层，不以“做过”充“做对”。

## 何时必须读取 support files

以下 support files 皆属 `dao`；对外统一写作 `道`。命中对应场景时必须展开：

- `references/dao/terminology-baseline.md`：涉问题整理/结构化提问/证据账本/工作草案/主文件字段/路径命名时必读。
- `references/dao/dao-shu-fa-playbook.md`：涉层间切换/压缩输出/主文件起草/常见失真判断时必读。
- `references/dao/parallel-consensus-playbook.md`：涉环境能力判断/多 agent 多模型合议/冻结后独立复查/分歧收束/用户复核升级时必读。
- `references/dao/model-selection-baseline.md`：总控已坐实当前用户真实可分配模型列表，且需给并行席位选模/校异构/解释未入完整合议时必读。
- `assets/dao/subagent-execution-template.md`：需拉起子 agent/组织冻结后独立复查/执行分工时必读；关键字段不全，不得进入并行形态。
- `assets/dao/main-template.md`：适合在项目内落第一版三层主文件时套用。

默认先按本文件推进；命中上述场景，再展开对应 support file。
