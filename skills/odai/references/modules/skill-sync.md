`skill-sync` 现在通过仓库内脚本 `scripts/skill-sync.js` 执行。

## 目标

默认同步统一入口 `odai`；若用户显式传入 skill 名称，也只接受当前仓库里真实存在的源 skill。

这里的职责是“同步分发”，不是“起草源文件”。当前仓库的标准源默认是 `skills/odai/`；内部模块与 support files 由统一 skill 目录直接维护，`skill-sync` 只负责在校验通过后把标准源同步到 Claude / GitHub / Trae，并回写 README。

## 输入约束

1. 默认无参同步 `odai`；若用户显式传名称，也只接受当前仓库里真实存在的 skill 名称。
2. 如果任一名称无法明确解析为 skill 名称，提示用户重新输入。
3. 只认 `工程根目录/skills/<skill-name>/` 目录，不从 `.agents` 目录取安装来源。
4. 不考虑依赖关系，不引入其他 skill 名称；统一入口内部模块不是独立同步对象。
5. 若传入多个名称，先完成全部校验，再逐个处理，避免只同步一半。

## 执行方式

1. Claude 命令层只做薄封装。
2. 实际生成逻辑统一由 `工程根目录/scripts/skill-sync.js` 完成。
3. 脚本是唯一执行真相；不要再让模型按规则手工生成目标文件。

## 脚本职责

1. 校验 `工程根目录/skills/<skill-name>/SKILL.md`、必要 support files 与 README 关键分节 / 锚点；任一关键校验失败就停止。
2. 以源文件 frontmatter 与正文作为唯一正文载荷；正文段落顺序、标题层级、代码块和措辞保持原样，不重组、不风格化重写。
3. 若同目录下存在 `references/`、`assets/` 或 `scripts/`，则把它们视为当前 skill 的附属资源，一并同步到各手动安装目标目录。
4. Claude 侧 wrapper 元数据只认 unified source frontmatter 中显式声明的字段；不要从现有 `.claude/` 产物反推或保留。
5. 生成手动安装文件时，要把入口与资源相对路径改写成目标目录内真实可用的路径，并直接覆盖 Claude / GitHub / Trae 目标文件。
6. 在当前仓库里，默认只处理 `odai`；若 frontmatter 提供 `replaces`，同步清理旧安装产物与 README 条目。
7. 更新 `README.md` 时，只处理“当前已提供”列表和对应分组落位，不修改 README 其他段落，不重写全文。

## 严格限制

1. 不修改 `.agents` 下任何内容。
2. 不扫描或操作无关目录。
3. 不发明新的文件格式，不扩展安装体系。
4. 不为了“参考现有格式”而搜索全仓文件。
5. 如果源 skill 不存在，不做任何写入。
6. 除 `工程根目录/skills/<skill-name>/` 与 `README.md` 外，不读取其他 skill 目录；统一入口内部模块属于当前 skill 目录的一部分，可直接处理。
7. 不为了“更像目标平台”而重写源 skill 正文；正文一律按源文件直拷，除固定头部、源仓库路径 / 资源相对路径改写与源 frontmatter 显式声明的 Claude 侧包装字段外不改。
8. 不调整正文章节顺序，不合并段落，不擅自增删规则。
9. 不自行推断平台差异，不自行设计额外字段；凡源文件未写明、规则未写死的内容，一律不补。
10. 写入目标文件时，凡规则中出现的 `$ARGUMENTS`，都按普通文本占位符处理，禁止展开、禁止替换、禁止引用当前传入参数。

## 失败输出

- 若任一源文件不存在：`错误：工程根目录/skills/<skill-name>/SKILL.md 不存在，无法生成手动安装版本。`

## 成功输出

成功时只简要说明：

1. 处理了哪些源文件；
2. 各自更新了哪些目标文件；
3. README 是已追加、已更新还是已存在；
4. 已按最小必要范围完成同步。
