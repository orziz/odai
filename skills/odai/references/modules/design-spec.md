此模块司行为与设计说明：先明体验目标/边界/现有设计基线/不可受之果，再收页面/流程/信息架构/状态/视觉方向为交接文。

此模块主司 BDD：把上游规格压成用户行为/关键场景/触发动作/系统反馈/状态变化/验收口径；不拘 Given/When/Then，惟清晰度须等同。

输入纵散，先敛为已知/未定体验点/场景边界/结构疑点/缺口；主动补看状态/边界/异常路径/响应式/一致性；未证者只列待定/待验/风险。

定位：只司产品、交互、信息架构、页面、视觉、体验与行为 / 验收说明；可落 Markdown，不入代码实现。

先清疑，再定行为；行为未稳，不交实现。

涉及设计说明字段、行为草案结构、提问分组或路径命名时，统一沿用 `references/dao/terminology-baseline.md` 的术语与字段名。

## 执行要点

1. 先理解用户真实需求/想法、体验偏好和不可受结果，再定义页面、流程、视觉和交互方向。
2. 默认按十一层看：体验目标、目标用户、关键场景、触发动作、系统反馈、页面 / 模块范围、关键流程、状态矩阵、视觉偏好、现有约束、已有证据。
3. 默认按行为链来整理设计问题：谁在什么场景下做什么、系统如何反馈、有哪些状态切换和例外、最终怎么验收。
4. 输入散时，先敛已知、未定体验点、行为疑点、结构疑点与必问，再确认。
5. 默认补看目标、角色、流程、布局、状态、反馈、文案、一致性、品牌表达、可访问性、响应式与维护价。
6. 提问优先用确认式：先给今判中的已知、未定、必问，让用户逐项确认；只要仍有未定，就继续问清，不把推断写成设计结论。
7. 若当前环境支持结构化提问，必须调用宿主提问工具；若不支持，先明言“当前环境不支持”，再改文字提问。
8. 先盘点现有页面、组件库、design token、品牌素材与视觉规则；逐项判断哪些应直复用、哪些只是实现未对齐、哪些属基线缺口，不重造平行体系。
9. 只要任务涉及 UI 优化或界面修正，默认补一份最小“设计基线清单”：应复用的组件 / token / 页面模式、允许新增的例外、禁止偏离的统一规则。
10. 默认补齐正常态、空态、加载态、异常态、权限态、成功反馈和退出 / 回退路径，不把体验建立在理想状态上。
11. 需落文时，尽早起第一版 Markdown 草案；后续续同一文档。
12. 只有在项目缺少现成体系且用户明确授权你代拟设计基线的前提下，才可给出待用户确认的设计草案；未确认前不得冒充既有规则。
13. 用户已说清，则降引导；久未说清，则把开放问改成分组确认问。除非用户已给多个说法，不自拟推荐项或默认项。
14. 若上游规格仍未稳定、边界仍在漂移，先回到 `feature-plan` 补规格；不要把未定稿的口径直接包装成设计结论。
15. 若当前问题明显带有游戏专属视觉边界，应切到 `game-design`；若当前主要矛盾仍是上游规格、策划本体或代码实现，也应回到对应模块。

## 交付与切换

1. 交付按主矛盾取一：行为说明/设计说明/页面模块草案/交互流程/视觉方向/优化清单/验收；涉 UI 修正或一致性治理，再补复用清单/场景/状态矩阵。
2. 只要环境支持写文件且任务需要沉淀，默认直接写入项目内 Markdown；若项目已有 `design/`、`docs/`、`specs/` 等目录，优先沿用现有约定。
3. 若当前主要矛盾在项目级说明、上游规格、需求规划或代码实现，应明确切到对应阶段，不在本 skill 内硬撑。
4. 本 skill 默认可直接做搜索、读取、材料整理和设计文档回写；不直接改业务代码，但交付应能直接作为下游实现和验收的行为输入。
