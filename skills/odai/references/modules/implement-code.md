此模块司实现。先明目标、边界、验收与不变项，再断直行抑或 TDD。

此模块默认两式：直行与 TDD。非事事皆当 TDD；惟目标行为已稳、逻辑险高、回归成本高、需重构保护，或用户明言测先行时，方启之。

输入纵散，先敛为实现目标、改动边界、不变项与验证计划；若实现路径、前提或验收仍不定，就先补确认或补证据，不把推断直接写进实现。

定位：只司实现、修复、重构、补测试与必要文档回写。若架构、需求、设计或执行边界未稳，只做最少必要确认，并明示应回上游。

先清前提，再下改动；前提未闭，不入改动。

## 模式选择

1. 默认使用直接实现模式：边界清楚、风险可控、没有明显测试先行收益时，直接改代码并做相关验证；能用更短、更直、更少层的实现达标，就不用更重方案。
2. 命中任一条件，优先切 `TDD`：用户明要测先行；bug 可稳复现；改动属核心规则/算法/状态机；任务为高风险重构；当前模块回归成本高。
3. 仅当目标行为/验收口径已稳，且可用自动化测试表达失败时，才真正进入 `TDD`；若行为仍漂浮或无法判正误，先回上游补规格或行为说明。
4. 一次性脚本、低风险样式/文案微调、纯搬运改动，或测试接缝明显不存在时，不强推 `TDD`。

## 实现与验证规则

1. 实现前先读相关代码、文档、测试与近似模式，不离上下文凭空写。
2. 输入散者，先敛实现目标、改动边界、约束与验证计划；路未定则先问后做。
3. 代码/测试/文档/日志/现有实现可直补之事实，不反问；补后仍有待确认前提，则不入实现。
4. 仍有待确认前提，或无法确认实现合用户真意者，停在补确认/补证据。
5. `TDD` 下，先写稳定失败测试表达目标行为或复现缺陷；先对外行为与验收，再最小实现使其过，并跑相关回归。
6. 优先局部改、局部验、必要抽象；先做减法：删旧码、并分支、减层级/参数/状态/依赖。多轮变更先查补丁叠补丁、旁路、临时兼容层与假通用层；能改旧路不开新路。代码求短链路、低嵌套、少状态、少分配、少往返、少间接层；抽象/组件化须有真实重复、稳定边界与维护净收益，不为预留未来造 helper/manager/wrapper/base/adapter/hook。
7. 已替历史遗留者，在同轮同界内删净死代码、废分支、废 props/状态/样式/测试与过期文档；不留新旧双轨。收尾只扫本轮改动文件、同函数/同类相邻废码及新增无用导出/类型/配置/兜底；跨文件旧债记风险，不借机全清。
8. 涉 UI，先读现有页面/组件库/样式系统/design token/设计文档/邻页模式；设计基线缺失/冲突/复用未知，或本轮含审美升级/品牌提质/Apple 感/Google 感/标杆站对标时，先升 `design-spec` 与其 `aesthetic-benchmark`，不自造规则；既有组件/页面模式可承接则复用或薄改；旧基线粗陋且伤核心体验/维护时，先交 `design-spec` 定最小改良基线。
9. 涉 UI，同查组件复用、一致性、正常/空/加载/异常/禁用/成功反馈、响应式、回退路径；不只修单一静态画面。
10. 同查功能正确性、边界、错误流、回退路径，以及监听/订阅/定时器/缓存/连接/持有引用是否久挂不释。
11. 性能与复杂度只优化真热路径、真瓶颈、真重复；无证据不为“可能更快”引入缓存、异步化、泛型层、配置层或过早优化。
12. 若本轮只是修小错，不借机重排架构、换范式、换状态模型或引入项目级约定；若确需升架构，先回上游说明收益与代价。
13. 改后优先跑相关测试/构建/lint/format/低风险验证；跑不了直说。涉 UI 而缺自动化时，至少补本轮设计基线、审美标尺、已查状态与未验证项。

## 优先级与冲突解决

当用户要求、项目现有代码、项目文档和生态约定冲突时，默认按以下顺序裁决：

1. 用户明确要求与边界
2. 项目现有代码、模式、目录与工具链
3. 项目文档、运行规则与上游设计
4. 当前语言、框架与生态主流约定
5. 本 skill 的兖底默认项

## 升档与越线边界

1. 改动若会改架构边界/模块职责/数据流/交互流程/长期抽象/项目级规范，或重定 UI 视觉/交互基线，先升方案/设计，不在实现阶段硬拍板。
2. 破坏/不可逆/大范围重构/项目级依赖/根配置/发布部署/外部访问/费用/数据安全相关改动，皆先问用户。
3. 默认不做全仓格式化、根级 lint/build/CI 改写，不因偏好改基础设施；除非任务明需且边界已清。
4. 不编造验证结果；没跑即没跑，没法确认即待确认。需求定稿/设计说明/项目级说明切对应模块；规格/行为未稳，不硬切 `TDD`。
