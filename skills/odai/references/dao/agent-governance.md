# Agent、模型与执行下放治理

## 何时读取

下一步包含直接切换模型、启动 / 委派 agent、并行分析、指定 / 比较模型或按成本路由时，在决定采用和实际调用前读取。需求判定、模块链路或执行前提已稳定到足以选择执行档位，且进入深分析、大范围读取、重执行或正式冻结前，也读取本文件先过一次模型 / agent 路由门。本文件只治理真实模型路由、agent 调用与执行下放；待冻结结论的独立挑战和合议征询归 `references/dao/decision-challenge.md`，提问、实施准入和真实性仍归 `references/dao/interaction-contract.md`。

**主流程路由能力**：宿主真实暴露的当前主流程模型切换，或可让当前主流程用指定模型继续执行的等价 model override。模型路由必须基于主流程路由能力，不等于启动 subagent；便宜 / 快模型只有在该能力真实可用、边界清晰且低风险时才可直办。若宿主只暴露 subagent 或新线程的 model override，不得包装成当前主流程直接切模型。Agent 可用来扩大证据面、生成候选路径 / 候选实现、并行只读审查、执行已冻结任务或做独立挑战；不得把 agent 数量当可靠性本身。候选生成可积极，采纳必须保守；未被主流程用证据复核的内容只作线索。

## 采用原则

- 只有宿主真实暴露相应能力且上层允许时，才能声称并行或调用了 agent / model；否则在主流程内反证。不得用同模型副本或只换提示词冒充多模型。
- 能由主流程低成本读证、跑验证或查文档解决的，不下放。
- 仅当任务可切分、边界清晰且收益明显高于冷启动与交接成本时使用 agent。单纯为了换便宜 / 快模型时，优先使用真实主流程路由能力；若宿主不支持，只能把 subagent / 新线程 model override 列为候选或退化路径，不得静默启动 subagent 冒充直接切模型。同模型仅为隔离上下文的调用走本门，不得包装成多模型或低成本执行下放；不为保留 agent 产出而改边界。
- 涉及成本档时，必须有宿主标签、现成价目或用户授权作证。成本档不明且会影响选择时，fail fast：输出必须分成「宿主可用档 / 能力」「未知成本项」「退化路径」三块让用户选，不凭记忆比价，不先做仓库排查或执行任务。
- 用户显式要求模型 / 成本 / agent 路由，或本门已裁定路由 / 下放收益明确时，候选存在但无法唯一选择，不猜、不静默采用。必须列出宿主真实暴露的模型 / agent 候选、已知标签、未知项和退化路径，给一个推荐及依据，再让用户选择或授权按推荐执行；非用户显式要求时，收益不明显或宿主能力不明则直接不下放。
- 实际切换模型或调用 agent 时，必须在用户可见输出中说明：采用对象、类型（直接切模型 / 只读 / 挑战 / 候选 / 执行 agent）、采用原因、授权或标签依据、边界与回交口径。执行后结果总结还必须只按真实发生情况回报；未实际调用不得列入参与者。

## 前置路由门

时点：入口判定完成、当前理解 / 边界 / 验收已足以选择执行档位，但尚未投入深分析、全仓扫描、重实现或正式冻结前。此门只做模型路由还是 agent 下放的短判断，不展开长分析。

门的输出只允许四类：

- 不下放：主流程低成本可解决、任务不可切分、需求未稳、收益不足或宿主能力不明。
- 直接切模型执行：任务简单、低风险、边界和验证已锁定，且宿主真实暴露主流程路由能力，并有真实便宜 / 快模型标签或用户已授权按成本路由；不另起 subagent，不制造交接成本。若只有 subagent / 新线程可指定模型，不命中本类。
- 先下放只读 / 候选，或转入挑战裁决：边界清晰，且并行证据、长上下文扫读、候选路线或候选实现能明显降低主流程成本；若动机是独立反证、挑战或待冻结结论，先读 `references/dao/decision-challenge.md` 裁决挑战题面，再回到本文件治理真实调用。
- 先停下确认：用户要求按成本 / 便宜模型 / 指定模型路由，或要求 agent 下放，但宿主成本档、可用模型、授权、停止条件或候选选择会影响路径。

只有用户显式要求按成本 / 指定 / 便宜模型或 agent 下放，且候选不唯一或标签不足；或本门已裁定路由 / 下放收益明确但候选选择会影响边界、授权、成本、停止条件时，才输出候选表并停下确认。其他收益不明显或宿主能力不明的情况，输出「不下放」并退回主流程，不为候选表阻断。需要候选表时，每个候选最多一行：候选名、类型（直接切模型 | 只读 / 挑战 / 候选 / 执行 agent）、已知能力 / 成本 / 上下文 / 可用性标签、适合点、未知项。随后给一个「推荐」和「为什么」，再问用户选择哪个候选，或是否授权按推荐执行。没有宿主真实暴露的候选时，说明无可选候选并退回主流程；不得凭模型名、厂商印象或记忆补标签。

优先考虑早路由 / 下放的信号：用户显式要求合议、多模型、指定模型、便宜模型或下放；待冻结结论高代价或难回退（先经 `references/dao/decision-challenge.md` 裁决是否挑战）；范围大到主流程深挖前可并行只读盘点；任务天然可分成独立审查 / 候选实现，或已由挑战裁决锁定一席挑战；已有执行单和验收，重执行量明显超过交接成本。

不得早派 subagent 的信号：仍有会改写目标、边界、验收或不可接受结果的待确认点；主流程一次读证或一次验证即可解决；只是简单局部改动；下放会要求子 agent 自行裁决范围、设计或验收。简单局部改动若要省成本，只有宿主支持主流程模型路由时才直接切模型；否则留在主流程或列退化路径，不走 subagent 交接。

## 执行下放

方案 / 执行单已锁死且无决策级判断时，可把重执行下放。下放前必须同时满足：宿主支持真实 agent / 模型档；范围、行为、验收、非目标和停止条件已锁定；执行量足以覆盖冷启动、交接与复核开销；主流程能读产物并运行必要验证。

最小下发包包含：任务题面、输入材料、允许动作、禁止越线、验收证据、停止条件和回交格式。不把背景聊天整段丢给子 agent；未读材料不得伪装成输入。涉及代码 / 配置 / 脚本写入时，实际写入方必须持有同版 `references/dao/interaction-contract.md` 与 `references/modules/implement-code.md` 全文，不得用摘要替代；无法确保就不得下放该写入段。

## 子 agent 通用边界

适用于执行下放、并行分析、审查和挑战席；各专项文件只能叠加，不得削弱：

- 子 agent 无用户通道，产出只是回交主流程的数据，不得冻结方案或宣布完成。
- 出现待确认点、冲突点、留中待问，或新证据触及范围、行为、验收时，立即停手回交；不得自行问用户或猜测后继续。
- 不再派生子 agent，不自行扩增轮次。
- 范围、设计、验收裁决只属主流程。
- 主流程复核（读 / 跑）通过后才可采纳，不得把子 agent 的自报完成当证据。
- 子 agent 输出必须区分已读材料、已跑动作、观察事实、推断、建议和未验证缺口；无法复验的结论只能当线索。
