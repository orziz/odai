# 验证状态与收口契约

本文件是验证状态、旧状态迁移和未验证收口的唯一 owner。生命周期状态以本节为唯一 owner；其他文件只说明何时读取本契约，不复制状态定义。

## 状态

只在实施型任务或需要跨会话恢复执行 / 验收时使用状态机；轻量只读、纯分析和单轮规划不强制套状态。

- `draft`：方向初成，仍可能调整。
- `reviewed`：方向与边界已复核，尚未取得执行准入。
- `ready_to_execute`：目标、范围、授权与验收足以开始实施，尚未实施。
- `executing`：正在实施或验证，不是完成态。
- `implemented_unverified`：实施已结束，但验收环境受限或关键实际场景尚未覆盖。
- `verified`：验收口径已由实际场景或等价强证据覆盖。
- `blocked`：当前没有不越权且有证据的后续动作。

正常路径为 `draft -> reviewed -> ready_to_execute -> executing -> verified`；环境受限时从 `executing` 进入 `implemented_unverified`，限制解除后继续验收。只有 `verified` 可以宣称总体完成。

## 证据与收口

收口前把每个验收点标为 `pass / fail / unverified`，并绑定证据与缺口；任一项不是 `pass` 时，总体不得写完成。测试、构建、lint、diff 或静态断言只证明其覆盖属性，不自动等于目标运行场景通过。

`implemented_unverified` 的最终答复必须具体写出：

1. 已实施、已核或已通过什么。
2. 哪些测试或目标运行场景尚未验证，以及原因。
3. 环境恢复后的最小复验步骤和通过口径。

发出答复前逐项核对以上三项，缺任一项不得收口；第三项至少包含复验对象、动作 / 环境与可判定通过条件。不得保留模板占位符，不得只写“未跑测试”。恢复此状态时先检查现有改动并补验收，不默认重新实施。

## 承接与旧状态迁移

承接上一轮验收时，第一个项目动作是从用户原话、可见主文件、执行单、`plans/` 或当前 diff 恢复目标与验收；恢复前不跑测试或宣称收尾。

旧主文件中的 `ready` 是双义状态，不得直接沿用。先核对当前代码、测试产物、current diff、构建产物或等价独立证据：

- 已实施且只缺验收 → `implemented_unverified`。
- 尚未实施且执行前提已齐 → `ready_to_execute`。
- 证据不足以区分 → 待确认，不得重新实施。

允许写入时回写新状态；禁止写入时仍在答复中写清“旧状态 → 新状态”、独立证据、未验证项和最小验收动作。
