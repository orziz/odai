# Diagnose Kit

Bug、报错、白屏、行为不符或性能回归进入诊断主阶段时读取。先遵守 `references/dao/inquiry-discipline.md`；回归与实施归 `references/modules/implement-code.md`。

## 默认路径

1. **对准现象**：把用户观察拆成可复述的环境、输入、步骤与结果。
2. **优先建红信号**：选择失败测试、CLI / HTTP 断言、日志观测、无头页面、trace 回放、最小 harness、差分或人在环验证中最便宜的一种。
3. **缩小空间**：对照最近变更、同仓成功实现、数据 / 控制流、配置差异和性能基线。
4. **最小修复与回归**：只修成立假设对应的缺口，重跑原始场景与有价值的相邻回归。

## 红信号质量

好的红信号应当：

- 能对准用户症状，而不只是“有错”。
- 结果可重复或已将偶发率提高到可调试。
- 尽量快，且可无人值守复跑。

先见红再修复仍是有稳定接缝时的首选，但不把“当前环境无法运行”等同于“不得做任何改动”。无法建红时，按 `implement-code` 的四条例外门决定是否可做局部可逆实施；否则停在补证。

## 偶发 / 跨层写入硬门

写入前必须同时满足：

1. 项目证据已明确把候选对象或调用链映射到用户所述页面 / 流程；“真实入口”特指该症状入口，不是任意可执行导出、渲染函数或相似对象。
2. 已有复现、日志、trace、失败测试或等价直接证据对准该症状。

任一不满足，本轮只列候选并索取最小补证，不写源码、测试或修复声明。项目同名问题记录、附近可疑代码、相似命名 / 功能，以及模型给相邻函数自行构造输入得到的异常，都不能单独通过本门；`implement-code` 的局部可逆例外也不得绕过症状映射缺口。

## 排查纪律

- 观察是事实；定位、根因和修法是待证假设。
- 偶发问题先循环、并发、加压、缩时窗或差分，再深挖代码。
- 偶发、环境相关或跨层症状若缺复现条件，先从项目证据补齐；仍缺时向用户一次询问环境、输入、步骤、频率与可用日志。附近的契约违例、可疑代码或一般性“可能导致”只能列为候选，不能替代症状验证或直接因果链。
- 项目里只有同名问题记录，只证明症状存在，不证明附近函数、拼写错误或契约违例属于该症状的调用链。目标页面、对象或使用链不存在时，把缺口当反证并索取最小补证，不改相邻对象代替原问题。
- 模型自行构造输入调用一个尚未证实属于症状链路的函数，即使得到异常，也只证明“该函数对该输入会失败”，不算对用户原症状的红信号；需先有调用链、日志、trace、真实入口或等价直接证据把两者连上。
- 探针只加能区分假设的观测，用唯一前缀并在收口清理；性能问题先量基线、profiler 或查询计划。
- 同一方向两轮未达成时，换证据路径或抽象层，不继续试变体。

收口区分：原始场景已验证消失；实施完成但只有静态证据的 `implemented_unverified`；仍待补证的假设。不用一条无关绿路径顶替用户症状。
