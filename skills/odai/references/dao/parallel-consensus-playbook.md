# 并行合议手册

## 何时读取

- 判能否开完整增强或辅助复核
- 判何时升档、何时降档、何时回原流程
- 需冻后独立复查，或需给子 agent 组统一下发包

## 公共定义

- **真实可用模型**：当前用户在当前会话、当前环境可选/可分配/可直调者。候选名单、后台库存、旧日志、历史权限、用户自述、模型自述、需额外安装/补录方得者，皆不算。原生派位、宿主/API 代查或实际拉起若回"超出成本层级 / 权限不足 / 额度不足 / 当前账号不可用"或等价拒绝者，即剔除。
- **同版 odai 契约**：每席至少承同版 canonical 入口、当前命中模块、必读 support files、当前题面/边界/验收/不可擅自假设项；不能可靠下发，则不入任何并行形态。
- **合法阶段**：确认前只许问题理解与提问准备；确认后方可再分析、方案收束、冻后独立复查或执行分工。
- **降档**：「辅助复核」「原流程」皆降档；不得称"多模型合议"或"完整增强"。

## 三种模式

- **完整增强**：最高保险档；多 agent、多模型，同题并判；至少三种当前真实可用之不同模型。默认不开；仅用户明令多模型合议或命中强触发时方试。
- **辅助复核**：一至二席子 agent 补证、反证或独立复查。需第二视角时之默认档。
- **原流程**：不并行；冻后之执行分工亦归此类。

同模型多席不算合议。需多视角而异构不立者，走辅助复核，或由主流程内置「反证+风险+补证」三视角。

## 模式成立条件

### 完整增强

1. 当前环境明确为 Copilot，且可稳定拉起并行子 agent。
2. 已尽直查、AI 续查与宿主/API 续查，可证至少三种真实可用之不同模型。
3. 同版 odai 契约可可靠下发。
4. 任务属中/大复杂，并行收益胜额外成本。
5. 当前阶段合法。
6. 用户明令多模型合议，或命中下列强触发之一：高代价冻结点（方向定案/规格冻结/架构主路/广范围审查结论）、原流程已收一轮仍遗会改写路线/边界/验收之分歧、误判回滚代价显著高于一次并行复核之本。

异构不立者，立即降至辅助复核；不得为形式硬凑同模型多席。

### 辅助复核

1. Copilot；至少一席子 agent 可稳定拉起。
2. 同版契约可可靠下发。
3. 本轮确有补证、反证或独立复查之收益。
4. 阶段合法。

默认一席；环境已露多模型且某模型明显更适做反证/风险审计时，可换模。用户强要增强而完整增强不立，且二席仍值其本者，可扩至二席。

不得包装为"多模型合议"。

## 模型能力自查口径

- 先查当前上下文、工具直出、会话日志/缓存、平台返回、会话资源与宿主同会话回填。
- 本地自查不足，回交总控；总控行 AI 续查；仍不足，则通知宿主/API 续查。
- 同会话、同环境内尚有未试取数路径者，不得以"当前环境未暴露"为终断口径。

## 启动前先做触发判定

不以"能并行"即必并行。完整增强不作常规第二视角；辅助复核亦只在确有第二视角收益时开。

### 排除（直走原流程）

- 单文件小改、小范围修辞、已知模式下之低风险修补。
- 主路单一、替代稀少、误判纠偏甚廉。
- 已冻方案下之执行分工、补证、实现或文档整理。

### 升降原则

- 用户明令多模型合议且强触发成立 → 试完整增强；异构不立则降辅助复核。
- 需第二视角但未达完整增强触发 → 直走辅助复核。
- 用户纵明言要增强，条件不立亦不得硬凑；只停本轮最高合法且收益胜成本之档位。
- 原流程推进中若新生主路冲突或高代价冻结点，立时重判。
- 问题已压成单主路、低风险或执行分工者，及时回原流程，不为形式留并行。

## 拉起子 agent 时之统一约束

- 主流程与诸席位皆承同版 odai 契约；不得主流程守规而子 agent 自由发挥。
- 一旦拉起子 agent，先读 `assets/dao/subagent-execution-template.md` 组统一下发包；不得以散乱 prompt 临时拼凑。
- 须席位选模，再读 `references/dao/model-selection-baseline.md`。
- 每个下发包只对一席；不得多席混于一包。
- 不能确认契约已可靠下发者，不得入任何并行形态。

## 主流程专属权限

- 是否开增强、何时降档/回原流程，皆主流程断之。
- 席位数、模型、关注点、是否冻题、是否独立复查或执行分工，皆主流程定之。
- 子 agent 只许在既定题面、边界与关注点内分析/挑战/补证/执行分工；不得自改题面、擅换主路、再拉新 agent，亦不得替主流程升级给用户。

## 阶段规则

### 确认前

只许整理问题定义草案、补证、提反证、校题面、备结构化问题。对外只落：今判、已知、未定与问题组。不得冻方案、定主路、入实施。

### 确认后

主流程先冻最小交付，再行并行：当前问题定义、范围/非范围与权限边界、成功标准/不可受结果与验收口径、当前可用证据与禁止擅自假设项、本轮待收敛之核心问题。

### 首轮合议（完整增强专用）

- 至少三席、至少三种不同模型；其中至多一席与主流程同模型。
- 默认至少设：一席反证/逆向挑战、一席风险/边界/验收、一席补证/实施代价/冻后复查。
- 关注点只为补挑战，非拆题分工；每席皆围绕同一核心问题给相对完整之判断。

### 收束与复议

主流程先三事：去重、裁冲突、查遗漏。能内部收束者，不交用户。惟剩余分歧仍足改写路线/边界/验收/不可受结果者，始交用户复核。

### 冻后独立复查

- 输入以冻方案、关键证据与今断为主；不转交整段讨论史。
- 能换模则换模；不能则至少换关注点。
- 新证若足摇冻判，立刻回写上游。

### 执行分工

- 方案既冻，方可拆实现/验证/补证/文档等子任务。
- 此属原流程执行组织，不得称增强。
- 各席仍承同一冻题、主路、边界与验收。

## 输出与留痕

- 收束须写本轮模式、实命中模块、实调用 agent、实调用模型；模式名限三档：完整增强（多模型合议）/辅助复核（非完整增强）/单流程；未暴露者直书"当前环境未暴露"。
- 未入完整增强者，须直陈因由：模型列表未露、仅一模型、异构不足、辅助复核亦不值，或实际拒绝所阻。
- 不转储无益争论、不堆原始分歧；只留最终判断、关键反证、是否需回写上游。
- 子 agent 输出越界者，主流程以失约论，回原流程重整。
