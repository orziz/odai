# 并行合议手册

## 何时读取

当你需要以下任一信息时，再读取本文件：

- 判断当前环境能否进入多 agent 多模型合议模式
- 需要自动拉起多个 agent、不同模型做方案合议
- 需要给并行席位分配模型，或判断当前暴露的模型列表是否足以成立完整多模型合议
- 需要在方案冻结后再组织独立复查
- 需要决定如何收敛分歧、何时交用户复核、何时回退原流程

## 四种模式

- 增强模式（完整多模型合议模式）：在完整能力成立时，围绕同一冻结题面与同一核心问题，让多 agent、多模型分别形成相对完整判断，先互相质疑纠错，再由主流程收束
- 当前模型三席模拟复核模式：在已完成自查、AI 续查和宿主/API 续查后仍无法合法证明 3 种不同模型，但当前环境仍能稳定拉起 3 个承接同一 skill 契约的 agent 时，默认复用主流程当前真实在用且可继续调用的模型拉起 3 个席位做模拟交叉复核；只有当前模型标识未暴露，或当前运行面无法显式钉住该模型但能可靠证明会沿用主流程同路真实可继续调用的默认模型时，才允许退到同路默认模型
- 子 agent 辅助复核模式：在无法确认多模型能力、但已具备子 agent 调用与统一 skill 下发能力时，使用 1 至 2 个子 agent 做题面校对、补证据、反例挑战或冻结后的辅助复查
- 原流程模式：沿用当前模块原有工作流；若模块自身已有并行辅助分析口径，仍按其原规则执行。方案冻结后的执行分工也归到这一模式，不单列为增强模式

新增增强模式不会改写原流程。它只是把子 agent / 多模型能力叠加到原流程同一条链路上，不会把“先理解问题、再向用户确认、再继续规划与执行”的顺序改掉。只要完整能力和部分能力都不成立，就保持原流程，不得虚构多模型参与。

只有第一种才叫“增强模式”。第二种和第三种都是降档方案：前者是同模型三席模拟，后者是 1 至 2 席辅助复核；两者都不得与增强模式或多模型合议混称。增强模式看的不是调起了多少不同名字的 agent，而是执行前的并行席位是否真的形成了足够的模型异构。
增强模式主要覆盖问题理解、方案讨论、决策收束与冻结后的独立复查。方案一旦冻结并回到执行阶段，允许按原流程分工推进实现、验证或交付，但那属于执行组织，不再称为增强模式。

## 完整能力成立条件

以下条件必须同时满足，才算“当前环境支持完整多模型合议”：

1. 运行环境被明确识别为 Copilot。
2. 当前环境明确支持并行辅助分析或并行 agent 调用。
3. 总控 agent 已在本轮里先使用当前环境已暴露给当前 agent 运行面的上下文、工具、原生 agent 能力、模型选择能力、当前会话绑定的日志 / 缓存证据与平台返回信息完成直查和 AI 续查，并最终确认当前用户在当前会话、当前环境里真实支持为并行任务分配模型，且这些模型未被当前主流程当前成本档位、权限或配额限制挡住，并可在执行前保证完整增强模式并行任务组至少包含三种当前用户真实可用的不同模型；若当前命中模块或当前子流程在本地判断点先做自查且结果不足，必须先回交总控，再由总控继续用当前环境可直接调用的取数路径做 AI 续查；这里的“平台返回信息”也包括宿主运行时在同一会话、同一环境内用平台原生 API 代查后，已经直接交到当前 agent 运行面的结果。其中最多可有 1 个席位与主流程同模型，但整个任务组仍必须达到至少三种当前用户真实可用的不同模型，不得用同模型副本、只换 agent 名称或只换提示词来补足异构性。
4. 当前环境明确支持把同一份 `odai` skill 约束传递给每个并行 agent，使其在各自模型下按同版 canonical 入口、当前命中模块与必要 support files 执行，而不是脱离本 skill 自行发挥。
5. 当前任务不是明显小问题，且并行收益高于额外成本。
6. 当前阶段属于合法并行阶段：要么是在用户确认前用于问题理解、需求澄清与结构化提问准备；要么是在用户确认后用于问题复核、方案收束或冻结方案后的独立复查。

若当前仍处于用户确认前，并行内容只允许用于理解问题、整理证据、准备提问与校验题面；不得在此阶段冻结方案、决定实施主路或进入实施。
执行分工不属于这里的合法并行阶段定义；它发生在阶段六回到原流程之后，但仍必须承接已冻结题面、主路、边界与验收。
这里的“当前用户真实可用”一律指：当前用户在当前会话、当前环境里不额外安装、不额外开通、无需另做人工补录，就真实可选、可分配或可直调的模型；仅仅在日志、缓存、后台名单或历史残留里出现过模型名，不足以算“真实可用”。若平台、宿主或原生派位接口已经明确返回“超出成本层级 / 权限不足 / 额度不足 / 当前账号不可用”或等价拒绝，该模型在本轮也不算“真实可用”。
若异构性不足，降档顺序与对外说明统一按后文“启动前先做触发判定”和“输出与留痕约束”执行。

## 当前模型三席模拟能力成立条件

以下条件必须同时满足，才允许进入“当前模型三席模拟复核模式”：

1. 运行环境被明确识别为 Copilot。
2. 当前环境明确支持至少 3 个子 agent 调用或等价的 3 席并行辅助分析。
3. 当前环境明确支持把同一份 `odai` skill 约束可靠传递给 3 个席位，使其按同版 canonical 入口、当前命中模块与必要 support files 执行，而不是脱离本 skill 自行发挥。
4. 总控 agent 已完成自查、AI 续查与宿主/API 续查，仍无法合法证明执行前可以凑齐至少 3 种当前用户真实可用的不同模型。
5. 当前环境默认必须能可靠地让 3 个席位继续使用主流程当前用户真实在用且可继续调用的模型；只有当前模型标识未暴露，或当前运行面无法显式钉住该模型但能可靠证明三席会沿用主流程同路真实可继续调用的默认模型时，才允许退到同路默认模型完成模拟复核；若当前模型标识未暴露，结果总结里必须明确写“当前模型未暴露”。
6. 当前任务不是明显小问题，且三席同模型交叉复核的收益高于额外成本。
7. 当前阶段属于合法并行阶段：要么是在用户确认前用于问题理解、需求澄清与结构化提问准备；要么是在用户确认后用于问题复核、方案收束或冻结方案后的独立复查。

说明：

- 该模式只算降档方案，不是多模型能力，也不是增强模式。
- 三个席位都必须围绕同一题面与同一核心问题给出相对完整判断，只允许关注点不同，不允许拆题分工。
- 若当前环境连 3 个承接同一 skill 契约的席位都拉不起来，或既无法可靠复用主流程当前模型、也无法在必要时可靠退到同路默认模型，就不成立。

## 部分能力成立条件

以下条件必须同时满足，才允许进入“子 agent 辅助复核模式”：

1. 运行环境被明确识别为 Copilot。
2. 当前环境明确支持子 agent 调用或并行辅助分析。
3. 当前环境明确支持把同一份 `odai` skill 约束可靠传递给子 agent，使其按同版 canonical 入口、当前命中模块与必要 support files 执行，而不是脱离本 skill 自行发挥。
4. 当前任务不是明显小问题，且子 agent 辅助收益高于额外成本。
5. 当前阶段属于合法并行阶段：要么是在用户确认前用于问题理解、需求澄清与结构化提问准备；要么是在用户确认后用于问题复核、方案收束或冻结方案后的辅助复查。

说明：

- 部分能力不要求环境已经明确暴露可分配模型列表。
- 部分能力也不要求主流程能证明本轮一定拿到三种不同模型。
- 只要模型多样性没有被环境明确证明，就不得把结果表述成“多模型合议”；主流程只能在“当前模型三席模拟复核模式”和“子 agent 辅助复核模式”等降档路径中择一，或回到原流程。

## 模型能力自查口径

总控 agent 在判断是否进入增强模式前，必须先做一轮环境自查；只要当前环境已经暴露了可直接读取的模型或 agent 信息，就不得跳过直查直接说“当前环境未暴露”。若当前命中模块、当前子流程或当前判断点先做了本地自查，但仍不足以确认模型列表或模型异构性，必须立即回交通知总控 agent 进入续查；总控收到回交后，必须继续用当前环境可直接调用的取数路径做 AI 续查，包括当前会话日志、调试日志、模型缓存、会话资源、平台返回信息、当前工具面可直调的查询路径，以及其他不要求用户额外操作的同会话取数方式；若 AI 续查后仍不足，则立即通知宿主在同一会话、同一环境内继续查，不得在中间层直接把“查不到”包装成最终结论。
只要同一会话、同一环境里仍存在未尝试、待回填或可由宿主继续代查的取数路径，就不得把“当前环境未暴露”当成最终结论，更不得据此直接 pass、直接降档或结束探测。

这轮直查只允许使用当前环境已经给到总控 agent 的能力与信息，例如：

1. 当前会话上下文里已经直接暴露的运行环境、agent 标识、模型标识或原生能力说明。
2. 当前工具面里已经直接暴露的原生 agent 调用能力、模型选择参数、模型标识或平台返回信息。
3. 当前会话绑定的调试日志、模型缓存、会话资源或其他同会话运行时留痕，只要来源仍绑定当前会话、当前环境，且字段足以指向当前用户真实可分配的模型或当前用户真实在用的模型标识。
4. 当前环境中本轮可直接调用的查询路径，例如当前工具面可直调的查询接口、当前环境内置的取数入口或其他不要求用户额外操作的同会话取数方式。
5. 当前主流程当前可见的成本档位、权限、配额与实际模型分配拒绝信息，只要这些信息已经直接暴露到当前 agent 运行面。
6. 宿主运行时在收到总控的续查通知后，于同一会话、同一环境内通过平台原生 API 代查并直接回填到当前上下文、当前工具返回或等价运行时直出字段里的模型 / agent 信息。

说明：VS Code / Copilot 的扩展 API 可能支持查询当前模型或候选模型，但只有当这些查询路径或查询结果已经直接暴露给当前 agent 运行面，且足以证明这些模型对当前用户当前会话、当前环境真实可用时，主流程才可把它当作“已暴露”的模型信息；平台层支持，不等于本轮 skill 已经拿到。若总控通过当前会话绑定的日志 / 缓存、当前环境可直调的查询路径，或宿主同会话回填实际拿到了结果，则这些结果都按直查 / AI 续查结果采信；若这些路径都没有实际给出结果，或只能证明“平台有这个模型名”而不能证明“当前用户真能用”，就不能因为平台理论上能查而视为已暴露。
同理，只有在总控已跑完上述自查、AI 续查与宿主/API 续查，且同会话同环境的可继续取数路径已经穷尽后，才允许把“当前环境未暴露”写成最终口径。若运行环境被明确识别为 Copilot，且当前子 agent / 派位链路不按成本限制，则理论成本档位标签、高成本标记或主流程自身成本感知只作记录，不单独作为模型剔除依据；只有当前原生派位、并行拉起或宿主/API 代查实际返回“超出成本层级”或平台明确说明当前链路受成本限制时，才按成本拒绝处理。
若任何一次当前原生派位、并行拉起或宿主/API 代查已经实际返回“超出成本层级 / 权限不足 / 额度不足 / 当前账号不可用”或等价拒绝，则按当前链路真实拒绝处理；但在明确识别为 Copilot 且当前子 agent / 派位链路不按成本限制时，理论成本档位标签、候选列表上的高成本标记或主流程自身成本感知都不算“超出成本层级”证据，只有实际返回的成本拒绝才可据此剔除模型。

以下信息不得当作模型列表已暴露的证据：

1. 用户自然语言声明。
2. 模型自述或子 agent 自行声称自己是什么模型。
3. 历史残留文本、旧日志、旧总结或未绑定当前会话的缓存信息。
4. 仅存在于平台层、扩展层或文档示例里，或宿主 / API / 缓存理论上能查但当前运行时并未把查询结果直接交给总控 agent 的查询能力。
5. 只显示后台候选、灰度试验名单、账号外库存、历史权限或其他无法证明当前用户在当前会话真能选、真能分配或真能直调的模型名。
6. 虽然出现在候选列表里，但当前主流程一旦实际派位就会因为成本档位、权限或额度被拒的模型名。
7. 需要用户额外安装插件、执行脚本、手工补录模型列表、补做环境配置或显式配合后才能得到的信息。

## 真实增强自测与验收口径

当当前任务本身是在测试、验收或校准完整增强能力，或总控准备依据本轮新暴露的模型列表首次把复杂任务自动升级为完整增强模式时，还必须补做一次最小真实派位探测；只凭模型列表、候选清单、单次未钉模型的成功返回，或某个席位的口头自报，都不足以把完整增强写成“已验证可用”。

真实增强自测与验收至少按以下顺序执行：

1. 先完成当前会话模型暴露检查，明确主流程当前模型 id、至少 2 个不同候选模型 id，以及权限、配额与实际派位拒绝过滤结果；若当前为 Copilot 子 agent 链路且成本不构成限制，则成本标签只记录，不纳入预过滤。
2. 若当前运行面支持显式选模，则必须优先组装“主流程当前模型 + 两个已过滤后的不同模型”的最小三模型探测组，并对同一只读小任务分别显式钉模型发起 3 个最小派位探测；探测任务不得依赖未钉模型的默认派位，也不得把多席位混在同一次调用里。
3. 只有当 3 个显式探测都成功返回，且未收到权限、配额、账号不可用或等价拒绝，也未在当前链路收到明确成本拒绝时，才可把完整增强模式记为“当前会话已验证可用”；任一席位若被拒绝，必须先剔除该模型并尝试其他合法候选，凑不齐 3 个不同模型就不得宣称真实增强成立。
4. 若当前运行面虽然暴露了候选模型列表，但本轮子 agent 调用不支持显式选模，或无法把显式模型 id 稳定传给各席位，则这轮最多只算“候选模型已暴露”，不能算“真实增强已验”；除非宿主/API 已直接回填等价的实际派位成功结果，否则仍应继续降档或回原流程。
5. 自测或验收结论必须显式写出：本轮探测实际使用了哪些模型、哪些模型因实际成本拒绝、权限或配额被过滤或被拒、最终是否达到三模型显式派位成功；若只是理论成本标签而未构成当前 Copilot 子 agent 链路限制，则不计入过滤理由。

若总控 agent 已按上面口径完成自查、AI 续查，并在需要时已发出对宿主/API 的续查通知，仍无法确认当前用户真实可分配的模型列表或当前用户真实可用的模型异构性，就不得乐观放行完整增强模式；后续降档顺序统一按“启动前先做触发判定”执行，不在本节重复展开。
若最终未进入任何增强模式，结果总结统一按“输出与留痕约束”执行；若当前任务确实存在增强收益，仍应补一句本轮未启用更合适的原因，或距离可启用还缺哪些前提。

## 启动前先做触发判定

默认不开小问题增强。总控必须先根据任务特征做一次复杂度初判和触发判定；只要完整能力或部分能力成立，不等于本轮就应该自动开启，但中/大复杂任务也不允许在未判完前被总控直接保守跳回原流程。

复杂度初判先看五类信号：

1. 同时存在两条以上合理主路，且取舍会改变范围、边界、验收、成本或风险接受度。
2. 任务跨两个以上专项模块，且模块间存在真实牵引。
3. 当前裁决属于高代价冻结点，例如方向定案、规格冻结、架构主路或广范围审查结论。
4. 原流程已经做过一轮收束，但仍留有会改写路线、边界、验收或不可接受结果的剩余分歧。
5. 若不开并行复核，误判后的回滚与纠偏成本会显著高于一次增强或降档复核的额外成本。

若命中其中任意两项，或单独命中第 3 项 / 第 5 项且证据明确，可先按中/大复杂任务处理；若同时明显命中排除信号，仍以排除信号优先。

触发判定至少先问四件事：

1. 当前是不是讨论/决策阶段或冻结方案后的独立复查，而不是已冻结方案下的执行分工、补证或落地实现。
2. 当前是否命中了上面的中/大复杂信号，或用户明确要求多模型第二视角。
3. 若不开增强模式或辅助复核，误判成本是否会显著高于一次并行复核的额外成本。
4. 当前任务是否其实落在单文件低风险修补、小范围修辞、已知模式下的局部实现，或只是把已冻结方案拆给不同席位执行。

若第 1 项不成立，或第 4 项明显成立，则直接回原流程。

若第 1 项成立，且第 2 至第 4 项综合判断后显示当前属于中/大复杂且未命中明显排除信号，则总控默认优先进入“增强尝试链”：先检查完整增强模式；完整条件不成立时，再检查当前模型三席模拟复核模式；再不成立但部分能力成立时，再检查子 agent 辅助复核模式，并停在当前环境允许、且收益仍高于额外成本的最高合法档位。

若未命中中/大复杂初判，则只有在第 2 至第 4 项综合判断后仍显示增强收益明显时，才值得进入增强模式、当前模型三席模拟复核模式或子 agent 辅助复核模式。

优先开启合议增强模式的强触发信号：

1. 当前任务已命中中/大复杂初判，且仍处于问题理解、方案讨论、决策收束或冻结方案后的独立复查阶段。
2. 用户明确要求多 agent、多模型合议，或明确要求冻结方案后的独立复查。
3. 同时存在两条以上合理主路，且取舍会改变范围、边界、验收、成本或风险接受度。
4. 任务跨两个以上专项模块，且模块间存在真实牵引；一旦误判，就会把后续实现、设计或审查整体带偏。
5. 当前裁决属于高代价决策，例如方向定案、规格冻结、架构主路、广范围审查结论。
6. 原流程已经做过一轮收束，但仍留有会改写路线、边界、验收或不可接受结果的剩余分歧。

默认不开合议增强模式的排除信号：

1. 单文件小改、小范围修辞、已知模式下的低风险修补。
2. 现有模式成熟、主路单一、几乎不存在真实替代方案。
3. 即使判断失误，回滚和纠偏成本仍然很低。
4. 当前只是把已冻结方案拆成实现、验证、补证或文档席位推进。

若用户明确要求增强模式或冻结方案后的独立复查，但完整能力不成立，不得硬凑伪增强模式。总控必须先按当前环境已暴露能力允许的最高强度完成能力探测：至少覆盖自查、AI 续查，以及当前原生并行 / 模型分配面若已暴露时的最小必要模型分配探测；若当前任务是在测试、验收或校准完整增强能力，或总控准备依据本轮新暴露的模型列表首次把复杂任务自动升到完整增强模式，还必须按上面的“真实增强自测与验收口径”完成显式三模型派位探测。
只要同会话同环境里仍存在未尝试或待回填的宿主/API取数路径，就不得把“当前环境未暴露”当作允许降档的充分条件。若本地自查不足，必须先回交通知总控；若总控的 AI 续查仍不足，必须再通知宿主/API 继续查。只有这些尝试仍无法证明完整能力时，才允许降档。
降档顺序统一按上面的“增强尝试链”执行；若进入强制请求下的最高合法降档，则优先拉满当前模型三席模拟复核，不能成立时再看 2 席辅助复核。每降一档都要重判收益；若更低一档也不值得，直接回到原流程，并在结果总结里写明为何“不启用更合适”或仍缺哪些前提。

若当前任务未命中中/大复杂初判，或已明显命中排除信号，就先走原流程；若已命中中/大复杂初判，则先走增强尝试链并停在最高合法档位。无论当前落在哪条路径，总控都必须保留动态升降档能力。原流程推进中，一旦出现新的主路冲突、跨模块牵引、高代价冻结点，或一轮内部收束后仍留有会改写路线、边界、验收或不可接受结果的剩余分歧，应立即重做触发判定：完整能力成立时切入完整多模型合议模式；若完整能力不成立但当前模型三席模拟能力成立，则切入当前模型三席模拟复核模式；若只满足部分能力，则切入子 agent 辅助复核模式。反过来，若问题已经压缩成单主路、低风险、已冻结方案下的执行分工或单纯补证，也应及时回到原流程，不为形式保留并行。

触发判定可以在首轮提问前做第一次，也可以在用户确认后结合新增信息再做一次；两次判定结果允许不同。

## 拉起子 agent 时的统一 skill 约束

- 主流程、并行 agent 与所用模型都属于同一条 `odai` skill 链路；无论是完整多模型合议，还是子 agent 辅助复核，都不是主流程按 `odai`、子 agent 自由发挥的混搭模式。
- 每个并行 agent 至少都要拿到同一版 canonical 入口、当前命中模块、必要 support files、当前题面（确认前阶段为问题定义草案，确认后与执行分工阶段为冻结题面）、边界、验收口径与禁止擅自假设项。不同模式下的席位数量、模型异构要求、同模型回退与关注点覆盖，统一按上文模式定义和 `references/dao/model-selection-baseline.md` 执行。
- 主流程一旦决定拉起子 agent，无论是做合议、辅助复核还是已冻结方案后的执行分工，必须先读取 `assets/dao/subagent-execution-template.md` 并按模板组装统一下发包；不得继续使用散乱、字段缺失或每席位口径漂移的临时 prompt。
- 若主流程需要为席位选模型，无论是完整增强还是当前模型三席模拟复核，都必须再读取 `references/dao/model-selection-baseline.md`；若当前环境已暴露权限、额度或实际派位拒绝信息，也要先写进模型判断，再组装统一下发包。
- 统一下发包按席位单独生成；每个下发包只对应 1 个子 agent、1 个模型和 1 份明确关注点，不得把多席位混在同一包里。
- 若已回到原流程但仍需拆出执行分工，统一下发包里必须显式写清冻结题面、既定主路、边界、验收与本席位分工范围；不得把尚未收束的方案讨论伪装成执行分工。
- 只有主流程已经拿到当前命中模块、当前必读 support files 与模板关键字段时，才允许为子 agent 组包；主流程自己都拿不齐时，直接回退原流程，不得猜补。
- 若当前环境能拉起子 agent，但不能把统一 skill 约束可靠下发给并行 agent，或无法确认子 agent 已按同一 skill 契约执行，就不得进入任何增强模式。

## 主流程专属权限

- 是否开启或关闭增强模式、何时回退原流程，由主流程决定。
- 调起哪些子 agent、给每个席位分配什么模型与关注点、是否重跑确认前后整条链路、是否进入冻结后的独立复查、是否拆出执行分工，都由主流程决定。
- 冻结题面、冻结方案、宣布已收束、发起定向复议、升级给用户复核、以及结果总结里回报实际调用的 agent 与模型，都由主流程负责。
- 子 agent 只负责在统一下发包约束内完成围绕同一题面的独立分析、挑战、审计、复查，或在已冻结方案下执行本席位分工；不得自行改题面、换主路、再拉新 agent、替主流程升级给用户，或把本席位输出包装成最终裁决。

## 统一下发包最小字段

统一下发包至少包含以下信息；任一关键字段缺失，都视为不满足可靠下发条件：

1. 同版 canonical 入口路径
2. 当前命中模块路径
3. 当前必读 support files 清单
4. 当前阶段与本席位关注点；若已进入执行分工，则这里还要写清本席位分工范围
5. 本席位 agent、本席位模型与模型选择理由
6. 当前理解、当前理解的用户诉求、当前已确认的用户真实想法与在意点
7. 已知事实、已验证事实、未确认点、冲突点
8. 当前题面（确认前阶段填写“问题定义草案”，确认后与执行分工阶段填写“冻结题面”）、范围 / 非范围、成功标准、不可接受结果
9. 禁止擅自假设项、输出要求、失败回退口径
10. 若当前环境已暴露成本档位、权限、额度或派位拒绝信息，还要写清：本席位模型为何仍可合法派位，或为何已改走更低一档降档

## 完整多模型合议模式

### 总链路

- 总控先判是否开完整多模型合议模式。
- 若开启，先在确认前让多模型并行分析用户真实意图、需求、问题与待确认点。
- 必须输出结构化问题并等待用户确认；增强模式只提高理解质量，不替代提问确认。
- 用户确认后，再做一轮问题复核与再分析。
- 再进入方案规划、内部收束与冻结方案后的独立复查。
- 只有仍会改变路线、边界、验收或不可接受结果的剩余分歧，才交给用户复核。
- 分歧收束后，回到原模块后续执行。

### 阶段零：确认前的问题理解与题面校验

主流程先形成当前问题定义草案，但不得把它当成已确认题面、冻结方案或实施主路。

- 至少要有 1 个并行 agent 负责问题定义挑战或校对；该席位优先使用不同于主流程的模型。若当前已经决定开启执行前完整增强模式，则确认前并行任务组的总体配模仍应满足至少三种不同模型；若存在与主流程同模型的子 agent，该席位也必须围绕同一题面提出独立判断和不同挑战点，而不是重复主流程判断。
- 问题定义校验至少要检查四件事：问题定义是否偏题、用户真实意图是否被偷换、范围与非范围是否漂移、现有证据是否足以支撑当前理解。
- 当前阶段对外只允许落到：当前理解、已知事实、未确认点与结构化问题组；不得跳过提问直接给定案。
- 只要问题定义校验指出了会改变主路、边界、验收或不可接受结果的理解偏差，就不得进入后续规划合议；必须先补证据或向用户确认。

### 阶段一：结构化提问与用户确认

- 不论是否进入增强模式，结构化提问与用户确认都是必经门槛。
- 完整多模型合议模式在这一阶段的目标，是把问题问准、问全、问到会影响路线、边界、验收与不可接受结果的点，而不是绕过用户确认。
- 未得到用户确认前，不得冻结方案、定实施路线或进入实施。

### 阶段二：确认后的再分析与冻结问题定义

收到用户确认后，主流程必须结合确认结果再做一次问题复核；完整多模型合议模式可在这一轮再次并行调用多模型做重新校准。

主流程先冻结以下内容，再分派并行任务：

- 当前要解决的问题定义
- 当前范围、非范围与权限边界
- 当前成功标准、不可接受结果与验收口径
- 当前可用证据与禁止擅自假设的部分
- 本轮要收敛的核心问题

若用户确认后仍出现新的关键不确定，并且会改写路线、边界、验收或不可接受结果，就回到提问确认，不在未重新确认的题面上继续细化。

### 阶段三：方案规划与首轮合议

- 默认至少 3 个并行 agent，且至少使用 3 个不同模型；复杂任务可增到 4 至 5 个，但不要仅为形式扩大并行规模。
- 并行任务组至少包含 3 种不同模型；其中最多可包含 1 个与主流程同模型的子 agent，但不得让整个任务组退化为 1 至 2 种模型，也不要复制多个同模型同关注点席位。
- 每个 agent 都要拿到同一版 `odai` skill 约束、已确认题面与边界，并围绕同一核心问题形成相对完整判断；允许首轮结论不一致。
- 首轮合议默认至少包含 1 个反证挑战关注点席位。该席位的任务不是为了形式站到对立面，而是基于同一题面与现有证据，尽力提出足以推翻暂定主流判断的最强反例、边界缺口或证据缺口；若首轮已经自然出现实质分歧，则不再额外强制制造一个相反结论。
- 可以为不同席位设置不同关注点，例如默认假设、边界与验收、风险定级、证据缺口或实施代价；但关注点只是附加挑战角度，不是拆任务分工。在本阶段，任何席位都不得只做自己那一块然后把其余部分留给别人；执行阶段若已回到原流程，可另按已冻结方案分工推进。
- 每个席位的统一下发包都应显式写出：当前阶段、本席位关注点、当前冻结题面、同一核心问题、必读文件与失败回退口径，避免子 agent 把自己误当成主流程，或误把合议做成分工。

### 阶段四：主流程收束与定向复议

主流程必须先做三件事：

1. 去重：合并等价结论与重复风险
2. 裁冲突：区分真分歧、表述差异与证据不足
3. 查遗漏：确认是否有任何关键边界、验收或风险被漏掉

能内部收敛的，必须先内部收敛，不把多份原始结论直接交给用户。

- 若首轮返回结果过快一致，主流程不得直接把“一致”视为可靠共识；必须先确认反证挑战席位已经针对暂定共识完成过一轮有效挑战。若尚未完成，先补一轮定向反证复议；只有反证后仍站得住，才允许进入已收束判断。

主流程在这一阶段只允许做“基于已返回结论与现有证据的收束”，不允许脱离返回结果与现有证据自行扩写主路、边界或风险。具体限制：

1. 不得新增首轮无人提出、且证据账本也不支持的新主路、新边界或新风险。
2. 若要否决多数意见、压过少数异议或改变某项风险定级，必须给出明确依据：项目事实、用户确认、代码或文档证据、测试或日志结果，或定向复议的新结果。
3. 若拿不出依据，就视为尚未收束，进入定向复议或交用户复核，而不是由总控直接裁定。

主流程在宣布“已收束”前，内部至少要对齐下面四项：

- 哪些是共识项
- 哪些是真分歧
- 每个关键判断分别由哪些 agent、哪些模型支持
- 每项裁断背后对应的证据是什么

上述四项未对齐前，不要宣称已经形成共识。

若首轮仍有实质冲突，主流程应优先发起定向复议，而不是立刻把问题升级给用户。定向复议只围绕仍会改变以下内容的分歧：

- 若首轮本就存在实质冲突，则不需要再人为制造反方；直接围绕真实分歧做定向复议与收束。

- 主路或备路选择
- 方案边界
- 验收口径
- 关键风险定级
- 不可接受结果
- 暂定共识能否经得住反例、边界缺口或证据缺口挑战（仅在首轮过快一致且尚未完成有效反证时启用）

### 阶段五：冻结方案后的独立复查

当主流程已经形成冻结方案、冻结规格或冻结裁决后，若当前环境仍满足完整能力条件，可再拉起一轮独立复查：

- 复查优先使用不同于首轮的 agent 组合与模型组合；若模型组合变化有限，至少更换关注点与挑战角度
- 复查输入优先给冻结后的方案、关键证据和当前裁决，不直接转交整段讨论历史
- 复查目标是验证遗漏、边界与验证链条是否成立，而不是重复首轮表述
- 若复查阶段已无法维持至少三种不同模型，就不继续称为完整增强模式；若当前模型三席模拟能力仍成立，应先降到当前模型三席模拟复核模式，否则再降到子 agent 辅助复核模式

### 阶段六：回到原流程执行

当问题定义、方案主路、边界与验收已经收束后，回到当前模块原流程继续规划落法、实施、验证或审查。

- 若执行效率或专业性需要，可在这一阶段按实现、验证、补证、文档等子任务分工推进；但分工前提是题面、主路、边界与验收都已冻结。
- 执行分工属于原流程执行组织，不再叫增强模式；任何执行席位都不得擅自改题面、改主路、改边界或重写验收。
- 执行中若出现足以动摇上游判断的新证据，仍要回写上游，而不是把完整多模型合议做成一次性前置仪式。

## 当前模型三席模拟复核模式

当完整能力不成立，但已完成自查、AI 续查与宿主/API 续查后仍无法合法证明多模型异构，且当前模型三席模拟能力成立时，允许进入“当前模型三席模拟复核模式”。

### 目标

- 在不虚构多模型能力的前提下，用 3 个承接同一 skill 契约的席位提高交叉复核密度。
- 在用户明确强制要求增强模式，或当前任务确实需要三席交叉复核时，提供比 1 至 2 席辅助复核更强的同模型模拟第二视角。

### 基本限制

- 不得把该模式表述成“多模型合议”或“增强模式”。
- 三个席位都必须围绕同一题面与同一核心问题给出相对完整判断，只允许关注点不同，不允许拆题分工。
- 默认固定 3 个席位；若当前环境拉不起 3 个席位，就不成立。
- 默认优先显式钉住主流程当前真实在用模型；只有当前模型标识未暴露，或当前运行面无法显式钉住但能可靠证明会沿用主流程同路默认模型时，才允许退到同路默认模型。
- 三席至少覆盖：1 个反证 / 逆向挑战席位、1 个边界 / 风险 / 验收挑战席位、1 个证据补强 / 实施代价 / 冻结后复查席位。
- 若当前模型标识未暴露，结果总结里必须明确写“当前模型未暴露”，不得伪造当前模型 id。

### 适用场景

1. 用户明确强制要求增强模式，且当前任务属于高代价决策、复杂收束或冻结后的高风险独立复查。
2. 完整多模型条件无法合法证明，但 1 至 2 席辅助复核又明显不够覆盖当前冲突面、边界面和证据面。

### 不适用场景

1. 明显小问题。
2. 当前环境无法可靠下发同版 `odai` skill 契约。
3. 当前环境无法保证 3 个席位优先承接主流程当前模型，且在需要退档时也无法可靠承接同路默认模型。

### 推荐链路

1. 主流程先冻结本轮要复核的问题与边界。
2. 组装统一下发包，显式标注当前模式为“当前模型三席模拟复核模式”。
3. 默认直接拉起 3 个席位，分别覆盖反证、边界 / 风险、证据 / 成本等关注点。
4. 主流程吸收结果后内部收束；能内部收束就不要升级给用户。

### 输出要求

- 对用户的统一表述与结果总结写法，统一按“输出与留痕约束”执行。
- 若进入当前模型三席模拟复核模式，必须显式写出：已完成自查、AI 续查与宿主/API 续查；完整多模型条件仍未成立，因此已进入同模型三席模拟降档；并说明本轮实际拉起了 3 个席位，以及当前模型标识是否暴露。

## 子 agent 辅助复核模式

当完整能力不成立、当前模型三席模拟能力也不成立，或主流程判断本轮只需要较轻第二视角且部分能力成立时，允许进入“子 agent 辅助复核模式”。

### 目标

- 用当前环境已可用的子 agent 能力，提高题面校对、证据补强、反例挑战、读档复核和冻结后辅助检查的质量。
- 在不虚构多模型能力的前提下，尽量自动调用子 agent，而不是一律退回纯单流程。

### 基本限制

- 不得把辅助复核表述成“多模型合议”。
- 若当前环境未暴露子 agent 的模型标识，结果总结里必须明确写“当前环境未暴露”。
- 默认只拉 1 至 2 个子 agent；若任务足够大，可设置不同关注点席位，但不要为了形式堆数量，也不要把当前这轮复核拆成各做一块；若方案已冻结并进入执行阶段，可另按原流程分工。
- 若用户明确强制要求增强模式，但完整能力和当前模型三席模拟能力仍无法证明，而部分能力成立，则这里默认进入“最高合法降档”的更低一档：优先拉 2 个子 agent，除非当前环境或任务规模明确不允许。
- 即使多个席位都由同一模型或未暴露模型完成，只要它们都围绕同一题面做独立复核、关注点不同、边界清楚、统一 skill 下发可靠，仍可作为“子 agent 辅助复核”成立。

### 适用场景

1. 中等复杂度任务，需要一个独立席位做题面校对或反例挑战。
2. 需要快速补证据、补目录、补相邻文档读取，但不值得开完整多模型合议。
3. 用户明确希望“别停在单流程里自己想”，希望自动调一个子 agent 做第二视角检查。

### 不适用场景

1. 明显小问题。
2. 子 agent 收益低于额外成本。
3. 当前环境无法可靠下发同版 `odai` skill 契约。

### 推荐链路

1. 主流程先冻结本轮要复核的问题与边界。
2. 组装统一下发包，显式标注当前模式为“子 agent 辅助复核模式”。
3. 常规情况下先拉起 1 个子 agent 做题面校对、反例挑战、补证据或冻结后辅助复查；若是用户强制要求增强模式但完整能力和当前模型三席模拟能力都不成立，则默认直接拉起 2 个子 agent，分别覆盖主路反证与运行约束 / 边界 / 验收挑战。
4. 若仍有真实主路分歧或风险分歧，再由主流程决定是否补齐到第 2 个子 agent，或在强制请求下说明为何当前环境无法补齐。
5. 主流程吸收结果后内部收束；能内部收束就不要升级给用户。

### 输出要求

- 对用户的统一表述与结果总结写法，统一按“输出与留痕约束”执行。
- 若只进入辅助复核模式，不得使用“多模型一致认为”“多模型合议结论”等字样。
- 若用户明确强制要求增强模式但最终只进入辅助复核模式，结果总结还必须显式写出：已按强制要求完成最高强度能力探测；完整多模型条件和当前模型三席模拟条件仍未成立，因此已进入更低一档合法降档，并说明本轮实际拉起了几个复核席位。

## 两类用户确认不要混淆

- 理解确认：这是原流程、完整多模型合议模式、当前模型三席模拟复核模式与子 agent 辅助复核模式都必须经过的门槛，发生在规划和实施之前；增强模式只能帮助把这一步问得更准，不能跳过它。
- 分歧复核：这是增强模式内部收束失败后的升级动作，只在完成内部收束或定向复议后、仍存在会改变路线与边界的剩余分歧时才交给用户。

## 何时才交给用户复核

只有满足以下全部条件时，才把分歧升级给用户：

1. 主流程已经完成至少一轮内部收束或定向复议
2. 分歧仍然真实存在，不是措辞差异或信息重复
3. 分歧会实质影响路线、边界、验收、风险接受度或不可接受结果
4. 该分歧无法通过项目事实、现有材料或低风险验证自行消除

交给用户时，只提交“最小必要分歧清单”，每项都要写清：

- 分歧点是什么
- 各选项分别意味着什么
- 对范围、成本、风险或体验的影响
- 为什么当前裁断需要用户确认

## 一致性判定

满足以下任一情况，可视为“已达成一致”：

- 所有并行判断在主路、边界、验收和关键风险上已经对齐
- 剩余差异只影响措辞、展示顺序或低优先级建议，不影响当前裁决
- 主流程已用现有证据压平了冲突，且没有留下会改变路线的未知项

未达成上述条件前，不要宣称“已经一致”。

## 主流程约束

- 不得用个人偏好覆盖多 agent 的共同结论。
- 不得无视问题定义校验指出的关键偏差。
- 不得把未被证据支持的判断包装成“综合裁定”。
- 不得把实质单模型结果或模型未暴露的辅助复核描述为多模型合议。

## 输出与留痕约束

- 对用户只输出一份统一表述，加最小必要的待用户复核项。
- 不转储多 agent 原始讨论记录，不把过程性争论当主交付。
- 若用户明确要求留档，最多只在 Markdown 中保留最小必要结论、剩余分歧和最终确认结果。
- 不得编造“不同模型”或“多 agent”已参与；能力不明时宁可回退原流程。
- 结果总结必须明确写出：本轮实际进入的是完整多模型合议模式、当前模型三席模拟复核模式、子 agent 辅助复核模式，还是原流程 / 执行分工；若环境未暴露当前模型或模型列表，也要如实写“当前模型未暴露”或“当前环境未暴露”。若未进入增强模式，但已经做过能力判定，还应显式补出本轮为何未进入增强模式；若进入的是当前模型三席模拟复核模式或子 agent 辅助复核模式，也应显式说明为什么没有进入完整多模型合议模式；能具体到条件层，就不要退化成模糊表述。
- 若用户明确强制要求增强模式，且最终仍未进入完整增强模式，结果总结必须进一步显式补出：本轮已完成哪些能力探测步骤、是否已进入当前模型三席模拟复核模式或其他最高合法降档、以及为何仍无法合法宣称完整增强模式。
- 若未进入增强模式且当前任务仍存在明显增强收益，结果总结应再补一句：若启用增强模式，主要会提升哪一项决策质量、复核强度或收束把握；若本轮不启用更合适，也应明确写出“不启用更合适”的判断，不制造“是不是漏开了”的悬念。

## 短规则

- 先完成问题定义校验，再决定是否进入完整多模型合议模式、当前模型三席模拟复核模式或子 agent 辅助复核模式。
- 能在内部收束，就不升级给用户。
- 优先用低风险求证消除伪分歧。
- 先收束，再复查。
- 复查尽量与首轮异构。
- 能回退就回退，不得虚构多模型参与。
