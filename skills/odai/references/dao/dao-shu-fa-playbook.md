# 道术法判定手册

## 核心映射

- 道：定志、正名、定界、定序；取道家少干预、儒家名实相符与心学知行合一，已稳即行，未稳不妄作。
- 术：察情、审势、定主路、备路、拆解与切换；取兵家庙算止损和鬼谷子因势沟通，但只透明澄清，不诱导用户接受模型偏好。
- 法：定先手、公开规则、验证、止损与回写；取法家及韩非循名责实、法术势分离和工程证据，但实际权限、沙箱与强制执行仍属于宿主。

**谋定而后动**：道层谋定做什么，术层谋定怎么走，法层谋定如何验证与止损；一旦三层已稳，不以继续谋划代替行动。

`SDD` / `BDD` / `TDD` 等驱动法是术层主路选择；选定后的工艺在法层 `implement-code` 执行。

## 互补与制衡

- 这些思想是同一主流程的校验面，不是多角色、多 agent 或多票表决；裁决仍落在同一份任务状态与证据上。
- 道家抑制过度治理，儒家校准名实，心学推动知行合一：少干预不等于不行动，内省与直觉不能替代证据。
- 兵家负责庙算、胜点与止损，鬼谷子负责察情与透明沟通：变通不得改写用户治理点，也不得以诱导、隐瞒或操控换取同意。
- 法家与韩非提供公开硬门、循名责实及法术势分离：`法` 是规则，`术` 是程序，`势` 只来自宿主能力与授权，不由 skill 自封。
- 冲突时按宿主硬边界 → 用户治理点 → 可复验证据 → 主路效率排序；不得拿效率越权，也不得拿法条制造无价值仪式。

## 判层与切换

- 卡在道：目标未明、名分混乱、边界冲突、取舍标准未定。
- 卡在术：目标已清，但主路、拆法、依赖或切换条件未稳。
- 卡在法：方向和路线都清了，只差先手、验证、止损与收口。
- 看似卡在法，实则上游漂移：一执行就不断改目标或路线。

判层前先对齐项目现实：现有文档、相似实现、命名与目录规则、权限边界。

切换条件：道 -> 术，要志、名、界、序稳定到足以比较主路；术 -> 法，要主路、胜点、切换条件稳定到足以落先手与验证；法 -> 回写，凡新证据动摇道或术都必须回写。复核查守道、主路、验证与回写。

## 辩证约束

- 冲突先显化，不以含混折中掩盖目标、边界、授权、验收或证据矛盾。
- 只抓当前会改写任务的主要矛盾，也就是下一个关键决定；其余问题保留为候选，不平均用力。
- 主路、状态与结论只随判别性新证据转化；没有新证据时不靠换说法、换角色或重复争论制造进展。

## 短规则

- 道不能导出可执行的术 → 回道层补清。
- 不知彼（环境、约束、限制）知己（能力、资源、现状）→ 不轻定主路。
- 不做庙算（主路、备路、代价、胜点）→ 不轻开打。
- 能伐谋（认知、策略、边界、协同层化解）→ 不蛮打。
- 不见胜点（可控、可验证的阶段性优势）→ 不急开打。
- 没做、没验、没确认 → 不标完成。
- 不切流、不甩单：借力模块后仍由主流程收口。

## 主文件与继续

主文件最少保留：当前理解、道、术、法、当前裁决、下一步、结果总结；blocked 时加阻断原因和最小解除条件。任一时点仅凭主文件应能恢复任务目标、已确认边界、当前裁决、剩余风险与下一步。落盘优先级：用户指定路径 → 当前任务既有主文件 → 项目约定目录 → `plans/<日期>-<任务名>.md`。

继续推进优先级：用户本轮指定文件 → 上轮主文件 → 主文件引用的执行单 → 项目内最匹配且可继续的既有文档 → 新建。多个合理候选或仍不确定时，问一个确认问题。

后续队列只写入当前任务主文件，不写全局 TODO。用户入口是主文件里的 `## 后续队列` Markdown 区块，可直接编辑、勾选、改状态、改优先级、补充或删除条目。

入队规则：

- 不判断“未来是否跨轮”，只按当前收口事实。
- 当前范围内仍未完成、未验证或待解除，本轮又不能继续安全、合规、有证据地完成，且已有明确下一动作与验收证据或解除条件时，才入队。
- 当前范围内能继续完成的，先继续完成，不得用入队代替执行。
- 可入队项包括未完成验收项、未验证风险、blocked 解除条件和相邻建议；每项必须有类型、状态、范围、下一动作和验收证据或解除条件。
- 每项使用当前主文件内稳定 ID；状态变化更新原项，不复制任务。已完成 / verified 项保留终态证据，除非用户或新证据明确重开，否则恢复时不得重派。
- 未确认建议、风险观察和技能改进候选不得自动变成可执行待办；未确认的相邻建议只能标为待确认。
- 过程讨论、临时策略、废弃路线、模型自评和调试残留不得入队。

恢复规则：

- 用户说“继续”时，先恢复队列，跳过已完成 / verified 项，按已确认且可执行的项逐个推进；待确认项先问。
- 用户直接编辑的队列内容视为用户维护的任务输入；生产、外部系统、破坏性或高风险动作仍按交互契约重新确认授权与停止条件。

无法把后续队列写入主文件时：

- 不得静默只留在模型记忆。收口必须显式列出「未落盘后续队列」，保留每项的类型、状态、范围、下一动作和验收证据或解除条件。
- 标明该队列必须进入上下文摘要；压缩时完整保留队列，不要求现场检索文件。
- 交接或恢复时，先检查用户指定文件、上轮主文件、主文件引用的执行单、`plans/` 与上下文摘要中的 `## 后续队列` 或「未落盘后续队列」。
- 未落盘后续队列是最高优先级任务状态，不得丢弃。
- 能锁定当前主文件时，先合并回主文件；来源或范围不唯一时，先让用户选范围。
- 不得凭记忆或推断新造历史队列，避免形成第二套真相。

用户要求查看、整理、维护或统一处理后续队列时，不创建全局 TODO；按用户指定路径 → 上轮主文件 → 主文件引用的执行单 → `plans/` 与项目约定目录下含「后续队列」的主文件顺序动态汇总，按来源文件列出。维护动作必须回写原来源主文件；跨多个主文件执行前，先让用户选范围或按已确认优先级推进。

实施步骤较长、需协作、需单独跟踪验证回滚，或主文件失焦时，才拆 `.execution.md`。模板按需用 `assets/dao/main-template.md` / `assets/dao/execution-template.md`。

过程讨论、临时策略、废弃路线、模型自评和调试残留不写入项目正式文件；若需要保留，只写可复验结论、已确认取舍、风险和后续动作。

## 严格模式

常见简称：`pg`、`fp`、`ds`、`gp`、`gd`、`sslb`、`ic`。用户点名内部模块或审查风格属强偏好；若当前只需继承提问强度、输出深度或结构，进入严格模式：`fp-strict`、`ds-strict`、`strict-sslb`。等价执行只能基于本会话已实读过的模块规则；`implement-code` 不适用等价执行或严格继承。

同一阻断重复、首轮未解、或用户嫌输出过简时升档：补证据账本、已确认 / 待确认路径、取舍理由、当前最高价值动作与推进理由。

## 生命周期与迁移

生命周期状态、验收证据、旧状态迁移和未验证收口统一按 `references/dao/verification-contract.md`；本手册只负责道术法判层、主文件与后续队列，不另存状态定义。
