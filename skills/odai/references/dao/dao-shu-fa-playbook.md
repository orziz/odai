# 道术法判定手册

## 核心映射

- 道：定志、定名、定界、定序。
- 术：定主路、备路、拆解与切换。
- 法：定先手、验证、止损与回写。

`SDD` / `BDD` / `TDD` 等「X-驱动开发」不是道术法三层的对应，而是**术的一类**：开发任务在**术层选主路**时的几种方法（规格驱动 / 行为驱动 / 测试驱动），彼此是可选的主路，别把它们摊到道、术、法三层上。**选哪条驱动法是术层的事；选定后的具体工艺——如 TDD 的红-绿-重构——在法层 `implement-code` 执行。**（这些洋概念模型本就熟，可作接地注脚，但道术法才是母语。）

道可道，非常道。术无定数，法无定法。以下只提供判断思路，不提供刚性模板。

## 先判断卡在哪一层

看四个信号即可：

- 更像卡在道：目标未明、名分混乱、边界冲突、取舍标准未定。
- 更像卡在术：目标已清，但主路、拆法、依赖或切换条件未稳。
- 更像卡在法：方向和路线都清了，只差先手、验证、止损与收口。
- 看似卡在法，实则上游漂移：一执行就不断改目标或改路线，这其实是道或术未定。

先判层前，先对齐项目现状、现有文档、相似实现、命名与目录规则、权限边界。能自己补齐的事实先补齐，不拿抽象原则代替项目现实。

## 切换、复核与 blocked

- 道 -> 术：志、名、界、序已经稳定到足以支撑主路比较。
- 术 -> 法：主路、胜点、切换条件已经稳定到足以落先手与验证。
- 法 -> 回写：新证据足以动摇道或术时，必须回写上游，不把流程做成单向瀑布。
- 复核至少查四项：是否仍守道、主路是否漂移、验证是否成立、是否需要回写。
- blocked 的判定与解除条件统一按 `references/dao/interaction-contract.md` 实施准入执行。

## 短规则

- 道不能导出可执行的术 → 说明道还没定成，需要回到道层补清。
- 不知彼（环境、约束、限制）知己（能力、资源、现状）→ 不轻定主路。
- 不做庙算（比较主路、备路、代价、胜点）→ 不轻开打。
- 能伐谋（在认知、策略、边界或协同层化解问题）→ 就不蛮打（不直接进入重实现）。
- 不见胜点（可控的、可验证的阶段性优势）→ 不急开打。
- 没做、没验、没确认 → 继续标未完成，不虚报进度。
- 不切流、不甩单：借力其他模块后仍要自己收口，不把结果总结丢给别人。

## 常见失真与纠偏

- 用法补道：用户上来就给动作，跳过方向判断。→ 先问：这个动作想守住什么？真正目标是什么？
- 用术顶道：方案很多，但取舍标准没定。→ 先定优先级与不可接受结果，再比方案。
- 空道压术：只剩大词，导不出具体路线。→ 把抽象目标压成可判断、可执行的原则。
- 把知行合一当鲁莽执行：前提未清就开干。→ 前提未清先补上游；前提清后不要空谈，直接落法。

## 压缩输出规则

默认保留三层最短信息即可：

- 道：为何这样定
- 术：为何走这路
- 法：现在怎么做

若压缩后看不出边界、主路、胜点和先手，就压过头了。

## 主文件建议

主文件最少保留：当前理解、道、术、法、当前裁决、下一步、结果总结；若进入 blocked，还要写阻断原因和最小解除阻断条件。落第一版主文件可套 `assets/dao/main-template.md`。

## 简称与严格模式继承

- 常见简称：`pg`(project-guide)、`fp`(feature-plan)、`ds`(design-spec)、`gp`(game-plan)、`gd`(game-design)、`sslb`(review-sslb)、`ic`(implement-code)。
- 用户点名某内部模块或审查风格属强偏好：优先实读对应模块规则；若当前阶段只需继承其提问强度、输出深度或结构，进入严格模式——`fp-strict`、`ds-strict`、`strict-sslb` 分别对应 `feature-plan`、`design-spec`、`review-sslb`，继承其提问强度、文档结构与产物要求，不得缩水。
- 等价执行只能基于本会话已实读过的模块规则；未读过就如实说明并按 道 自身规则执行，不得声称继承。`implement-code` 不适用等价执行或严格继承，进入实施一律实读全文。

## 升档规则

- 默认追求更高首轮解决率、更少总轮次，不把“轻量试一轮”当默认起手。
- 同一阻断重复出现、首轮未解，或用户嫌过度压缩时，必须升档：补齐证据账本、已确认/待确认路径、取舍理由、当前最高价值动作与推进理由。
- 只有关键问题清空、路线已稳后，才允许降回更轻的展示强度。

## 落盘与继续推进优先级

- 落盘优先级：① 用户指定的路径/文件 ② 当前任务已存在的主文件 ③ 项目现成约定（docs/specs/design/plans/notes）④ 默认新建 `plans/<日期>-<任务名>.md`。
- 继续推进优先级：① 用户本轮指定的文件 ② 上轮主文件 ③ 主文件引用的执行单 ④ 项目内最匹配且可继续的既有文档 ⑤ 都没有再新建。存在 2 个以上合理候选或仍有不确定时，先用一个问题确认接哪一个，不凭“最匹配”硬继续。
- 实施步骤较长、需协作、需单独跟踪验证回滚，或主文件失焦时，才拆 `.execution.md`（套 `assets/dao/execution-template.md`）。

## 状态速查

`draft`(方向初成、仍有待确认) → `reviewed`(已复核、拿到裁决) → `ready`(可按下一步推进；也可表示实施已完成但明确验收点因环境暂未验证) → `executing`(已落地执行/持续更新) → `verified`(已按验收口径核对实际场景并总结；不以编译或“无报错”代替) → `blocked`(被关键信息/权限/环境/分歧卡住且无不越线后续动作)。

## 完整推进的完成态

三条同时满足才算没停半途：① 本轮明确范围内的问题已推进到可交付结论 ② 有改动的，已完成当前环境下有判别力、成本与范围合理的验证及必要复核；实际场景不可执行时不以编译凑验收，而是明确未验证项并停在 `ready` ③ 已给出验收结论、未完成项与下一步。
