# 模型选择基线

## 何时读取

- 总控已坐实当前用户真实可分配之模型列表，需给席位选模
- 需判异构是否足成立完整增强
- 需给收束补出未入完整增强之模型层原因
- 需给辅助复核或冻结后执行分工安排模型与关注点

## 总原则

1. 先判模式是否合法，再谈选模优劣；模式不立，不得靠“看似不同”硬称多模型合议。
2. 主流程求稳，席位求异；讨论 / 决策阶段不拆题分工，执行分工阶段方可按子任务性质选模。
3. 只采当前会话/宿主已暴露之信息、平台文档、仓库内已验证经验与本轮可直观察之表现；不得编造模型特长。
4. 实际派位、计费、预算、权限、额度拒绝高于理论标签；理论成本标签只作参考，不单独构成剔除依据。
5. 信息不足，不能判“是否异构”或“此席该用何模”时，则降档，不作臆断。

## 进入选择前必须拿到的输入

1. 当前总控经直查、AI 续查或宿主/API 续查已确认之模型列表、当前模型信息，或可可靠沿用之同路默认模型。
2. 当前可见之权限、配额、计费/预算、实际派位拒绝信息，以及可稳定识别之模型字段：provider、model id、display name 或等价字段。
3. 已决定之模式：完整增强、子 agent 辅助复核，或冻结后执行分工。
4. 席位清单与关注点；若为执行分工，则写明子任务范围。
5. 当前任务之主要约束：稳定性、长上下文、速度、成本、读档规模、反证强度。

## 直查与可派位口径

- 模型列表、可用性与取数穷尽口径，统一沿用 `references/dao/parallel-consensus-playbook.md` 的“取数与选模”。
- 平台文档只能助查，不单独构成“当前环境已暴露模型列表”之证。
- 字段若缺、别名若乱、可用性若未坐实，则不继续假设；直接降档。
- 若原生派位、宿主/API 代查或实际拉起已回“计费/预算/权限/额度/账号不可用/需额外付费且未获授权”或等价拒绝，则该模型即刻剔除。

## 模型归类

先按已暴露信息与已验证经验，给候选模型加最保守之标签：

- 综合收束型：适合主流程、长链路总结、冲突裁决。
- 题面挑战型：适合挑偷换概念、补未定、提反例。
- 风险审计型：适合边界、验收、遗漏与冻结后复查。
- 速扫补证型：适合大量读文件、补目录、快速查漏。
- 长上下文型：适合长文档、多文件与大证据集。

若信息不足以可靠打标签，只记：`可用未验`。

## 模型倾向

以下只是对合法候选模型之排序参考，不是硬约束：

- 视觉/文案：优先 Gemini 家族。
- 分析/规划：优先 Claude Opus；若 Opus 非合法候选，回退其他 Claude。
- 验收/执行：优先 ChatGPT / OpenAI GPT 家族。
- 多类命中：先看阶段与席位关注点，再看主模块；不得令诸席同质化。
- 仓库倾向只作合法候选排序，不压过异构性、反证席与风险审计需要。
- 已验表现、权限/配额/实际拒绝若与倾向相冲，服从已验事实，并在理由中直陈。

## 席位分配基线

### 主流程

- 优先综合收束型。
- 若多模皆可胜任，优先更稳、长上下文更足、总结偏差更少者。

### 完整增强模式

- 至少三席，且至少三种不同模型；其中至多一席与主流程同模型。
- 默认至少保：一席反证 / 逆向挑战、一席风险 / 边界 / 验收、一席补证 / 实施代价 / 冻结后复查。
- 席位皆围绕同一核心问题给出相对完整判断；关注点只是补充，不是拆题分工。

### 子 agent 辅助复核模式

- 默认一席；若用户强要增强而完整增强不立，且二席仍值其成本，方可扩至二席。
- 若环境已露模型列表，且某模型明显更适做反证或风险审计，可优先选之。
- 若模型列表未露，或只露一模，仍可辅助复核，但收束必须写清未入完整增强之直接原因。

### 冻结后执行分工

- 只在题面、主路、边界与验收已冻结后，才按实现、验证、补证、文档等子任务性质选模。
- 若某执行席位之工作足以改写路线、边界、验收或不可接受结果，则不再当执行分工处理；立回 `references/dao/parallel-consensus-playbook.md` 重判。
- 若同轮确需多种执行产物，默认多次独立指派：执行方案骨架 -> 验收与风险校验 -> 交付文档收口；若首席产物已足，则不强凑后续席位。

## 不同模型的判定口径

1. 只有可单独分配、且已坐实当前用户当前会话真实可用之稳定模型标识，方可算“不同模型”。
2. 同一基础模型之不同 prompt、温度、推理档位、角色设定或展示名，不算不同模型。
3. 同一模型之包装别名、路由别名或只换显示名者，不算不同模型。
4. 平台若明确把不同版本、不同尺寸或不同发布日期之模型列为独立可分配模型，可视作不同；然完整增强仍优先跨家族或跨能力侧重点之组合。
5. 只要不能确认是不是不同模型，就按“不能确认”处理，不得乐观放行。

## 选择步骤

1. 先将本轮已露之模型字段标准化，并去掉同模型之别名、包装与重复项。
2. 先算可确认之不同模型数；若目标为完整增强而不足三种，退出完整增强；辅助复核另按授权与收益判。
3. 依当前阶段与席位关注点打标签；无把握者记作 `可用未验`。
4. 先定主流程，再按模式分配其余席位；讨论阶段配同题挑战与审计，执行阶段配子任务模型。
5. 将每席之模型与理由写入统一下发包。
6. 任一步仍拿不准，则退出当前档，不为形式保留完整增强。

## 收束必须回报的模型层信息

- 当前进入的是完整增强、子 agent 辅助复核，还是冻结后执行分工。
- 当前环境是否已暴露当前用户真实可分配之模型列表，或仅暴露当前模型，抑或仍未暴露。
- 本轮实际调用之模型；若未暴露，直书“当前环境未暴露”。
- 哪些模型因实际成本拒绝、权限、额度或账号不可用而被排除。
- 若未入完整增强，模型层面的直接原因：模型列表未露、仅一模型、不能坐实三种异构、辅助复核亦不值，或实际拒绝所阻。
