# 执行编排与 Agent 交接

## 何时读取

`references/dao/agent-governance.md` 已决定使用 agent，且存在两个以上可独立验收的任务、任务依赖、跨压缩恢复、大体量回交或审查后的修复波时读取。一次性小任务、三言两语可回交的结果或主流程直办更省时，不读本文件。

本文件是任务生命周期、文件化交接和修复波的 owner；不改写 `agent-governance` 的能力真实性、权限和成本门，也不改写主文件与后续队列规则。

## 输出硬卡

- 已知会产生大 diff、长报告或多轮日志时，编排答复必须外显：原始产物写入 scratch / tmp；主流程只保留任务 ID、路径、摘要、变更文件、验证、未决项和状态。
- 修复波编排至少外显以下四行：

```text
冻结清单：<去重后的完整 finding IDs>
Review agent：READ_ONLY，不写入
Fix agent：独立 implement-code agent，承接冻结清单
Main flow：逐项复验后才标 VERIFIED
```

## 执行编排账本（owner）

不建立第二套真相。把任务行写入当前主文件引用的执行单；没有执行单时，复用主文件后续队列。每项至少保留：

- 稳定任务 ID、目标、范围 / 非范围、依赖和当前负责人。
- 状态、验收证据、停止条件及 brief / report / diff 等产物路径。

任务局部状态只用于编排：

- `PENDING`：依赖未满足或尚未下发。
- `RUNNING`：已有唯一执行者；同一 ID 不并发重派。
- `NEEDS_CONTEXT`：agent 缺少明确输入并已停手；由主流程决定补证、询问或改路。
- `DONE`：agent 已回交，尚未经主流程复验。
- `VERIFIED`：主流程已按验收证据复核。
- `BLOCKED`：agent 在冻结范围内无路可走；只是一项回交结果，整体是否 blocked 仍由主流程裁决。

只下发依赖已 `VERIFIED` 的 `PENDING` 项。恢复或压缩后先读账本：`RUNNING` 先核对真实存活状态，`DONE / VERIFIED` 不重派；新证据要求重开时沿用原 ID，记录重开原因，不复制任务。

独立任务在隔离收益高于冷启动与交接成本时使用新 agent；共享上下文强、顺序紧密或任务很小时沿用主流程或同一执行者，不把“每任务新 agent”设为铁律。

## 文件化交接

- 短 brief 和短结果直接回交。大 diff、长扫描报告、多轮日志或会挤占主上下文的材料才文件化。
- 题面已明确会产生上述大产物时，只列“大 diff / 长报告”作为输出不算文件化交接。
- 同轮原始大产物优先放宿主 scratchpad / tmp；brief 只含冻结目标、范围、禁区、验收和停止条件，不复制整段聊天。
- 主流程只接任务 ID、产物路径、摘要、变更文件、验证结果、未决项和状态。跨会话确需保留时，把可复验证据蒸馏进既有主文件 / 执行单；不默认把原始报告写进用户项目。
- 文件路径不存在、内容无法复验或 agent 只自报完成时，不采纳该交接。

## 审查与修复波

审查 agent 保持只读。主流程先去重并复核 findings，冻结完整问题清单，再进入修复；不在审查结果陆续到达时反复启动零散修复。

只列修复步骤或泛称“之后复核”不算角色分离。

- 多个已确认问题共享代码上下文、可在同一 checkout 安全批修，且一次交接更经济时，另派一个 `implement-code` 修复 agent，携带完整冻结清单。
- 问题跨域、修改所有权冲突、依赖顺序明显或单波风险过高时，按依赖拆波或串行，不为减少 agent 数量强行合并。
- 修复 agent 回交 diff 与验证证据，只能标 `DONE`；主流程逐项复验后才标 `VERIFIED`。新发现另起下一波，不静默扩入已冻结范围。
- 规格符合性与质量审查只在高风险、共享边界或正式准入时分成独立裁决；普通任务仍做与风险成比例的一次复核。

收口时把任务状态、证据、未决项和必要产物路径回写同一账本；不以 agent 数量、文件存在或自报完成替代验收。
