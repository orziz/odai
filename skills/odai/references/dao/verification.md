# 验成与完成判定

需要判断是否完成、设计验证、承接旧结果或在环境受限时收口时读取。

## 何谓成事

- 以用户定义的可观察结果和验收为主判据；新发现只有会实质破坏该结果、已确认边界或关键安全时才阻断，不用范围外完整性无限抬高完成门。
- 验证求最小充分证据，不求最大覆盖。关键验收与主要回归风险已有相称证据就停止；额外检查必须仍能改变完成判断。

## 选择证据

1. 把目标和关键风险映射到可判定证据，优先选择最接近真实使用场景且成本合理的方式。
2. 测试、构建、lint、类型、静态检查、diff、截图和人工观察各自只证明覆盖到的属性；不要彼此冒充。
3. 高风险、跨层或用户可见改动需要更强的场景证据；局部低风险改动可用较轻验证。
4. 工具或环境不可用时先尝试等价路径或缩小验证范围；不要因第一次失败就判 blocked，也不要用弱证据冒充强证据。

## 高影响参数

- 名称、注释、配置字段和设计只证明 `declared`，局部观测为 `observed`；只有真实调用链达到 `verified_end_to_end`，才能称已有保护、已幂等或可安全重试。三者不得互相冒充。
- 方向、幅度或保护链任一未定时不写入参数。若提出待验证的实验候选，明确环境或流量边界、单一变量、观察窗口、通过 / 停止指标和恢复原值的回退；候选不冒充生产决定。
- 因保护链不足而停手时，写清待证明的不变量、跨层关联字段或标识、取证点与通过 / 停止条件；不要只说“验证幂等、重试或安全”。

## 状态与收口

普通任务用自然语言报告即可。实施型、跨会话或需要精确交接时才使用这些状态：

- `ready_to_execute`：目标与前提已足以实施，尚未实施。
- `executing`：正在实施或验证。
- `implemented_unverified`：改动已落地，但关键场景尚未覆盖。
- `verified`：目标验收已被实际场景或等价强证据覆盖。
- `blocked`：当前没有不越权且有证据的可行后续动作。

只有关键验收均通过时才宣称总体完成。若为 `implemented_unverified`，说清已实施和已验证内容、缺失场景及原因、恢复后的最小复验动作和通过标准。

## 承接旧任务

用户要求“继续、验收、收尾”时，先从用户原话、当前产物、主文件、执行账本和 diff 恢复目标与真实状态。不要默认重新实施，也不要沿用缺少证据的旧“完成”标签；根据当前证据继续验证或推进。
